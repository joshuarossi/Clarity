# Contract: WOR-161 — Deployed app renders completely unstyled: src/main.tsx never imports src/styles/globals.css

## Why this work exists

The deployed application renders with zero styling because `src/main.tsx` — the Vite entry point — never imports `src/styles/globals.css`. Without this import, Vite's CSS pipeline never picks up the global stylesheet, so no design tokens, resets, or component styles reach the browser. This is a one-line fix with high visual impact.

## Files and exports

### `src/main.tsx` (modify)

This is the app's entry point. It currently imports React, Convex, the router, and `App`, but is missing the CSS import. The fix adds `import './styles/globals.css';` as a side-effect import near the top of the file (conventionally before component imports, after library imports). No new exports are introduced.

## Data dependencies

None. This ticket touches no Convex queries, mutations, or actions.

## Invariants

1. **CSS import must exist in the entry point.** `src/main.tsx` must contain `import './styles/globals.css'` (or equivalent path). This ensures Vite includes the CSS in both dev mode (injected via `<style>`) and production builds (extracted to a `.css` asset). If this import is removed, the entire app loses styling.

2. **Relative path from src/main.tsx.** The import path must be `'./styles/globals.css'` — relative to `src/main.tsx`'s location in `src/`. An absolute or aliased path would also work but would deviate from the existing import style in the file.

## Edge cases

**Import ordering:** The CSS import should appear before component imports to ensure styles are loaded before any component renders. This matters for SSR or snapshot tests but in practice Vite handles it regardless of order. Convention: place it after third-party imports, before local component imports.

**File existence:** `src/styles/globals.css` already exists and imports `./components.css`. No file creation is needed.

## Non-goals

- No changes to `globals.css` content, structure, or the files it `@import`s.
- No addition of CSS tooling (Tailwind, PostCSS plugins, CSS modules).
- No refactoring of how styles are organized in the project.
- No changes to `vite.config.ts` — Vite handles `.css` imports natively.

## Test coverage

Both ACs are covered by a single unit test file: `tests/unit/globals-css-import.test.ts`.

- **AC: src/main.tsx imports globals.css** — Unit test reads `src/main.tsx` source and asserts it contains an import statement matching `styles/globals.css`. This is a static analysis test (similar to the existing `tests/unit/vite-config.test.ts` and `tests/unit/no-node-builtins.test.ts` patterns in this project). Unit layer is sufficient because the invariant is "the import statement exists in the source file."

- **AC: Design tokens are available** — Unit test can additionally verify that `src/styles/globals.css` exists and contains expected CSS custom property declarations (e.g. `--bg-canvas`). This confirms the import target is valid. Full runtime verification (tokens applied to rendered DOM) would require e2e, but the root cause here is purely the missing import — if the import exists and the file exists, Vite guarantees the CSS is bundled.
