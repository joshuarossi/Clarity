# Contract: WOR-128 — Draft coach AI action — generateResponse with readiness detection

## Why this work exists

The Draft Coach is a private, exploratory AI conversation that helps a party craft a polished joint-chat message. WOR-127 built the session/message CRUD layer with a stub `generateResponse`. This ticket replaces that stub with the full AI action: multi-turn coaching conversation, readiness detection to know when the user wants a final draft, structured draft output, streaming delivery, template integration, and error handling with retry. Privacy is critical — the Draft Coach must never see the other party's synthesis or private content.

## Files and exports

### `convex/draftCoach.ts` (modify)

The existing file (created by WOR-127) contains session/message queries and mutations plus a no-op `generateResponse` stub. This ticket replaces the stub with a full implementation and adds the internal queries/mutations needed by the action.

**Existing exports preserved as-is:** `session`, `startSession`, `sendMessage`, `sendFinalDraft`, `discardSession`. These are WOR-127's work and are not modified.

**`generateResponse` — internalAction (replace stub).** This is the core of the ticket. The action:

1. Reads the session and its messages via `getSessionForGeneration`.
2. Reads the case to get `templateVersionId` via `getCaseForDraftCoach`.
3. Reads the template version for `draftCoachInstructions` via `getTemplateVersionForDraftCoach`.
4. Reads the acting user's `partyState` for `synthesisText` via `getPartyStateForDraftCoach`.
5. Reads joint-chat messages for context via `getJointMessagesForDraftCoach`.
6. **Readiness detection:** Checks the latest USER draftMessage content against the canonical readiness signals (case-insensitive): `["i'm ready", "draft it", "write the message", "looks good, write it"]` and the exact string `"Generate Draft"` (the canonical button message). Sets a boolean flag `isReadyForDraft`.
7. Calls `assemblePrompt` with `role: "DRAFT_COACH"`, passing joint-chat history and the acting user's synthesis only (not the other party's). When `isReadyForDraft` is true, appends an instruction to the prompt telling the AI to respond with a structured `{ "draft": "..." }` JSON block.
8. Inserts a STREAMING draftMessages row via `insertStreamingDraftMessage`.
9. Calls Claude (`claude-sonnet-4-5`) with streaming. Batches content updates (~50ms) via `updateStreamingDraftMessage`. On completion, finalizes via `finalizeStreamingDraftMessage` with total token count.
10. If `isReadyForDraft` is true, parses the AI response for a `{ "draft": "..." }` JSON block and writes the extracted draft text to `draftSession.finalDraft` via `setSessionFinalDraft`. If parsing fails, the response is still stored as a normal message (graceful degradation — user can keep conversing).
11. **Error handling:** On API failure, retries once after 2s backoff. On second failure, marks the message as ERROR via `markDraftMessageError`.
12. **Mock mode:** When `CLAUDE_MOCK=true`, uses `getMockClaudeResponse("DRAFT_COACH")` with the same streaming simulation pattern as `privateCoaching.generateAIResponse`. Mock failure simulation via `CLAUDE_MOCK_FAIL_COUNT` / `CLAUDE_MOCK_FAIL_STATUS`.

**`retryLastDraftAIResponse` — public mutation.** Takes `{ sessionId }`. Finds the last AI draftMessage with `status: "ERROR"` in the session, deletes it, and re-schedules `generateResponse`. Enforces auth + ownership. Follows the same pattern as `privateCoaching.retryLastAIResponse`.

**Internal queries (all new):**

- **`getSessionForGeneration`** — reads the draftSession by ID and joins its draftMessages via `by_draft_session` index. Returns `{ session, messages }` or `null` if session not found.
- **`getJointMessagesForDraftCoach`** — reads jointMessages by `caseId` using `by_case` index. Returns all joint messages (sorted by `createdAt` in the caller).
- **`getPartyStateForDraftCoach`** — reads the acting user's partyState via `by_case_and_user` index. Returns the partyState doc or `null`.
- **`getCaseForDraftCoach`** — reads the case doc by ID via `ctx.db.get()`.
- **`getTemplateVersionForDraftCoach`** — reads the templateVersion doc by ID via `ctx.db.get()`.

**Internal mutations (all new):**

- **`insertStreamingDraftMessage`** — inserts a `draftMessages` row with `role: "AI"`, `content: ""`, `status: "STREAMING"`, `createdAt: Date.now()`.
- **`updateStreamingDraftMessage`** — patches `content` on the streaming row.
- **`finalizeStreamingDraftMessage`** — patches `content`, `status: "COMPLETE"`, and adds token count. Note: `draftMessages` schema does not have a `tokens` field, so token count is logged but not persisted on the row (unlike privateMessages which has a `tokens` field). The token count is recorded in the console log for cost tracking.
- **`markDraftMessageError`** — patches `status: "ERROR"`.
- **`setSessionFinalDraft`** — patches `draftSession.finalDraft` with the extracted draft text.

**Why no privacy filter:** Unlike the Coach (which sees both parties' syntheses and could leak cross-party content), the Draft Coach's context includes only the acting user's own synthesis and the shared joint-chat history. There is no private data from the other party in the context window, so the response filter from `convex/lib/privacyFilter.ts` is not needed.

**Why no Haiku classification gate:** The Coach uses a Haiku classifier to decide whether to respond (silent on NORMAL_EXCHANGE). The Draft Coach always responds to every user turn — it's a dedicated 1:1 coaching conversation, not a facilitated group chat.

## Data dependencies

### Queries consumed

- **`draftSessions` table** — read by `getSessionForGeneration` via `ctx.db.get(sessionId)`. Fields consumed: `_id`, `caseId`, `userId`, `status`.
- **`draftMessages` table** — read by `getSessionForGeneration` via `by_draft_session` index. Fields consumed: `_id`, `draftSessionId`, `role`, `content`, `status`, `createdAt`. The conversation history is mapped to `PromptMessage[]` format for `assemblePrompt`.
- **`jointMessages` table** — read by `getJointMessagesForDraftCoach` via `by_case` index. Fields consumed: `authorType`, `content`, `status`, `createdAt`. Filtered to `status === "COMPLETE"` and mapped to `PromptMessage[]` for the joint-chat history context.
- **`partyStates` table** — read by `getPartyStateForDraftCoach` via `by_case_and_user` index. Fields consumed: `synthesisText` (the acting user's own synthesis).
- **`cases` table** — read by `getCaseForDraftCoach` via `ctx.db.get(caseId)`. Fields consumed: `templateVersionId`.
- **`templateVersions` table** — read by `getTemplateVersionForDraftCoach` via `ctx.db.get(templateVersionId)`. Fields consumed: `globalGuidance`, `draftCoachInstructions`.

### Mutations called

- **`insertStreamingDraftMessage`** — writes to `draftMessages`.
- **`updateStreamingDraftMessage`** — patches `draftMessages`.
- **`finalizeStreamingDraftMessage`** — patches `draftMessages`.
- **`markDraftMessageError`** — patches `draftMessages`.
- **`setSessionFinalDraft`** — patches `draftSessions.finalDraft`.

### External API

- **Anthropic API** — `claude-sonnet-4-5` via `@anthropic-ai/sdk` streaming. Called with `messages.stream()`. Same pattern as `privateCoaching.generateAIResponse` and `jointChat.generateCoachResponse`.

## Invariants

**Privacy: only the drafting user's data in context.** The action reads joint-chat messages (visible to both parties anyway) and the acting user's synthesis from their partyState. It does NOT query the other party's partyState, synthesis, or private messages. This is enforced structurally — `getPartyStateForDraftCoach` takes `userId` as an argument and queries `by_case_and_user`, and there is no query for the other party's data.

**Readiness detection is deterministic.** The canonical readiness signals are: `["i'm ready", "draft it", "write the message", "looks good, write it"]` (case-insensitive, trimmed) and the exact string `"Generate Draft"`. The check is against the latest USER draftMessage in the session. The implementation should use `.toLowerCase().trim()` and check `.includes()` or exact match. This is a substring match for the natural-language phrases and an exact match for the button message.

**On readiness, finalDraft is set but never sent.** When the AI produces a draft in response to a readiness signal, the action writes it to `draftSession.finalDraft` via `setSessionFinalDraft`. It does NOT insert into `jointMessages` or call `sendUserMessage`. The send gate is enforced by the existing `sendFinalDraft` mutation (WOR-127), which requires explicit user action.

**Non-readiness turns produce normal messages.** When the user's message does not match a readiness signal, the AI's response is stored as a normal draftMessages row. `setSessionFinalDraft` is not called. The session's `finalDraft` field remains whatever it was (typically `undefined`).

**Streaming lifecycle matches sibling actions.** The STREAMING → batched updates → COMPLETE pattern is identical to `privateCoaching.generateAIResponse`: insert row with `status: "STREAMING"` before the API call, batch content updates every ~50ms, finalize with `status: "COMPLETE"` and token count.

**Error retry: exactly one retry.** On API failure (rate limit, timeout, network error), the action retries once after a 2s delay. On second failure, the draftMessages row is marked `status: "ERROR"`. No third retry. This matches the pattern in `privateCoaching.generateAIResponse` and TechSpec §6.5.

**Model is `claude-sonnet-4-5`.** Per TechSpec §6.1, all user-facing generation uses Sonnet. No Haiku classification for Draft Coach.

## Edge cases

**Session not found or not ACTIVE.** If `getSessionForGeneration` returns null, or the session status is not ACTIVE, the action returns early without inserting any message. This can happen if the user discards the session between scheduling and execution.

**No prior messages in session.** On session start (`startSession` schedules `generateResponse` immediately), there are no USER messages yet. The action should detect this and have the AI generate an opening coaching message (e.g., "What would you like to communicate to the other party?"). The readiness check is skipped when there are no USER messages.

**Readiness detected but AI response doesn't contain structured draft.** If the AI fails to produce a parseable `{ "draft": "..." }` JSON block despite the readiness instruction, the response is stored as a normal message. `finalDraft` is not set. The user can try again or continue the conversation. This is graceful degradation, not an error.

**Template has no draftCoachInstructions.** The `assemblePrompt` function handles this via `appendTemplate` — it falls back to `globalGuidance` if `draftCoachInstructions` is undefined. No special handling needed in the action.

**Loading state.** The STREAMING row with empty content is visible to the UI immediately after the action starts. The UI renders a streaming indicator (same pattern as private coaching and joint chat).

**Error state.** The ERROR status on the draftMessages row signals the UI to show a Retry button. The `retryLastDraftAIResponse` mutation deletes the error row and re-schedules generation.

**Mock mode failure simulation.** `CLAUDE_MOCK_FAIL_COUNT` and `CLAUDE_MOCK_FAIL_STATUS` env vars control simulated failures, identical to the pattern in `privateCoaching.generateAIResponse`.

## Non-goals

**Draft Coach UI / side panel.** The frontend components are a separate ticket. This ticket is the backend AI action only.

**Privacy filter on Draft Coach output.** The Draft Coach's context contains no other-party private data, so the `filterResponse` check from `convex/lib/privacyFilter.ts` is not applicable. The Coach (joint chat) needs it because it sees both syntheses; the Draft Coach does not.

**Haiku classification gate.** The Draft Coach always responds. It doesn't need the "should I speak?" classification that the Coach uses.

**Solo mode viewAsRole.** Not in the acceptance criteria. Can be added in a follow-up.

**Transcript compression.** TechSpec §6.4 describes compression when context exceeds 60k tokens. This is a separate concern and not in scope for this ticket.
