# Contract: WOR-155 — Convex deploy fails: compression.ts imports Node "crypto"

## Why this work exists

`convex/lib/compression.ts` imports `createHash` from Node's built-in `"crypto"` module (line 1). Convex functions run in a V8 isolate that does not provide Node built-in modules. When `npx convex deploy` bundles the file, the import fails, blocking all CI deployments. The fix is to replace the Node crypto dependency with a pure-JS synchronous hash function.

## Files and exports

### `convex/lib/compression.ts` (modify)

Remove `import { createHash } from "crypto"` and replace the `hashContent` internal function with a pure-JavaScript hash implementation. A simple non-cryptographic hash (e.g. a DJB2/FNV-1a variant or a basic string hash producing a hex-like string) is sufficient because the hash is used only for in-memory cache keying within a single action invocation — not for security. The key properties needed are: deterministic, low collision for typical conversation text, and synchronous.

All public exports (`CompressibleMessage`, `CompressResult`, `compressTranscript`, `estimateTokens`, `COMPRESSION_PROMPT`) remain unchanged in name, type, and behavior.

## Data dependencies

None. This module has no Convex query/mutation dependencies. It receives messages as arguments and returns compressed messages.

## Invariants

**No Node built-in imports.** The Convex runtime is a V8 isolate without Node.js APIs. After this fix, the file must have zero imports from `crypto`, `fs`, `path`, `buffer`, or any other Node built-in. This is enforced at deploy time by `npx convex deploy`.

**Deterministic hashing.** The replacement hash function must be a pure function: same input string always produces the same output string. This is required for the WeakMap-based cache to work correctly.

**Cache behavior preserved.** The existing test suite (AC 4 in WOR-101) verifies that calling `compressTranscript` twice with identical messages results in only one API call. This behavior must continue to pass.

## Edge cases

**Empty message array.** `hashContent([])` is called with an empty array when all messages fit in the oldest half of a single-element array. The hash function must handle an empty string input without throwing.

**Very long messages.** The hash function must not have pathological performance on large strings. A linear-time algorithm (single pass over the string) is required.

## Non-goals

- **No async hashing.** `crypto.subtle.digest` returns a Promise. Adopting it would require making `hashContent` async, which changes the internal control flow of `compressTranscript`. The fix uses a synchronous pure-JS hash instead.
- **No new npm dependencies.** The fix is a self-contained inline function (5–15 lines).
- **No test changes.** The existing `tests/unit/compression.test.ts` already covers the cache behavior. No new test file is needed for this bug fix.

## Test coverage

All three acceptance criteria are verified by the existing `tests/unit/compression.test.ts`:

- **AC "Remove Node crypto import"** — verified indirectly: if the import were still present, `npx convex deploy` would fail (integration-level), but at unit level the tests exercise `hashContent` through the cache behavior tests (AC 4 describe block). The tests will also fail to _run_ under vitest if a Node import breaks module resolution in CI.
- **AC "Deterministic cache keys"** — verified by the "second call with same messages returns cached result without API call" test (line 248).
- **AC "Existing tests pass without modification"** — verified by running `npx vitest run tests/unit/compression.test.ts` with zero changes to the test file.
