# Contract: WOR-104 — Shared chat components

## Why this work exists

Private Coaching, Joint Chat, and Draft Coach all share the same chat experience. Without a shared component system, each surface would re-implement message rendering, scroll behavior, and input handling — leading to visual inconsistencies and duplicated logic. These four components form the foundational UI layer that every conversational surface in Clarity will compose.

## Files and exports

### `src/components/chat/ChatWindow.tsx` (presentational, create)

Scrollable container that renders an array of `ChatMessage` objects as `MessageBubble` children. The outer element carries `role="log"` and `aria-live="polite"` for screen reader announcements. Implements sticky auto-scroll: tracks whether the user has scrolled up from the bottom, and only auto-scrolls on new messages when they haven't. Each new message wrapper applies the `cc-bubble-enter` animation class on mount (via a key-based approach or ref tracking) ensuring the animation fires once and does not replay on re-render.

**Why presentational:** The ticket explicitly states "ChatWindow receives [messages] as a prop; it does not fetch data itself." Parent pages (Private Coaching view, Joint Chat view, Draft Coach panel) will call Convex queries and map the results into the `ChatMessage` shape before passing them down. This keeps ChatWindow reusable across all three surfaces without coupling it to any specific query.

### `src/components/chat/MessageBubble.tsx` (presentational, create)

Renders a single message with one of 7 visual variants and one of 3 statuses. The variant determines which `cc-bubble-*` CSS class is applied. The status determines conditional children:

- `STREAMING`: renders content + inline `<StreamingIndicator />`, no copy button, no timestamp.
- `COMPLETE`: renders full content + copy button + timestamp (shown on hover).
- `ERROR`: renders content with `cc-bubble-error` class + Retry button.

Exports the `BubbleVariant` and `MessageStatus` types so parent components and tests can import them.

**Variant-to-CSS mapping:**
| BubbleVariant | CSS class |
|---|---|
| `"user"` | `.cc-bubble` |
| `"coach"` | `.cc-bubble-coach` |
| `"coach-joint"` | `.cc-bubble-coach-joint` |
| `"coach-intervention"` | `.cc-bubble-coach-intervention` |
| `"party-initiator"` | `.cc-bubble-party-initiator` |
| `"party-invitee"` | `.cc-bubble-party-invitee` |
| `"error"` | `.cc-bubble-error` |

Note: The `"error"` variant is used when `status === "ERROR"`. When status is ERROR, the component applies `cc-bubble-error` regardless of the `variant` prop value. The `variant` prop's `"error"` value exists so callers can be explicit, but the ERROR status is the authoritative trigger for error styling.

### `src/components/chat/MessageInput.tsx` (presentational, create)

A `<textarea>` with a Send button. Handles keyboard behavior: Enter triggers `onSend` with the current text and clears the textarea; Shift+Enter inserts a newline. When `isAiResponding` is true, the Send button is disabled but the textarea remains enabled so the user can pre-type their next message. The `onSend` callback receives the trimmed text string; it is not called if the text is empty/whitespace-only.

### `src/components/chat/StreamingIndicator.tsx` (presentational, create)

A pure presentational element: a `<span>` with the existing `cc-streaming-cursor` CSS class from `components.css`. No props beyond optional `className`. This component exists so that `MessageBubble` can render it inline at the end of streaming content, and so tests can assert its presence/absence independently.

## Data dependencies

None. All four components are presentational and receive data via props. No Convex queries, mutations, or actions are called. The `ChatMessage` interface is a UI-layer type defined in `MessageBubble.tsx` (alongside the variant/status types) — it is not a Convex `Doc` type. Parent pages will map from `Doc<"privateMessages">` or `Doc<"jointMessages">` into `ChatMessage` when composing these components.

The `ChatMessage.id` field is a string (not `Id<"privateMessages">`) to keep the components backend-agnostic. Parents will pass `doc._id` as the string id.

## Invariants

**All components are presentational.** They must not import from `convex/` or call any hooks like `useQuery`/`useMutation`. This is the most important architectural decision: it keeps the chat UI reusable across Private Coaching (which reads `privateMessages`), Joint Chat (which reads `jointMessages`), and Draft Coach (which reads `draftMessages`) — three different Convex tables with different schemas that all map to the same `ChatMessage` shape at the UI layer.

**Sticky auto-scroll.** ChatWindow must detect whether the user is "at the bottom" of the scroll container. When they are, new messages auto-scroll into view. When they've scrolled up (e.g., to re-read earlier messages), new messages append but do not force a scroll jump. The detection should use a small threshold (e.g., 50px from bottom) to account for rounding. Scrolling back to the bottom re-engages auto-scroll.

**Copy button gating.** The copy button must only render when `status === "COMPLETE"`. A STREAMING message that completes will re-render with status COMPLETE and gain the copy button at that point. Copy should use `navigator.clipboard.writeText`. If `onCopy` is provided, it is called after the copy succeeds (for analytics/feedback); if not provided, the component handles copy silently.

**Retry button gating.** The Retry button renders only when `status === "ERROR"`. It calls the `onRetry` callback. If `onRetry` is not provided, the Retry button is not rendered (graceful degradation).

**Enter animation fires once.** The `cc-bubble-enter` animation (150ms fade + 8px translateY) must fire when a message first appears in the DOM but must not replay when the component re-renders (e.g., when a STREAMING message updates its content). Use the message's `id` as the React `key` to ensure stable identity, and apply the animation via the CSS class on the wrapper element. CSS `animation-fill-mode: both` (already in `components.css`) handles this correctly as long as the element isn't unmounted and remounted.

**Timestamp visibility.** Timestamps use `--text-tertiary` color, `--font-size-timestamp` (12px), and are hidden by default (CSS `opacity: 0` or `visibility: hidden`). On hover of the bubble, the timestamp becomes visible. Timestamps only render for COMPLETE messages (not STREAMING or ERROR).

**MessageInput Enter behavior is non-negotiable.** Enter sends. Shift+Enter inserts newline. The textarea must not submit a form — it intercepts `keydown` events directly. Empty/whitespace-only submissions are blocked (onSend is not called).

## Edge cases

**Loading state.** ChatWindow with an empty `messages` array renders the scrollable container (with `role="log"` and `aria-live="polite"`) but no message bubbles. No special empty-state UI is needed — the parent page is responsible for showing loading spinners or empty-state messages above or around ChatWindow.

**Empty input submission.** MessageInput must not call `onSend` when the textarea contains only whitespace. The Send button should appear visually disabled (same styling as `isAiResponding`) when the input is empty.

**Rapid message arrival.** When multiple messages arrive in quick succession (e.g., during a streaming update), each new message gets its own enter animation. The auto-scroll logic should handle batched updates without visual jitter.

**Very long messages.** Messages are constrained to `max-width: min(640px, 80%)` by the bubble CSS. Long text wraps naturally. No truncation or "show more" behavior in v1.

**Copy failure.** If `navigator.clipboard.writeText` fails (e.g., insecure context), the copy button should not throw. Fail silently — no error toast in v1.

**Keyboard accessibility.** The copy button and Retry button must be focusable and activatable via keyboard (Enter/Space). They should have appropriate `aria-label` attributes ("Copy message" and "Retry message" respectively).

## Non-goals

**No data fetching.** These components do not call Convex queries. Parent pages map backend data into `ChatMessage[]`.

**No markdown rendering.** Messages render as plain text. Markdown support may be added later as a wrapper or plugin; these components render `content` as-is.

**No file/image attachments.** Out of scope for v1.

**No message editing or deletion.** Messages are immutable once sent.

**No emoji picker or rich text formatting.** MessageInput is a plain textarea.

**No Draft Coach integration.** The Draft Coach panel (a separate ticket) will compose `MessageInput` and `ChatWindow` but the "Draft with Coach" button and side-panel UX are not part of this ticket.

**No optimistic UI for sending.** The `onSend` callback fires and the parent page is responsible for optimistic updates. MessageInput clears itself after calling `onSend`; it does not render a pending state.

## Test coverage

**`tests/unit/chat-window.test.tsx`** covers three ACs:

1. _ChatWindow scrollable container with ARIA_ — render with a messages array, assert `role="log"` and `aria-live="polite"` on the scroll container.
2. _Sticky auto-scroll_ — render with enough messages to overflow, verify scroll is at bottom. Simulate user scrolling up, add a new message, verify scroll does NOT jump. Reset to bottom, add another, verify it follows.
3. _New message fade-in animation_ — add a message, assert the wrapper element has the `cc-bubble-enter` animation class. Verify it does not re-apply on subsequent re-renders.

**`tests/unit/message-bubble.test.tsx`** covers four ACs:

1. _7 variants_ — render each of the 7 `BubbleVariant` values, assert the correct CSS class (`.cc-bubble`, `.cc-bubble-coach`, etc.) is present.
2. _Status rendering_ — render with STREAMING: assert `cc-streaming-cursor` present, no copy button. Render with COMPLETE: assert copy button + timestamp. Render with ERROR: assert error styling + Retry button.
3. _Timestamps on hover_ — render a COMPLETE bubble, assert timestamp is hidden by default, simulate hover, assert timestamp becomes visible.
4. _Copy button only on COMPLETE_ — render STREAMING: no copy button. Render COMPLETE: copy button present. Render ERROR: no copy button.

**`tests/unit/message-input.test.tsx`** covers one AC:

1. _Enter/Shift-Enter + disabled state_ — simulate Enter keypress: assert `onSend` called. Simulate Shift+Enter: assert newline inserted, `onSend` not called. Set `isAiResponding=true`: assert Send button disabled, textarea still enabled. Assert empty input does not trigger `onSend`.

**`tests/unit/streaming-indicator.test.tsx`** covers one AC:

1. _StreamingIndicator rendering_ — render the component, assert a `<span>` with `cc-streaming-cursor` class exists.

All tests are unit-only because every component is presentational and receives mock data via props. No backend interaction means no e2e coverage is needed for this ticket specifically.
