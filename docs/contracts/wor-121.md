# Contract: WOR-121 — CaseDetail orchestrator page — routes by case status

## Why this work exists

The dashboard navigates to `/cases/:caseId` when a user clicks a case row. Today that route renders a placeholder stub. This ticket replaces the stub with a real orchestrator page that reads the case's live status from Convex and renders the correct subview — private coaching (or the invitee's initial perspective form), ready-for-joint, joint chat, or closed. This ensures the UI always reflects the server-side state machine, preventing users from viewing a phase the case hasn't reached.

## Files and exports

### `src/routes/CaseDetailPage.tsx` (create, connected)

The main orchestrator. It is a **connected component** — it calls `useQuery(api.cases.get)` and `useQuery(api.cases.partyStates)` internally and renders the correct subview based on `caseDoc.status`. The top-level export is `CaseDetailPage` which takes no props (it reads `caseId` from `useParams`).

Internal structure:
- `CaseDetailInner` — receives a typed `caseId`, subscribes to queries, handles the status→subview switch.
- `statusToPhase(status)` — pure helper mapping each `CaseStatus` to `{ phaseName: string; subroute: string }`. The mapping:
  - `DRAFT_PRIVATE_COACHING` / `BOTH_PRIVATE_COACHING` → `{ phaseName: "Private Coaching", subroute: "private" }`
  - `READY_FOR_JOINT` → `{ phaseName: "Ready for Joint Session", subroute: "joint" }` (Note: subroute is "joint" because from the user's perspective they'll advance into joint)
  - `JOINT_ACTIVE` → `{ phaseName: "Joint Discussion", subroute: "joint" }`
  - `CLOSED_RESOLVED` / `CLOSED_UNRESOLVED` / `CLOSED_ABANDONED` → `{ phaseName: "Closed", subroute: "closed" }`
- `InviteeFormView` — inline form component for the invitee perspective (category is inherited from the case; form fields: mainTopic, description, desiredOutcome). Calls `api.cases.updateMyForm` on submit. After submission, the partyState query re-fires and the orchestrator re-renders into PrivateCoachingView.

The component renders `<PhaseHeader>` (imported from `src/components/layout/PhaseHeader.tsx`) with `caseName={caseDoc.category}` and `phaseName` derived from `statusToPhase`. For solo cases, `<PartyToggle>` is rendered inside PhaseHeader's children slot, following the same pattern as `CasePrivatePage.tsx`.

**Subview rendering by status:**
- Private coaching statuses: if the current user is the invitee and `partyState.self.formCompletedAt` is null, render `<InviteeFormView>`. Otherwise, render `<Navigate to={/cases/${caseId}/private}>` (delegates to the existing `CasePrivatePage` route).
- `READY_FOR_JOINT`: render a placeholder `<ReadyForJointView>` (a simple div with `data-testid="subview-ready-for-joint"` and heading text).
- `JOINT_ACTIVE`: render a placeholder `<JointChatView>` (a simple div with `data-testid="subview-joint-chat"` and heading text).
- Closed statuses: render a placeholder `<ClosedCaseView>` (a simple div with `data-testid="subview-closed"` and heading text).

**Non-party error handling:** When `useQuery(api.cases.get)` throws a ConvexError with code `FORBIDDEN`, the component catches this and redirects to `/dashboard` with an error toast. Implementation: wrap the query error via a `try/catch` in an effect, or use the Convex error boundary pattern. The toast uses shadcn/ui's `useToast` or a simple inline approach consistent with the project.

### `src/App.tsx` (modify, route)

Replace the inline `CaseDetailPage` stub function with an import of `CaseDetailPage` from `src/routes/CaseDetailPage.tsx`. The route entry at `/cases/:caseId` stays the same — just the component changes. No new routes are added; the existing subroutes (`/private`, `/joint`, `/closed`) remain.

## Data dependencies

### `api.cases.get`
Returns the full case document: `{ _id, status, isSolo, category, initiatorUserId, inviteeUserId, createdAt, updatedAt, ... }`. The orchestrator consumes `status` (for subview routing), `isSolo` (for solo mode toggle), `category` (for PhaseHeader), and `initiatorUserId`/`inviteeUserId` (to determine if current user is the invitee). Throws `FORBIDDEN` if caller is not a party — this is the auth gate.

### `api.cases.partyStates`
Returns `{ self: PartyStateDoc, other: { role, hasCompletedPC } | null }`. The orchestrator reads `self.formCompletedAt` to decide whether to show the invitee form. Also reads `self.role` to know if the current user is the invitee. Accepts optional `viewAsRole` for solo mode.

### `api.cases.updateMyForm`
Mutation accepting `{ caseId, mainTopic, description, desiredOutcome }`. Called by the invitee form on submit. Patches the partyState with the form fields. Note: this mutation does NOT set `formCompletedAt` — the implementation must also patch `formCompletedAt: Date.now()` (this is an existing gap in `updateMyForm` that this ticket should address by adding `formCompletedAt` to the mutation, or by using a separate mutation). **Decision:** The invitee form will call `api.cases.updateMyForm` extended to also set `formCompletedAt` if it's the first submission. The implementation author should add `formCompletedAt` patching to `updateMyForm` when all three form fields are provided and `formCompletedAt` is not yet set.

## Invariants

### Status-to-subview mapping is exhaustive
Every `CaseStatus` value from the schema maps to exactly one subview. The `statusToPhase` helper handles all seven statuses. If a new status is added to the schema without updating this mapping, TypeScript's exhaustive check (a `never` default case) should catch it at compile time.

### Non-party users never see case content
The `api.cases.get` query calls `requirePartyToCase` server-side. If the user is not a party, the query throws `FORBIDDEN`. The component must handle this error and redirect to `/dashboard` with an error message. At no point does the component render case content before the query resolves successfully.

### Subroute mismatch redirects
If a user navigates directly to `/cases/:caseId/joint` but the case is in `BOTH_PRIVATE_COACHING`, the orchestrator at `/cases/:caseId` would redirect to `/private`. However, note that `/cases/:caseId/joint` is a *separate route entry* in App.tsx pointing to the stub `CaseJointPage`. The subroute redirect logic therefore lives in the *subroute pages themselves* (not in CaseDetailPage). **Clarification:** CaseDetailPage handles the bare `/cases/:caseId` route. For subroute mismatch, each subroute page (CasePrivatePage, CaseJointPage, CaseClosedPage) should independently validate that the case status matches and redirect if not. This ticket adds that validation to CaseDetailPage's navigation (it always `<Navigate>`s to the correct subroute from the bare route), but it does NOT modify the subroute pages themselves to add mismatch checks — that's either already handled or a follow-up.

### Invitee form gating
The invitee form renders only when: (1) case status is `DRAFT_PRIVATE_COACHING` or `BOTH_PRIVATE_COACHING`, AND (2) the current user's role is `INVITEE` (determined from partyStates), AND (3) `self.formCompletedAt` is null/undefined. All three conditions must be true. If any is false, the normal subview renders.

### Reactive updates
Because `useQuery(api.cases.get)` is a Convex reactive subscription, any server-side status transition (e.g., both parties complete coaching → status changes to `READY_FOR_JOINT`) automatically triggers a re-render. The orchestrator re-evaluates the status and renders the new subview. No explicit polling or refetching is needed.

## Edge cases

### Loading state
While `useQuery(api.cases.get)` returns `undefined` (initial load), the component renders `<LoadingSpinner />` (imported from `src/components/layout/LoadingSpinner.tsx`), matching the existing pattern in `CasePrivatePage.tsx`.

### Case not found
If the case ID in the URL doesn't correspond to a real case, `api.cases.get` throws `NOT_FOUND`. The component handles this the same way as `FORBIDDEN` — redirect to `/dashboard` with an error message.

### Invalid caseId param
If `useParams` returns no `caseId`, the component renders an "Invalid case URL" message, matching the guard pattern in `CasePrivatePage.tsx`.

### Invitee form validation
The perspective form requires all three fields (mainTopic, description, desiredOutcome) to be non-empty before submission. Client-side validation only — the mutation will also reject empty strings.

### Focus management on phase transitions (NFR-A11Y)
When the case status changes and the subview switches, focus should move to the new primary heading. This can be done with a `useEffect` that calls `.focus()` on the heading element when the status changes.

## Non-goals

- **ReadyForJointView, JointChatView, ClosedCaseView implementations** — these are stub placeholders in this ticket. Their real implementations are separate tickets (later in the epic plan).
- **Subroute pages adding their own status-mismatch redirects** — this ticket only handles the bare `/cases/:caseId` route. If a user bookmarks `/cases/:caseId/joint` and the case regresses, the subroute page is responsible for its own guard (future ticket or already done).
- **New Convex queries** — this ticket consumes existing queries only. The one modification is extending `updateMyForm` to set `formCompletedAt`.
- **Toast infrastructure** — if shadcn/ui toast is not yet wired up, use a simple inline error banner on redirect. Do not build a full toast system for this ticket.

- **AC: PhaseHeader phase name** — For each status group, assert the PhaseHeader renders the correct `phaseName` text.
- **AC: Invitee form** — Mock partyStates with `self.formCompletedAt = undefined` and `self.role = "INVITEE"`, assert the invitee form renders. After simulating form submission, assert the form disappears (partyStates query would re-fire).
- **AC: Loading state** — Mock `useQuery` returning `undefined`, assert `LoadingSpinner` renders.

- **AC: Invitee form** — Log in as invitee who hasn't completed form, navigate to case, assert form is shown. Submit the form, assert PrivateCoachingView renders.
