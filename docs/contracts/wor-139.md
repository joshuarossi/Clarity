# Contract: WOR-139 — Fix typecheck regression in convex/users.ts (closure-narrowing on identity.email)

## Why this work exists

PR #14 (WOR-102, App shell) added a guard `if (!identity.email) return null;` to the `me` query in `convex/users.ts`. The guard is semantically correct, but TypeScript's flow-narrowing does not survive into the callback passed to `.withIndex()` — the compiler pessimistically widens `identity.email` back to `string | undefined` inside the closure. This causes `tsc --noEmit` to fail with TS2345 on line 15. The fix is mechanical: extract the narrowed value to a local `const email` before the closure captures it.

## Files and exports

### `convex/users.ts` (query, modify)

This is the only file that changes. The `me` query export keeps its identical public API: `args: {}`, returns `Doc<"users"> | null`. The internal change is:

**Before (lines 10-15):**
```ts
if (!identity.email) {
  return null;
}
const user = await ctx.db
  .query("users")
  .withIndex("by_email", (q) => q.eq("email", identity.email))
```

**After:**
```ts
const email = identity.email;
if (!email) {
  return null;
}
const user = await ctx.db
  .query("users")
  .withIndex("by_email", (q) => q.eq("email", email))
```

The key decision: extract `identity.email` into `const email` **before** the guard, then guard on `email`. After the guard, `email` is narrowed to `string`, and closures capture `email` (a local const) rather than `identity.email` (a property access that TS cannot narrow across closure boundaries).

## Data dependencies

### `api.users.me`

This IS the query being fixed. It reads the current user's identity via `ctx.auth.getUserIdentity()`, then looks up the user row in the `users` table via the `by_email` index. Returns the full user document or `null`. No other queries or mutations are called.

## Invariants

**Closure-narrowing fix:** The `email` local const must be declared and assigned from `identity.email` BEFORE the `if (!email) return null` guard. The guard then narrows `email` from `string | undefined` to `string`. The `.withIndex()` callback captures this local const, which TypeScript knows is `string` — unlike `identity.email`, which TypeScript cannot narrow across closure boundaries because the callback might be deferred.

**Auth-guard semantics preserved:** The function must still return `null` in exactly two cases: (1) `identity` is null (not authenticated), (2) `identity.email` is undefined. The refactor changes where the value is read from, not what values are checked.

**No public API change:** The export name (`me`), the `args` validator (`{}`), and the return type (`Doc<"users"> | null`) are all unchanged. Downstream consumers (e.g., `AdminRoute` from WOR-102) are unaffected.

## Edge cases

**No identity (unauthenticated):** Returns `null` — unchanged from current behavior. The `identity.email` extraction happens after the `if (!identity)` guard, so there is no risk of accessing `.email` on null.

**Identity exists but email is undefined:** Returns `null` — unchanged. The `if (!email) return null` guard handles this identically to the current `if (!identity.email) return null`.

**Identity exists with valid email but no user row:** Returns `null` from `.unique()` — unchanged. This path is unaffected by the refactor.

## Test coverage

All three ACs are covered by a single unit test file: **`tests/unit/users-typecheck.test.ts`**.

- **AC1 (tsc --noEmit passes):** A test invokes `tsc --noEmit` programmatically (via `execSync` or similar) and asserts zero errors, or alternatively imports the module and asserts TypeScript compilation succeeded. Given the project uses `convex-test`, the more practical approach is: the test file imports `me` from `convex/users.ts` and exercises it with `convex-test` — if the file has type errors, the test suite itself will fail to compile. This provides implicit typecheck coverage.

- **AC2 (auth-guard semantics preserved):** The test uses `convex-test` to exercise `me` in three scenarios: (1) no identity → returns null, (2) identity with no email → returns null, (3) identity with email and matching user row → returns the user doc. These directly verify the guard semantics are intact.

- **AC3 (existing tests continue to pass):** The existing `tests/unit/auth.test.ts` already tests the auth helpers that depend on the `users` table and `by_email` index. Those tests are not modified and continue to pass. The contract does not create a separate test for this AC — it is verified by running the full test suite (`npm test`), which the CI pipeline does automatically.
