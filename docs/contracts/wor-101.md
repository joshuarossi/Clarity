# Contract: WOR-101 — Transcript compression module (convex/lib/compression.ts)

## Why this work exists

As conversation transcripts grow longer across private coaching, joint chat, and draft coaching sessions, they risk exceeding the context-window budget that each AI action relies on. Generation actions have a 60,000-token budget; classification actions have 10,000. Without compression, long conversations would either be truncated (losing context) or fail entirely. This module implements TechSpec §6.4's deliberately simple compression strategy: detect when messages exceed budget, summarize the oldest half via Haiku, and cache the result. Both generation and classification action callers will use this module before assembling prompts.

## Files and exports

### `convex/lib/compression.ts` (create, helper)

A helper module alongside the other `convex/lib/` helpers (`prompts.ts`, `privacyFilter.ts`, `errors.ts`, `auth.ts`). Unlike those modules, this one is **async** because it calls the Anthropic API (Claude Haiku) for summarization. It has no Convex runtime dependencies — no imports from `convex/server` or `convex/_generated`. The Anthropic client is injected via an optional `options.client` parameter for testability (callers pass their SDK instance; tests pass a mock).

Exports:

- **`CompressibleMessage`** — the input/output message shape: `{ role: string; content: string }`. Uses `string` for `role` (not a union) because the module handles messages from multiple sources (private coaching, joint chat) which have different role vocabularies, plus it produces synthetic `"system"` role SUMMARY messages. This type is intentionally minimal and compatible with the `PromptMessage` type from `convex/lib/prompts.ts`.

- **`estimateTokens`** — a synchronous function that approximates token count for a string. Uses a word-count-based heuristic (split on whitespace, divide by 0.75 to approximate Claude tokenization). This is the "token-counting utility" referenced in the implementation notes. It is deliberately approximate — exact tokenization would require a tokenizer library, which is overkill for budget-checking purposes. Exported so callers can also use it for pre-compression budget checks.

- **`compressTranscript`** — the main async function. Accepts a message array, a budget in tokens, and an optional options object containing an Anthropic client. If total tokens (summed via `estimateTokens` over all message contents) is within budget, returns the input unchanged. Otherwise, takes the oldest `Math.ceil(messages.length / 2)` messages, hashes their concatenated content with SHA-256, checks the in-memory cache, and either returns the cached result or calls Haiku to generate a summary. The compressed array is: `[summaryMessage, ...newestHalfOfMessages]`.

- **`COMPRESSION_PROMPT`** — the verbatim prompt string from TechSpec §6.4: `"Summarize this conversation segment in 500 tokens or fewer, preserving facts, decisions, emotional tone, and unresolved threads."` Exported as a constant so tests can assert prompt fidelity by reference.

**Why the Anthropic client is injected, not imported directly:** This follows the same pattern the codebase will use for all AI-calling modules. The module doesn't `import Anthropic from "@anthropic-ai/sdk"` at the top level — instead, the caller constructs the client (with the API key from environment) and passes it in. This makes the module trivially testable with a mock client and avoids import-time side effects.

**Why `estimateTokens` uses a heuristic, not a tokenizer:** The implementation notes say "the project already has a token-counting utility." No such utility exists in the codebase yet. Rather than adding a dependency on a tokenizer library (like `tiktoken`), this module provides its own lightweight estimator. The heuristic (word count / 0.75) overestimates slightly, which is safe — it triggers compression a bit earlier than strictly necessary, but never allows budget overruns. Callers needing exact counts in the future can swap in a precise tokenizer.

**Cache implementation:** A module-level `Map<string, string>` keyed by SHA-256 hex digest of the concatenated content of the messages being compressed. The value is the summary string. Since Convex actions are short-lived processes, this in-memory cache lives for the duration of a single action invocation. Its primary value is avoiding redundant Haiku calls when the same action compresses the same segment multiple times (e.g., during retry logic).

### `tests/unit/compression.test.ts` (create, test-infrastructure)

Unit tests using Vitest. The module is async but has no Convex runtime dependencies, so tests import directly and use a mock Anthropic client — no `convex-test` needed. Follows the project convention of `tests/unit/<module>.test.ts` (matching `tests/unit/prompts.test.ts`, `tests/unit/privacyFilter.test.ts`, etc.).

## Data dependencies

None from Convex. This module receives all data via parameters and returns results. It does not call any Convex queries, mutations, or actions.

The only external dependency is the Anthropic API (Claude Haiku), accessed via the injected client. The client interface needed:

```typescript
// Minimal interface the mock must satisfy:
interface AnthropicClient {
  messages: {
    create(params: {
      model: string;
      max_tokens: number;
      messages: Array<{ role: string; content: string }>;
    }): Promise<{ content: Array<{ text: string }> }>;
  };
}
```

The module calls `client.messages.create` with `model: "claude-haiku-4-5-20251001"`, `max_tokens: 600` (slightly above the 500-token instruction to avoid truncation), and a single user message containing the concatenated content of the oldest 50% of messages, preceded by the `COMPRESSION_PROMPT` as the system instruction.

## Invariants

### Pass-through when under budget

If the total estimated token count of all messages is at or below `budgetTokens`, the function returns the input array unchanged. This must be a deep equality — same array contents, same order. No Haiku API call is made. This is the most common path (most conversations fit in budget).

### Oldest 50% replaced with SUMMARY

When compression fires, exactly `Math.ceil(messages.length / 2)` messages from the beginning of the array are removed and replaced with a single synthetic message. The "oldest" messages are the first elements of the array (index 0 through `Math.ceil(length/2) - 1`). The newest messages (the remainder) are preserved unchanged and in order.

### SUMMARY message format

The synthetic replacement message has `role: "system"` and `content` prefixed with `"SUMMARY: "` followed by the Haiku-generated summary text. This marker lets downstream consumers (prompt assembly, UI rendering) distinguish summaries from real user/AI messages.

### Prompt fidelity

The prompt sent to Haiku is exactly: `"Summarize this conversation segment in 500 tokens or fewer, preserving facts, decisions, emotional tone, and unresolved threads."` This is specified verbatim in TechSpec §6.4. The module must not paraphrase, extend, or modify this prompt.

### Cache correctness

The cache key is the SHA-256 hex digest of the concatenated `content` fields of the messages being compressed (joined with a delimiter like `\n`). If the same set of messages is compressed again within the same action invocation, the cached summary is returned without calling Haiku. The returned compressed arrays from both calls must be deeply equal.

### Module purity (no Convex dependencies)

The module imports nothing from `convex/server`, `convex/_generated`, or any Convex package. It uses only standard Node.js APIs (`crypto` for SHA-256) and the injected Anthropic client. This makes it testable without any Convex runtime setup.

## Edge cases

### Single message over budget

If the message array has exactly one message and its token count exceeds the budget, the oldest 50% is `Math.ceil(1/2) = 1` message — the entire array. The result is a single SUMMARY message. This is a degenerate case but the math works correctly.

### Two messages over budget

`Math.ceil(2/2) = 1` message is compressed. The first message becomes the summary; the second is preserved. This is the minimal non-trivial compression case.

### Odd number of messages

`Math.ceil(n/2)` ensures the oldest half is the larger half when `n` is odd. For 5 messages, 3 are compressed and 2 are preserved. This is correct — we err on the side of compressing more to stay within budget.

### Empty message array

An empty array has zero tokens, which is under any positive budget. Returns the empty array unchanged.

### Haiku API failure

Not handled by this module. If the Anthropic client throws, the error propagates to the caller. The caller (a Convex action) is responsible for retry logic per TechSpec §6.5 (retry once with backoff, then surface error).

### Budget of zero

If `budgetTokens` is 0 or negative, any non-empty message array will trigger compression. The module does not validate budget values — callers are expected to pass sensible defaults (60,000 or 10,000 per TechSpec §6.4).

## Non-goals

**No persistent caching.** The in-memory `Map` lives only for the duration of a single Convex action invocation. Persistent caching (e.g., storing summaries in a Convex table keyed by content hash) can be layered later if needed. For v1, action-scoped caching is adequate.

**No vector search or RAG.** TechSpec §6.4 explicitly calls this out as "deliberately deterministic — no vector search, no RAG, nothing clever." This module is simple on purpose.

**No recursive compression.** If the compressed result (SUMMARY + newest half) still exceeds the budget, the module returns it as-is. The caller can re-invoke compression, but the module does not loop. This keeps the implementation simple and predictable.

**No streaming.** The Haiku call is a standard `messages.create`, not a streaming call. Summaries are short (500 tokens) and not user-facing, so streaming adds no value.

**No Convex integration.** The module doesn't read from or write to the database. Callers fetch messages, call `compressTranscript`, and use the result in prompt assembly. The module is a pure transformation.

## Test coverage

All tests live in `tests/unit/compression.test.ts`, following the project convention of `tests/unit/<module>.test.ts`.

**AC 1 (under budget → pass-through) → `tests/unit/compression.test.ts` (unit).** Supply a message array whose total estimated token count is below `budgetTokens`. Assert the returned array is deeply equal to the input. Assert the mock Anthropic client was never called (zero invocations).

**AC 2 (over budget → compression) → `tests/unit/compression.test.ts` (unit).** Supply a message array that exceeds `budgetTokens`. Mock the Haiku API to return a canned summary string. Assert: (a) the returned array's first element has `role: "system"` and `content` starting with `"SUMMARY: "` containing the canned summary, (b) the oldest `Math.ceil(n/2)` messages are absent, (c) the newest messages remain intact and in order, (d) the Haiku mock was called exactly once.

**AC 3 (prompt fidelity) → `tests/unit/compression.test.ts` (unit).** When compression fires, capture the arguments passed to the mock Haiku client. Assert that the system message or prompt contains the exact string: `"Summarize this conversation segment in 500 tokens or fewer, preserving facts, decisions, emotional tone, and unresolved threads."` Also assert that the `COMPRESSION_PROMPT` exported constant equals this string.

**AC 4 (cache hit) → `tests/unit/compression.test.ts` (unit).** Call `compressTranscript` twice with the same messages and budget. Assert the Haiku mock was invoked exactly once (the second call returns the cached result). Assert both return values are deeply equal.

**AC 5 (comprehensive Vitest suite) → `tests/unit/compression.test.ts` (unit).** This AC is a meta-requirement that the other ACs collectively satisfy. The test file covers: pass-through (under budget), compression (over budget), cache hit (no redundant API call). Additionally, test `estimateTokens` directly: verify it returns a positive number for non-empty strings and 0 for empty strings, and that its estimates are in the right ballpark (e.g., "hello world" ≈ 2-4 tokens).
