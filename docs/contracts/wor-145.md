# Contract: WOR-145 — @-mention the Coach is unreachable in joint chat (US-10c)

## Why this work exists

The PRD (US-10c) specifies that parties can @-mention the Coach to force an intervention. The backend action `generateCoachResponse` already accepts a `triggerType` argument and correctly bypasses the NORMAL_EXCHANGE suppression gate when `triggerType === "mention"`. However, `sendUserMessage` never passes `triggerType` and no @-mention parsing exists anywhere in the codebase. This ticket connects the existing dead-code path to the UI by adding mention detection in the mutation layer.

## Files and exports

### `convex/lib/mentionDetect.ts` (create)

A pure helper module exporting `detectCoachMention`. It performs a case-insensitive regex match for `@coach` at a word boundary. The regex: `/(?:^|\s)@coach\b/i`. This lives in `convex/lib/` alongside other helpers (`auth.ts`, `errors.ts`, `privacyFilter.ts`). It is a pure function with no Convex context dependency, making it trivially unit-testable.

### `convex/jointChat.ts` (modify — `sendUserMessage` mutation only)

The only change is in the `sendUserMessage` handler. After inserting the joint message, before scheduling `generateCoachResponse`, it calls `detectCoachMention(content)`. If true, it passes `triggerType: "mention"` in the scheduler args. If false, it omits `triggerType` (preserving existing default behavior). No other functions in this file change.

### `src/routes/JointChatView.tsx` (modify — minimal)

Update the `MessageInput` placeholder from `"Type a message..."` to `"Type a message... (use @Coach to ask the facilitator)"` to surface the affordance. This is a single string change.

## Data dependencies

- **`api.jointChat.sendUserMessage`** — the mutation the client calls. Its external args signature (`caseId`, `content`, `viewAsRole?`) does not change. Internal behavior changes: it now conditionally passes `triggerType` downstream.
- **`internal.jointChat.generateCoachResponse`** — the internal action already accepts `triggerType?: "message" | "mention" | "timer"`. No changes needed to this function; it already handles `"mention"` correctly at line 556.

## Invariants

1. **Mention triggers forced response**: When a message matches the `@coach` pattern, `generateCoachResponse` is called with `triggerType: "mention"`, which bypasses the `NORMAL_EXCHANGE` suppression gate. The Coach will always respond to mentions regardless of classification.

2. **No false positives on non-mentions**: The regex requires a word boundary before "coach" preceded by `@`. Text like "email@coach.com" should NOT match (the `(?:^|\s)` prefix requires start-of-string or whitespace before `@`). However, intentional mentions like `@Coach please summarize` or `hey @coach what do you think` must match.

3. **Backward compatibility**: Messages without `@Coach` continue to flow through the existing path with no `triggerType` argument (defaults to `"message"` in `generateCoachResponse`). Zero behavioral change for non-mention messages.

4. **Detection is server-side only**: The regex runs in the Convex mutation, not on the client. This ensures the gate cannot be bypassed by a modified client.

## Edge cases

**Case variations**: `@Coach`, `@coach`, `@COACH`, `@CoAcH` all match (case-insensitive regex).

**Mention mid-sentence**: `"Hey @Coach can you summarize?"` matches. The mention does not need to be at the start.

**Mention with punctuation**: `"@Coach, summarize please"` matches — the word boundary after "coach" allows trailing punctuation.

**No mention in quoted text**: If a user writes `she said "@Coach help"`, the regex will still match. This is acceptable — the user is still addressing the Coach. There is no need for quoted-text exclusion in v1.

**Empty or whitespace-only content**: `sendUserMessage` already requires non-empty `content` (Convex string validator). `detectCoachMention("")` returns false, which is correct.

## Non-goals

- **No @-mention UI autocomplete**: No dropdown, no typeahead. Detection is invisible to the user beyond the placeholder hint. A richer UX is out of scope for v1.
- **No visual highlighting of @Coach in messages**: The `@Coach` text renders as plain text in the chat. Styled mentions are a design enhancement for later.
- **No multi-target mentions**: Only `@Coach` is recognized. `@Alex` or `@Jordan` do nothing.
- **No changes to `generateCoachResponse` logic**: The action's gate logic already handles `triggerType === "mention"` correctly. This ticket only ensures the caller passes it.
