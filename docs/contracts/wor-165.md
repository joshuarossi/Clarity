# Contract: WOR-165 — Comment out the ci.yml e2e job

## Why this work exists

The Playwright e2e test suite is not yet runnable in CI — it requires a Convex preview deployment with secrets (`CONVEX_DEPLOY_KEY_CI`) that are not configured, and the tests themselves reference infrastructure that isn't fully wired. Because the `e2e` job is the final gate in the CI pipeline (`lint → typecheck → unit → e2e`), its failure causes the entire `ci` workflow to report as failed. The `deploy` workflow triggers only when `ci` completes with `conclusion == 'success'`, so every push to main requires a manual `workflow_dispatch` to deploy. Commenting out the e2e job unblocks automatic deploys until the Playwright suite is ready.

## Files and exports

### `.github/workflows/ci.yml`

The entire `e2e:` job block (lines 63–100 currently) will be commented out using YAML `#` comment syntax. The job definition stays in the file as a visible record of what needs to be re-enabled. The `unit` job remains the terminal job in the chain; no other job references `e2e` via `needs:`.

### `tests/unit/ci-workflow.test.ts`

This file currently asserts that the workflow contains exactly four jobs (`["e2e", "lint", "typecheck", "unit"]`) and validates the e2e job's structure extensively. It must be updated to:
1. Expect only three jobs: `["lint", "typecheck", "unit"]`.
2. Remove or skip all `describe("WOR-108: E2E job", ...)` tests.
3. Update the `"enforces lint -> typecheck -> unit -> e2e chain"` test to only validate `lint → typecheck → unit`.
4. Any test that iterates over all jobs (e.g., "all jobs run on ubuntu-latest") will naturally pass since the e2e job won't be parsed from the commented YAML.

## Data dependencies

None — this is purely a CI configuration change with no runtime data dependencies.

## Invariants

1. **CI must still pass**: The remaining three jobs (lint, typecheck, unit) must form a valid workflow that GitHub Actions executes successfully.
2. **Deploy auto-triggers**: With CI passing, `deploy.yml`'s `workflow_run` trigger with `conclusion == 'success'` will fire automatically on main pushes.
3. **Chain intact**: The `needs` chain `lint → typecheck → unit` is preserved. The unit job must NOT gain a `needs: e2e` or any broken reference.
4. **Commented, not deleted**: The e2e job block remains in the file as comments so it can be uncommented when Playwright CI is ready. This makes it trivial to restore.

## Edge cases

### Partial commenting

Every line of the e2e job block must be commented — YAML does not support block comments. Missing a single line will produce invalid YAML and break the workflow entirely.

### Test file parsing

The test file parses `ci.yml` with `js-yaml`. After commenting out the e2e job, `js-yaml` will parse the file and see only three jobs. Tests that previously accessed `workflow.jobs["e2e"]` will get `undefined` — these must be removed, not left to fail.

## Non-goals

- **Fixing Playwright CI**: That's a separate ticket. The e2e tests, config, and test files remain untouched.
- **Modifying deploy.yml**: The deploy workflow is already correctly configured; it just needs CI to stop failing.
- **Removing e2e test files**: The `e2e/` directory and all its `.spec.ts` files remain in place for local development and future CI enablement.

## Test coverage

All three ACs are covered by `tests/unit/ci-workflow.test.ts`:
- The test parses the YAML and asserts only three jobs exist (validating e2e is commented out).
- The chain test validates that the remaining jobs form a passing pipeline (which unblocks deploy).
- The test file itself is the artifact being modified, so its successful execution validates AC3.

Layer: **unit only**. No e2e coverage needed — this is a CI infrastructure change verified by parsing the YAML file.
