# Contract: WOR-148 — Unilateral case close sends no notification

## Why this work exists

The ClosureModal UI tells the closing party that the other party "will be notified" when they walk away. The `unilateralClose` mutation in `convex/jointChat.ts` performs the status transition but never inserts a `notifications` row. The mechanism exists (used by `convex/abandonedCases.ts`) but is simply missing from this code path. This ticket closes that gap so the UI's promise is truthful.

## Files and exports

### `convex/jointChat.ts` (modify)

The `unilateralClose` mutation handler is extended. After the existing `ctx.db.patch(caseId, ...)` call that transitions status to `CLOSED_UNRESOLVED`, add a `ctx.db.insert("notifications", ...)` call that creates a notification for the other party.

The other party's userId is determined from the case document:

- If `user._id === caseDoc.initiatorUserId`, the other party is `caseDoc.inviteeUserId`.
- Otherwise, the other party is `caseDoc.initiatorUserId`.
- In solo mode (`caseDoc.isSolo === true`), both roles belong to the same user. The notification is still inserted (using `caseDoc.initiatorUserId` since it's the same user) to keep behavior uniform and testable.
- If `inviteeUserId` is undefined (invitee never joined), no notification is inserted — there is no user to notify.

No new exports are added. The existing `unilateralClose` export signature is unchanged.

## Data dependencies

The mutation already reads the `cases` document (via `requirePartyToCase`). It needs `caseDoc.initiatorUserId` and `caseDoc.inviteeUserId` to determine the notification recipient. Both fields are already available from the existing `requirePartyToCase` call which returns the case doc.

The `notifications` table schema (from `convex/schema.ts`):

```ts
{
  userId: v.id("users"),       // recipient
  caseId: v.id("cases"),
  type: v.string(),            // "CASE_CLOSED_UNRESOLVED"
  read: v.boolean(),           // false
  createdAt: v.number(),       // Date.now()
}
```

This matches the shape used by `abandonedCases.ts` (type `"CASE_ABANDONED"`).

## Invariants

**Notification targets the other party, never the caller.** The caller initiated the walk-away — they don't need to be notified of their own action. The recipient is always the non-calling party.

**Exactly one notification per unilateral close.** The mutation inserts one row. If the invitee has not yet joined (`inviteeUserId` is undefined), zero notifications are inserted — there is no recipient.

**Atomic with the status transition.** Because both the `ctx.db.patch` (status) and `ctx.db.insert` (notification) happen within the same Convex mutation, they are transactional. Either both succeed or neither does.

## Edge cases

**Invitee never joined:** If `caseDoc.inviteeUserId` is `undefined`, skip the notification insert. There is no user to notify. The state machine already allows unilateral close from `JOINT_ACTIVE` which requires both parties to have joined, but defensive code should still handle this.

**Solo mode:** Both roles are the same user. The notification is still inserted targeting that user. This keeps tests simple and avoids a special case. A future notification UI can filter or annotate solo-mode notifications if desired.

**Concurrent closure attempts:** The `validateTransition` call will throw if the case is already closed (idempotent protection via the state machine). No duplicate notifications possible.

## Non-goals

- **No notification display UI.** There is no bell icon, notification drawer, or badge in this ticket. The `notifications` table is a write-only sink for now; a future ticket will add read surfaces.
- **No email/push delivery.** US-16 (email notifications) is P2/deferred. This ticket only creates the database record.
- **No ClosureModal UI changes.** The copy "{otherPartyName} will be notified" is already correct. The bug is backend-only.

## Test coverage

The single AC ("the other party is notified") is verified in `tests/unit/joint-chat.test.ts`:

- **Unit test (convex-test):** Call `unilateralClose` mutation on a `JOINT_ACTIVE` case. Assert that a `notifications` row is inserted with `type: "CASE_CLOSED_UNRESOLVED"`, `userId` equal to the other party's ID, `read: false`, and the correct `caseId`. Also assert no notification is inserted when `inviteeUserId` is undefined.

This is testable at the unit layer because `convex-test` provides full database access to verify the inserted row. No e2e test is needed — the behavior is purely a database write within a mutation, with no UI surface to verify against.
