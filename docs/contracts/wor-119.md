# Contract: WOR-119 — PrivateCoachingView page

## Why this work exists

Private coaching is the core of Clarity's value proposition — the confidential, one-on-one AI conversation where each party articulates their perspective before the joint session. The backend (WOR-117: queries/mutations, WOR-118: AI streaming action) and shared chat components (T10: ChatWindow, MessageBubble, MessageInput) are already implemented. This ticket builds the page that wires them together: the `/cases/:caseId/private` route that each party visits to have their private conversation with the AI coach.

## Files and exports

### `src/routes/CasePrivatePage.tsx` (create, connected)

The public export is the **connected** component `CasePrivatePage`. It takes no props — all data comes from route params and Convex hooks. This follows the established project convention: every page-level component in `src/routes/` is connected (see `DashboardPage`, `InviteAcceptPage`, `InviteSharingPage`, `NewCasePage`).

The page orchestrates five concerns:

1. **Privacy banner** — Uses the existing `PrivacyBanner` component from `src/components/ui/PrivacyBanner.tsx`, passing the other party's display name as copy: `"🔒 This conversation is private to you. {otherPartyName} will never see any of it."`. The Lock icon click opens the privacy explanation dialog (handled internally by `PrivacyBanner`).

2. **Chat window** — Uses the existing `ChatWindow` from `src/components/chat/ChatWindow.tsx`. Maps `Doc<"privateMessages">[]` to `ChatMessage[]`: role `"USER"` → variant `"user"`, role `"AI"` → variant `"coach"`. The `onRetry` and `onCopy` callbacks are passed per-message through the ChatWindow/MessageBubble pipeline. Since ChatWindow currently passes messages to MessageBubble without `onRetry`/`onCopy`, the page will need to provide these callbacks. **Decision: CasePrivatePage wraps ChatWindow and provides per-message onRetry/onCopy by rendering MessageBubble directly instead of relying on ChatWindow's internal map** — OR — the page passes messages with callbacks attached. Given that ChatWindow's existing API only accepts `ChatMessage[]` without callbacks, the page will render the scroll container and MessageBubble instances directly, reusing the auto-scroll logic pattern from ChatWindow. This avoids modifying the shared ChatWindow component for private-coaching-specific needs (onRetry per error message).

   **Revised decision:** The page renders its own message list using `MessageBubble` directly (not ChatWindow), with the same scroll container pattern (role="log", aria-live="polite", auto-scroll). This is the correct choice because:
   - ChatWindow doesn't support per-message `onRetry`/`onCopy` callbacks
   - Private coaching needs `onRetry` on ERROR messages to re-trigger the AI
   - Private coaching needs `onCopy` on COMPLETE messages
   - Modifying ChatWindow to support callbacks would affect all consumers

3. **Message input** — Uses the existing `MessageInput` from `src/components/chat/MessageInput.tsx`. The `isAiResponding` prop is derived from the message list: `messages.some(m => m.status === "STREAMING")`. The `onSend` callback calls `sendUserMessage` mutation.

4. **Mark complete footer** — An internal `MarkCompleteFooter` component rendered below the message input. Uses ghost/secondary button styling (`cc-btn cc-btn-ghost cc-btn-sm` or similar subtle styling) to avoid rushing the user. The text is "Mark private coaching complete". Clicking opens a confirmation dialog.

5. **Confirmation dialog** — An internal `ConfirmCompleteDialog` that uses the existing `Dialog` components from `src/components/ui/Dialog.tsx`. Shows: "You've had {N} messages with the Coach. Ready to move on to the joint session with {otherPartyName}?" where N is the count of user messages (`messages.filter(m => m.role === "USER").length`). Two buttons: "Continue Coaching" (closes dialog) and "Mark Complete" (calls `markComplete` mutation, closes dialog).

6. **Read-only state** — When `self.privateCoachingCompletedAt` is truthy, the input area and footer are replaced by a status banner: "You've completed private coaching. Waiting for {otherPartyName} to finish." (or "Both parties have completed private coaching." if both are done). The chat history remains visible and scrollable but no new messages can be sent.

## Data dependencies

### `api.privateCoaching.myMessages` (existing, reactive)

Returns `Doc<"privateMessages">[]` for the authenticated user on a given case, sorted by `createdAt` ascending. Each message has:
- `_id` — unique ID, used as React key and for retry targeting
- `role` — `"USER"` or `"AI"`, maps to bubble variant
- `content` — message text, rendered in the bubble
- `status` — `"STREAMING"` | `"COMPLETE"` | `"ERROR"`, drives streaming indicator, copy button, and retry button
- `createdAt` — timestamp, displayed on COMPLETE messages
- `userId` — always the authenticated user (privacy invariant)

This query is reactive — new messages from `sendUserMessage` and streaming updates from `generateAIResponse` push to the client automatically via Convex's WebSocket subscription.

### `api.cases.get` (existing, reactive)

Returns the full case document. This page consumes:
- `status` — to determine if the case is in a private-coaching phase (DRAFT_PRIVATE_COACHING or BOTH_PRIVATE_COACHING) or has moved on (read-only gating)
- `initiatorUserId` / `inviteeUserId` — to determine the other party for display name resolution

### `api.cases.partyStates` (existing, reactive)

Returns `{ self, other }` where:
- `self` — the caller's full `partyStates` doc. Fields consumed: `privateCoachingCompletedAt` (non-null = completed, triggers read-only state)
- `other` — `{ role, hasCompletedPC }` or null. Fields consumed: `hasCompletedPC` (for the read-only status message: "Waiting for X" vs "Both complete")

**Resolving the other party's display name:** The `partyStates` query doesn't return names. The page needs the other party's name for the privacy banner and confirmation dialog. Two approaches:
1. Use `cases.get` to get `initiatorUserId`/`inviteeUserId`, determine which is the other party, then call a user lookup query.
2. Derive from `listForDashboard` which already resolves `otherPartyName`.

**Decision:** The page will use `api.cases.get` to get the case doc, determine whether the current user is initiator or invitee, and then look up the other party's user record. However, there is no existing `api.users.getById` public query. To avoid adding a new query, the page will use a new lightweight query `api.cases.otherPartyName` that returns just the other party's display name given a caseId. **Alternatively**, since `listForDashboard` already resolves this and the dashboard page already uses it, we can avoid a new query by passing the name via React Router state from the dashboard navigation. But this is fragile (direct URL navigation won't have state).

**Final decision: add a new query `api.cases.otherPartyName`** that takes `{ caseId }`, authenticates the caller, determines the other party, and returns `{ displayName: string | null }`. This is a small, focused query that serves this page's needs cleanly. It also serves the joint chat page and closed case page which will need the same information.

### `api.cases.otherPartyName` (NEW)

```typescript
export const otherPartyName = query({
  args: { caseId: v.id("cases") },
  handler: async (ctx, { caseId }) => {
    const user = await requireAuth(ctx);
    const caseDoc = await requirePartyToCase(ctx, caseId, user._id);
    const callerIsInitiator = caseDoc.initiatorUserId === user._id;
    const otherUserId = callerIsInitiator ? caseDoc.inviteeUserId : caseDoc.initiatorUserId;
    if (!otherUserId) return { displayName: null };
    const otherUser = await ctx.db.get(otherUserId);
    return { displayName: otherUser?.displayName ?? otherUser?.name ?? null };
  },
});
```

Returns `{ displayName: string | null }`. Null when the invitee hasn't joined yet.

## Invariants

**Privacy: only the caller's messages are ever fetched.** The `myMessages` query uses the `by_case_and_user` index scoped to `[caseId, userId]`. There is no code path in this page that requests, renders, or even has access to the other party's private messages. This is enforced at the Convex query layer — the client cannot bypass it.

**Phase gating: private coaching phase only.** The page checks `caseDoc.status` from `api.cases.get`. If the status is not `DRAFT_PRIVATE_COACHING` or `BOTH_PRIVATE_COACHING`, the page renders in read-only mode — no input, no send, no mark-complete. This handles the case where a user navigates to `/cases/:caseId/private` after the case has progressed past private coaching.

**Read-only after completion.** When `self.privateCoachingCompletedAt` is set (from `api.cases.partyStates`), the message input and mark-complete footer are removed from the DOM (not just disabled), and a status banner replaces them. The chat history remains visible.

**No auto-send for mark-complete.** The mark-complete button is deliberately styled as a subtle footer CTA (ghost/secondary variant), and clicking it opens a confirmation dialog that requires an explicit second click on "Mark Complete". The dialog shows the message count to help the user make an informed decision.

**isAiResponding derivation.** `isAiResponding` is `messages?.some(m => m.status === "STREAMING") ?? false`. This correctly disables the Send button while the AI is streaming, while leaving the textarea enabled for pre-typing.

**Error retry path.** When a message has `status: "ERROR"`, clicking Retry calls `sendUserMessage` with a content like the last user message, which schedules a new `generateAIResponse`. The ERROR message row remains visible. Actually, per the backend implementation, the ERROR status is on the *AI* message row (the one that `generateAIResponse` marked as error). The retry should re-trigger `generateAIResponse` for the case+user. Since there's no public action to re-trigger `generateAIResponse` directly, and `sendUserMessage` would add a duplicate user message, **the contract specifies a new mutation `retryAIResponse`** that:
1. Validates the message is an AI message with status ERROR owned by the caller
2. Deletes the ERROR message row
3. Schedules `generateAIResponse`

**Revised error retry decision:** Rather than adding a new mutation, the simpler approach matching the existing pattern is: the Retry button calls `sendUserMessage` with an empty-ish trigger content (e.g. the user's last message repeated). But this creates a duplicate user message which is wrong.

**Final error retry decision: add `retryLastAIResponse` mutation to `convex/privateCoaching.ts`.** This is the cleanest approach:

```typescript
export const retryLastAIResponse = mutation({
  args: { caseId: v.id("cases") },
  handler: async (ctx, args) => {
    const user = await requireAuth(ctx);
    await requirePartyToCase(ctx, args.caseId, user._id);
    // Find the last AI message with ERROR status for this user
    const messages = await ctx.db
      .query("privateMessages")
      .withIndex("by_case_and_user", (q) =>
        q.eq("caseId", args.caseId).eq("userId", user._id))
      .collect();
    const errorMsg = messages
      .filter(m => m.role === "AI" && m.status === "ERROR")
      .sort((a, b) => b.createdAt - a.createdAt)[0];
    if (!errorMsg) throw conflict("No error message to retry");
    // Delete the error row
    await ctx.db.delete(errorMsg._id);
    // Schedule new AI response
    await ctx.scheduler.runAfter(0, internal.privateCoaching.generateAIResponse, {
      caseId: args.caseId, userId: user._id,
    });
  },
});
```

## Edge cases

**Loading state.** While any of the three queries (`myMessages`, `cases.get`, `partyStates`) return `undefined`, the page renders a loading state. This should be a centered `LoadingSpinner` (from `src/components/layout/LoadingSpinner.tsx`) — not skeleton chat bubbles, since we don't know how many messages exist. The privacy banner can render immediately with a placeholder name while `otherPartyName` loads.

**Empty state (no messages yet).** When `myMessages` returns an empty array, the chat area shows an empty state prompt: "Start your private coaching session by sharing what's on your mind." This guides first-time users. The input is enabled and the mark-complete footer is visible (though marking complete with zero messages is allowed — the user may have decided private coaching isn't needed).

**Error state (query failure).** If `cases.get` throws (e.g. user is not a party to this case), the page should show an error message. In practice, `requirePartyToCase` throws a FORBIDDEN ConvexError. The page catches this via Convex's error boundary behavior and shows "You don't have access to this case."

**AI error message.** When a message has `status: "ERROR"`, it renders with `variant: "error"` in the MessageBubble (which applies `cc-bubble-error` styling) and shows a Retry button. The error message content may be empty string (the AI never produced output). The bubble should show a fallback text like "The AI coach encountered an error. Please retry." when content is empty and status is ERROR.

**Case not in private coaching phase.** If the user navigates to `/cases/:caseId/private` when the case status is `READY_FOR_JOINT`, `JOINT_ACTIVE`, or any `CLOSED_*` status, the page renders in read-only mode with an appropriate status message (e.g. "Private coaching is complete. This conversation is now read-only."). The chat history is still visible.

**Other party hasn't joined yet.** When `otherPartyName` returns `{ displayName: null }`, the privacy banner should use a generic fallback: "This conversation is private to you. The other party will never see any of it."

## Non-goals

**No navigation to joint session.** After marking complete, the page shows a read-only state. It does not automatically redirect to the joint session or show a "Go to joint session" button. That navigation happens from the dashboard or case detail page. The reason: both parties must complete before joint session is available, and this page belongs to one party's private experience.

**No AI synthesis display.** Synthesis is generated server-side after both parties complete private coaching. It's displayed on a separate view (the synthesis/pre-joint view), not on this page.

**No draft coach integration.** Draft coach is a joint-chat feature. Private coaching is a direct user ↔ AI conversation with no drafting step.

**No message editing or deletion.** Messages are immutable once sent. This is by design — the coaching conversation is a record of the exploration process.

**No markdown rendering.** Message content is rendered as plain text in v1. Markdown rendering may come later but is explicitly out of scope.

**No file or image attachments.** Text-only for v1.
