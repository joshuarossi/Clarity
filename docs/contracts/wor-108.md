# Contract: WOR-108 — CI pipeline — GitHub Actions (lint, typecheck, unit, e2e)

## Why this work exists

The Clarity codebase currently has a placeholder CI workflow (`.github/workflows/ci.yml`) that conditionally skips every step if scripts aren't defined yet. Now that ESLint, TypeScript, Vitest, and Playwright are all configured (via prior tickets), the CI pipeline needs to be upgraded from placeholder to production-grade: four sequential jobs that catch lint violations, type errors, unit test failures, and E2E regressions before any code merges to `main`.

## Files and exports

### `.github/workflows/ci.yml` (modify)

The existing file is a single-job placeholder with conditional step execution. This ticket replaces it with a four-job workflow:

1. **`lint`** — Checks out code, installs dependencies with caching, runs `npm run lint` (ESLint) and `npx prettier --check .`. Fails the workflow immediately on any violation.

2. **`typecheck`** — Depends on `lint` via `needs: lint`. Runs `npm run typecheck` (`tsc --noEmit`). Fails on any type error.

3. **`unit`** — Depends on `typecheck` via `needs: typecheck`. Runs `npx vitest run` (non-watch mode). Fails on any test failure.

4. **`e2e`** — Depends on `unit` via `needs: unit`. The most complex job:
   - Installs Playwright browsers (`npx playwright install --with-deps`).
   - Provisions an ephemeral Convex dev deployment using `CONVEX_DEPLOY_KEY` from GitHub secrets.
   - Seeds test data via the existing `convex/seed.ts` script.
   - Sets `CLAUDE_MOCK=true` environment variable.
   - Runs `npx playwright test`.
   - On failure, uploads Playwright HTML report and trace files as GitHub Actions artifacts using `actions/upload-artifact`.

Each job independently checks out code, sets up Node.js LTS with npm caching, and runs `npm ci`. This is necessary because GitHub Actions jobs run on separate runners and don't share filesystems.

No TypeScript exports — this is a YAML configuration file.

## Data dependencies

This ticket has no Convex data dependencies. The E2E job calls `convex/seed.ts` (which was created by prior work) to populate test data, but the CI workflow itself does not define or modify any Convex functions.

## Invariants

**Sequential job gating**: Jobs must chain via `needs:` — `lint` → `typecheck` → `unit` → `e2e`. This ensures fast checks (lint: ~30s, typecheck: ~30s) gate slower checks (unit: ~1-2min, e2e: ~3-5min), minimizing wasted compute on broken PRs.

**No hardcoded secrets**: The `CONVEX_DEPLOY_KEY` (or equivalent token for ephemeral Convex deployments) must come from `${{ secrets.CONVEX_DEPLOY_KEY }}`. No API keys, OAuth secrets, or deploy tokens appear in the YAML file.

**CLAUDE_MOCK=true for E2E**: The E2E job must set this environment variable so that all AI actions route through the deterministic stub responder (from WOR-107's `convex/lib/claudeMock.ts`). Without this, E2E tests would attempt real Anthropic API calls, which would fail (no API key in CI) and introduce non-determinism.

**Artifact upload on failure**: The E2E job uses `if: failure()` on the artifact upload step so that Playwright traces and HTML reports are available for debugging failed runs. This is critical for diagnosing flaky or broken E2E tests in CI where you can't re-run interactively.

**Node.js LTS and caching**: Every job uses `actions/setup-node@v4` with `node-version: 'lts/*'` and `cache: 'npm'` to leverage npm's built-in caching. This is a key factor in meeting the sub-10-minute target.

**Placeholder logic removed**: The current `if [ -f "package.json" ] && npm run | grep -q ...` conditionals are replaced with direct `npm run lint`, `npm run typecheck`, etc. since all scripts now exist in `package.json`.

## Edge cases

**Prettier not installed globally**: The lint job runs `npx prettier --check .` rather than a bare `prettier` command. Since `prettier` is not in `devDependencies` (the project has no explicit Prettier config file), the implementation must either: (a) add `prettier` to `devDependencies` and add a `format:check` script, or (b) use `npx prettier --check .` which downloads it on-the-fly. The contract specifies running `npx prettier --check .` as stated in the implementation notes, but the implementation author should verify Prettier is available. If Prettier is not a project dependency and no `.prettierrc` exists, the lint job should skip the Prettier check and document this decision — the AC says "Prettier check" but the project may intentionally not use Prettier.

**Convex deployment provisioning in CI**: The E2E job needs an ephemeral Convex backend. The exact mechanism depends on the Convex CLI's CI support — likely `npx convex deploy` with a `CONVEX_DEPLOY_KEY` secret, or `npx convex dev --once` for a throwaway instance. The implementation author should consult Convex docs for the recommended CI pattern. The key requirement is that CI runs don't pollute the production or staging deployments.

**Playwright browser installation**: `npx playwright install --with-deps` installs browser binaries and OS-level dependencies (e.g., `libgbm`, `libasound2`). On `ubuntu-latest` this is well-supported but adds ~30-60s to the E2E job. This time is included in the 10-minute budget.

**Cache misses on first run**: The very first CI run (or after a `package-lock.json` change) will have a cold npm cache, adding ~30-60s. The 10-minute target applies to "clean runs" per the AC, meaning a run with warm caches on a passing codebase.

**E2E job failure does not block lint/typecheck/unit results**: Because jobs are sequential (not parallel), a lint failure stops the pipeline before typecheck/unit/e2e even start. This is the intended behavior per the AC — fast-failing jobs gate slower ones. If future needs require seeing all results at once, the implementation can be changed to parallel jobs, but that's out of scope.

## Non-goals

**No new test files or configurations**: All test infrastructure (vitest.config.ts, playwright.config.ts, e2e/fixtures.ts, e2e/helpers.ts, convex/seed.ts) was created by prior tickets (WOR-107 and others). This ticket only wires them into CI.

**No staging/production deployment**: This pipeline runs on push/PR to `main` for validation only. Deployment to staging or production is a separate concern.

**No package.json script changes**: The scripts `lint`, `typecheck`, `test`, and `test:e2e` already exist. If a `format:check` script is needed for Prettier, that's an implementation detail within this ticket's scope but is not a new export.

**No parallel job execution**: The AC explicitly calls for sequential execution (lint → typecheck → unit → e2e) via `needs:`. Running jobs in parallel would be faster but contradicts the spec's intent of fast-failing gates.

## Test coverage

**AC 2 (lint job with ESLint + Prettier)** — Asserts a job named `lint` exists with steps that run ESLint (via `npm run lint`) and Prettier (via `npx prettier --check` or equivalent).

**AC 3 (typecheck job)** — Asserts a job named `typecheck` exists with `needs: lint` (or array containing `lint`) and a step that runs `tsc --noEmit` (via `npm run typecheck` or direct invocation).

**AC 4 (unit job)** — Asserts a job named `unit` exists with `needs: typecheck` and a step that runs `vitest run` (via `npx vitest run` or `npm test`).

**AC 5 (E2E job with Convex + CLAUDE_MOCK)** — Asserts a job named `e2e` exists with `needs: unit`, has `CLAUDE_MOCK: "true"` in its env, includes steps for Playwright test execution, and includes an artifact upload step with `if: failure()`.

**AC 6 (Node.js LTS + caching)** — Asserts every job uses `actions/setup-node` with `node-version` containing `lts` and has npm caching configured.

**AC 7 (under 10 minutes)** — This is a runtime performance metric that cannot be structurally validated. The unit test can verify prerequisites (caching is configured, sequential gating is present), but actual timing is verified by monitoring GitHub Actions run durations post-merge. The `tested_by` entry uses layer `e2e` with a reason explaining this limitation, but in practice the test file validates structural prerequisites only.
