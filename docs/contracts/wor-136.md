# Contract: WOR-136 — Abandoned case cron job (convex/crons.ts)

## Why this work exists

Clarity needs to automatically close stale cases so participants' dashboards don't accumulate dormant sessions. When a JOINT_ACTIVE case has had no activity for 30 days, the system transitions it to CLOSED_ABANDONED. This implements the daily Convex cron job described in TechSpec §5.3.

## Files and exports

### `convex/crons.ts` (create, config)

Convex's built-in cron scheduling file. Exports a default `cronJobs()` instance that registers a single daily job targeting `internal.abandonedCases.scanAndCloseAbandoned`. This file is the entry point Convex uses to discover scheduled jobs — it must be the default export.

### `convex/abandonedCases.ts` (create, mutation)

Contains the `scanAndCloseAbandoned` internalMutation. This is an **internalMutation** (not a public mutation) because it is only invoked by the cron scheduler, never by a client. It:

1. Queries all cases where `status === "JOINT_ACTIVE"` and `updatedAt <= Date.now() - 30 * 24 * 60 * 60 * 1000`.
2. For each matching case, calls `validateTransition(case.status, "ABANDON")` to get the new status (`CLOSED_ABANDONED`).
3. Patches the case: `{ status: newStatus, closedAt: Date.now(), updatedAt: Date.now() }`.
4. For each affected case, inserts a notification record for both the initiator and invitee (if present).

### `convex/schema.ts` (modify)

Two changes:
1. Add a `by_status` index on the `cases` table: `.index("by_status", ["status"])` — enables efficient filtering of JOINT_ACTIVE cases without a full table scan.
2. Add a `notifications` table for dashboard badge notifications:

```ts
notifications: defineTable({
  userId: v.id("users"),
  caseId: v.id("cases"),
  type: v.string(),           // e.g. "CASE_ABANDONED"
  read: v.boolean(),
  createdAt: v.number(),
}).index("by_user", ["userId"])
  .index("by_user_and_read", ["userId", "read"]),
```

### `tests/unit/abandoned-cases.test.ts` (create, test)

Vitest unit tests using `convex-test` (same pattern as `tests/unit/cases.test.ts`). Imports the schema and uses `convex-test` to run the internal mutation directly.

## Data dependencies

- **cases table** — reads all cases with `status === "JOINT_ACTIVE"` via the `by_status` index. Consumes `_id`, `status`, `updatedAt`, `initiatorUserId`, `inviteeUserId`.
- **partyStates table** — not read; the initiator/invitee user IDs are on the case itself.
- **notifications table** (new) — written to, not read. Each notification record has `{ userId, caseId, type: "CASE_ABANDONED", read: false, createdAt }`.
- **convex/lib/stateMachine.ts** — `validateTransition("JOINT_ACTIVE", "ABANDON")` returns `"CLOSED_ABANDONED"`. The ABANDON transition is already defined in the state machine.

## Invariants

**Only JOINT_ACTIVE cases with stale updatedAt are affected.** The query filters on `status === "JOINT_ACTIVE"` AND `updatedAt <= threshold`. Cases in any other status — including other CLOSED_* states — are never examined or modified.

**Transition via state machine helper.** The mutation never writes `status: "CLOSED_ABANDONED"` directly. It calls `validateTransition(currentStatus, "ABANDON")` and uses the returned status. This ensures the state machine remains the single source of truth for valid transitions. If the state machine ever changes which transitions are legal, the cron will fail loudly rather than silently corrupting state.

**Notification for all parties.** A notification record is created for both `initiatorUserId` and `inviteeUserId` (when present). Solo cases have `inviteeUserId` unset — in that case only the initiator is notified.

**Internal-only execution.** The mutation is `internalMutation`, not `mutation`. It cannot be called from a client. Only the Convex scheduler can invoke it.

## Edge cases

**No stale cases exist.** The scan finds zero matching cases and completes without error. No notifications are created.

**Case with updatedAt exactly 30 days ago.** The comparison is `updatedAt <= now - 30_DAYS` (less-than-or-equal). A case at exactly the 30-day boundary IS closed. The test uses 31 days for clarity, but the boundary is 30 days inclusive.

**Solo case (no invitee).** `inviteeUserId` may be `undefined`. The notification loop skips undefined user IDs — only the initiator gets a notification.

**Multiple stale cases.** The scan processes all matching cases in a single mutation transaction. At v1 scale (max 50 concurrent cases), this is well within Convex's transaction limits.

## Non-goals

- **Email notifications** — the AC explicitly says "no email in v1." Only a dashboard badge (notification record) is created.
- **UI changes** — the ClosedCaseView already handles CLOSED_ABANDONED rendering per DesignDoc §4.7. No frontend work in this ticket.
- **Configurable threshold** — the 30-day value is hardcoded. A future ticket could make it an environment variable or admin setting.
- **Batch processing / pagination** — at v1 scale (≤50 active cases), iterating all JOINT_ACTIVE cases in one transaction is safe. No chunking needed.

## Test coverage

All five ACs are covered by `tests/unit/abandoned-cases.test.ts` at the unit layer using `convex-test`:

1. **AC 1 (cron definition):** Import `convex/crons.ts` and assert it exports a cron schedule targeting the abandoned-case scan at a daily interval.

2. **AC 2 (query filter):** Seed cases with varying statuses (JOINT_ACTIVE, BOTH_PRIVATE_COACHING, CLOSED_RESOLVED) and updatedAt timestamps. Run `scanAndCloseAbandoned`. Assert only JOINT_ACTIVE cases with `updatedAt > 30 days ago` are selected.

3. **AC 3 (state transition):** Seed a JOINT_ACTIVE case with updatedAt 31 days ago. Run the scan. Assert the case status is now CLOSED_ABANDONED and `closedAt` is set.

4. **AC 4 (notification):** After a case is abandoned, assert notification records exist for each party (initiator + invitee) with `type: "CASE_ABANDONED"` and `read: false`.

5. **AC 5 (boundary test):** Seed two JOINT_ACTIVE cases — one with updatedAt 29 days ago, one with updatedAt 31 days ago. Run the scan. Assert only the 31-day case transitions. The 29-day case remains JOINT_ACTIVE.
