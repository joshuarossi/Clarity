# Contract: WOR-100 — Privacy response filter (convex/lib/privacyFilter.ts)

## Why this work exists

Clarity's privacy model allows the Coach AI server-side access to both parties' private content, but strictly forbids surfacing the other party's raw words in any output. The prompt-level anti-quotation instruction (WOR-99) is the first layer of defense; this module is the second: a deterministic server-side response filter that programmatically detects near-verbatim leaks before AI-generated text reaches a user. Both the Synthesis action and the Coach action depend on this filter (TechSpec §6.3.2), so it must be built before either of them. This is classified as NFR-PRIVACY — the highest-severity privacy control in the system.

## Files and exports

### `convex/lib/privacyFilter.ts` (create, helper)

A pure synchronous helper module with no Convex runtime dependencies. It imports nothing from `"convex/server"` or `convex/_generated`. All data is passed in by the caller. The module exports three things:

- **`FilterResult`** — the return type interface: `{ passed: boolean; matchedSubstring?: string }`. When `passed` is `false`, `matchedSubstring` contains the matched token sequence reconstructed as a space-joined string (for logging/debugging by the caller, not for display to users).

- **`tokenize`** — a function that splits text into tokens on whitespace and punctuation boundaries. Exported so the test suite can verify tokenization behavior directly. Punctuation characters (commas, periods, exclamation marks, question marks, semicolons, colons, parentheses, quotes, dashes, etc.) are treated as separate tokens. Multiple whitespace characters are collapsed. The function returns an array of non-empty strings, each lowercased for case-insensitive matching.

- **`filterResponse`** — the main function. It accepts `candidateText` (the AI-generated output to check) and `otherPartyMessages` (an array of plain-text strings representing the other party's private messages). It tokenizes both the candidate and each message, then checks whether any contiguous run of 8 or more tokens from any single message appears in the candidate's token stream. Returns `{ passed: true }` if no match is found, or `{ passed: false, matchedSubstring: "..." }` on the first match found.

**Why `otherPartyMessages` is `string[]` not a richer type:** The module is pure and has no knowledge of the database schema. Callers extract the `content` field from each private message record and pass the plain strings. This keeps the module free of Convex type dependencies and trivially testable.

**Why `tokenize` lowercases:** The implementation notes explicitly require case-insensitive matching to catch trivial evasion (AI capitalizing differently). Lowercasing at tokenization time means the matching loop is a simple equality check.

### `tests/unit/privacyFilter.test.ts` (create, test-infrastructure)

Unit tests using Vitest. Since `convex/lib/privacyFilter.ts` is a pure module, tests import directly and call functions with fixture data — no `convex-test` needed. This follows the project's existing convention where all unit tests live under `tests/unit/` (matching `tests/unit/errors.test.ts`, `tests/unit/prompts.test.ts`, etc.).

## Data dependencies

None. This module is a pure function. It receives all data via parameters and returns a result synchronously. It does not call any Convex queries, mutations, or actions. It does not make any network requests.

The callers (Synthesis action, Coach action) are responsible for:

1. Fetching the other party's private messages from the `privateMessages` table.
2. Extracting the `content` field from each message into a `string[]`.
3. Passing the complete AI-generated candidate text and the message strings to `filterResponse`.
4. Handling retry/fallback logic if the filter returns `passed: false`.

## Invariants

### No false negatives on verbatim matches (highest priority)

If 8 or more consecutive tokens from any single private message appear in the candidate text, the filter MUST return `passed: false`. This is the fundamental safety guarantee. The system explicitly prefers false positives (blocking safe content) over false negatives (leaking private content). This is NFR-PRIVACY — the highest-severity privacy control.

### Case-insensitive matching

All token comparison is case-insensitive. The tokenizer lowercases all tokens. This catches the scenario where the AI reproduces private content but changes capitalization (e.g., the original says "i feel really frustrated" and the AI output says "I Feel Really Frustrated").

### Per-message independence

Each message in `otherPartyMessages` is checked independently against the candidate text. A run of 8+ tokens must come from a single message to count as a match. Tokens cannot be combined across messages to form a match. This prevents false positives from common phrases that happen to span message boundaries.

### Short message immunity

Messages with fewer than 8 tokens cannot produce a match by definition (there is no contiguous 8-token subsequence to extract). The implementation may skip tokenization of such messages as an optimization, but the behavior is the same either way.

### Purity

The module has no runtime dependencies beyond standard TypeScript. No imports from `convex/server`, no `async`, no database reads, no external API calls, no side effects. This makes it trivially unit-testable and safe to call from any context.

## Edge cases

### Empty otherPartyMessages array

When `otherPartyMessages` is an empty array (no private messages from the other party), `filterResponse` returns `{ passed: true }`. There is nothing to match against.

### Empty candidate text

When `candidateText` is an empty string, `filterResponse` returns `{ passed: true }`. An empty response cannot contain any leaked content.

### Very short messages

A private message like "ok" or "I agree" (fewer than 8 tokens) can never trigger a match. The filter naturally passes because no 8-token window can be extracted from the message.

### Punctuation-heavy content

Content like "Wait... really?! That's — honestly — unbelievable!" tokenizes punctuation as separate tokens: `["wait", "...", "really", "?!", "that's", "--", "honestly", "--", "unbelievable", "!"]`. Punctuation tokens participate in matching. This means punctuation-rich passages have a wider token window and are less likely to false-positive, which is the correct behavior (punctuation-dense text is more distinctive).

### Multiple matches across different messages

The filter returns on the first match found. It checks each message in order and, for each message, scans for the first 8+ token overlap. The `matchedSubstring` reflects the first match, not all matches. This is sufficient for the caller's retry logic.

### Candidate contains the matched substring multiple times

One occurrence is enough to fail. The filter does not count occurrences — any single match triggers `passed: false`.

## Non-goals

**No retry or regeneration logic.** The filter returns a result; the caller decides what to do. Per TechSpec §6.5, callers retry generation up to 2 times, then fall back to a generic response and flag for admin review.

**No embedding similarity matching.** TechSpec TQ2 explicitly chose token-substring matching for v1 because it is simpler and deterministic. Embedding similarity may be added in a future version if false negatives are found in production.

**No database access.** Callers fetch private messages and pass them in. The filter has no knowledge of the database schema.

**No streaming integration.** The filter runs on the complete candidate text after generation finishes. Streaming-aware filtering (checking partial output mid-stream) is out of scope.

**No prompt assembly.** That is WOR-99 (`convex/lib/prompts.ts`). This module only post-processes AI output.

## Test coverage

All tests live in `tests/unit/privacyFilter.test.ts`, following the project convention of `tests/unit/<module>.test.ts`.

**AC 1 (function signature) -> `tests/unit/privacyFilter.test.ts` (unit).** Call `filterResponse` with known inputs. Assert the return value has `passed` (boolean). When `passed` is `false`, assert `matchedSubstring` is a non-empty string. When `passed` is `true`, assert `matchedSubstring` is `undefined`.

**AC 2 (tokenization) -> `tests/unit/privacyFilter.test.ts` (unit).** Call `tokenize` directly with strings containing mixed whitespace, punctuation, and normal words. Assert expected token arrays. Examples: `"Hello, world! How are you?"` produces `["hello", ",", "world", "!", "how", "are", "you", "?"]`. Multiple spaces collapse. Tabs and newlines are treated as whitespace boundaries.

**AC 3 (8-token threshold) -> `tests/unit/privacyFilter.test.ts` (unit).** Construct a private message of 10+ tokens. Build a candidate containing exactly 8 consecutive tokens from it — assert `passed: false` and `matchedSubstring` contains those tokens. Build a candidate with only 7 consecutive tokens from it — assert `passed: true`.

**AC 4 (Vitest suite) -> `tests/unit/privacyFilter.test.ts` (unit).**

- Exact 8-token match: `passed: false`.
- 7-token match: `passed: true`.
- Paraphrased content (same meaning, different words): `passed: true`.
- Empty `otherPartyMessages` array: `passed: true`.
- Multiple messages: each message is checked independently; a match in any one triggers failure.

**AC 5 (adversarial cases) -> `tests/unit/privacyFilter.test.ts` (unit).**

- Quoted text with one word substituted in the middle of an 8-token run: `passed: true` (the substitution breaks the consecutive token sequence).
- Verbatim copy-paste of 10+ tokens: `passed: false`.
- Case variation (all-caps copy of original): `passed: false` (case-insensitive matching catches this).

**AC 6 (purity) -> `tests/unit/privacyFilter.test.ts` (unit).** Verify the module has no imports from `convex/_generated`, `convex/server`, or any database layer. This can be tested by reading the module source as a string and asserting no forbidden import patterns, or simply by the fact that the tests run without any Convex runtime setup.
