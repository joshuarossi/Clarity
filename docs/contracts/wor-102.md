# Contract: WOR-102 — App shell — Vite + React + ConvexProvider + AuthProvider + routing

## Why this work exists

Without this shell, no frontend page can render, authenticate, or subscribe to live Convex data. This task provides the foundational wiring — provider tree, route definitions, auth guards, navigation chrome, and error mapping — that every subsequent UI ticket builds on. It's the skeleton that makes the app bootable.

## Files and exports

### `src/main.tsx` (config, create)

The React 18 entry point. Creates a `ConvexReactClient` using `import.meta.env.VITE_CONVEX_URL`, wraps the app in `<ConvexAuthProvider client={convex}>` → `<BrowserRouter>` → `<App />`. `ConvexAuthProvider` wraps `ConvexProvider` internally. This file has no named exports — it's a side-effect module that calls `createRoot` and renders.

### `src/App.tsx` (route, create)

Defines the full `<Routes>` tree matching TechSpec §9.2. Each route renders either a placeholder stub component (for pages not yet built) or wraps children in `<ProtectedRoute>` / `<AdminRoute>` as appropriate. The `App` component also renders `<TopNav>` in the layout (outside routes but reading route info to determine variant).

### `src/components/layout/TopNav.tsx` (presentational, create)

A **presentational** component. It receives a `variant` prop to decide which layout to show:
- `"logged-in"`: Dashboard link + user display name + logout button/menu.
- `"case-detail"`: Back arrow (ArrowLeft icon, 14px) linking to the case's parent route + phase display text.

TopNav does NOT call hooks itself. The parent layout (in `App.tsx`) determines which variant to render based on route and passes the needed data as props. This keeps TopNav testable in isolation and avoids coupling it to routing/auth internals.

### `src/components/layout/ProtectedRoute.tsx` (connected, create)

A connected wrapper component that calls `useConvexAuth()`. Three states:
1. `isLoading` → renders a centered spinner (accessible, with `aria-label="Loading"`)
2. `!isAuthenticated` → `<Navigate to="/login" replace />`
3. Authenticated → renders `children`

### `src/components/layout/AdminRoute.tsx` (connected, create)

Extends the protected-route pattern. After confirming authentication, queries the current user's record via `api.users.me` to check `role`. If `role !== "ADMIN"`, redirects to `/dashboard`. Handles the loading state of the user query with a spinner.

### `src/lib/errorHandler.ts` (helper, create)

Maps `ConvexError` instances (with their `AppErrorData` shape from `convex/lib/errors.ts`) to user-friendly strings suitable for toast display. The function inspects `error.data.code` and returns a curated message per code. Unknown errors get a generic fallback. This utility does NOT display toasts — it only produces the message. Toast UI is a future concern.

### `vite.config.ts` (config, modify)

Minimal change: the current config is already valid for `VITE_` env vars (Vite exposes them by default). No functional change needed unless we add path aliases or other build config. The contract confirms this file remains valid and includes the `react()` plugin.

## Data dependencies

### `api.users.me`

A query that returns the current authenticated user's record from the `users` table. Expected shape:

```ts
{ _id: Id<"users">; email: string; displayName?: string; role: "USER" | "ADMIN"; createdAt: number }
```

This query is called by `AdminRoute` to determine if the user has admin access. It must return `null` or throw if unauthenticated (the component handles both). **Note:** This query may not yet exist — the contract declares it as a dependency this ticket will consume. If it doesn't exist, this ticket creates a minimal implementation in `convex/users.ts` (a simple `query` that resolves the current user by auth identity email).

## Invariants

**Provider nesting order:** `ConvexAuthProvider` (wraps `ConvexProvider` internally) → `BrowserRouter` → `Routes`. This order is non-negotiable. The router needs the auth provider above it so that route components can call `useConvexAuth()` and `useQuery()`.

**Loading vs. redirect:** When `useConvexAuth()` returns `isLoading: true`, the UI MUST show a spinner, not redirect. Redirecting during loading creates a flash-of-login-page on every page load even for authenticated users.

**Admin check ordering:** `AdminRoute` must confirm authentication FIRST (via `useConvexAuth`), then query the user role. Never query user data for unauthenticated visitors — it would fail and create confusing error states.

**Error handler opacity:** The `handleConvexError` function never passes through raw `error.data.message` from the server to the user. It maps codes to pre-written strings. This prevents leaking internal details (table names, query paths) in user-facing UI.

**TopNav is presentational:** It takes props and renders. It does not call `useConvexAuth`, `useQuery`, or `useNavigate` internally. This is a deliberate architecture decision so that TopNav can be rendered in test harnesses without provider wrappers, and so the rendering logic lives in one place (the layout in App.tsx).

## Edge cases

**Loading state:** All protected/admin routes show a spinner while auth state resolves. The spinner is centered in the viewport with an accessible label. No content flashes before auth resolves.

**Unauthenticated at protected route:** Immediate `<Navigate to="/login" replace />`. The `replace` ensures hitting "back" doesn't loop back to the protected URL.

**Non-admin at admin route:** After auth confirms and user query resolves, redirect to `/dashboard` with `replace`. No flash of admin content.

**VITE_CONVEX_URL missing:** If the env var is not set, `ConvexReactClient` constructor will throw at startup. This is acceptable — it's a developer configuration error, not a user-facing state. The error will appear in the browser console.

**Unknown error codes:** `handleConvexError` receives something that isn't a `ConvexError` or has an unrecognized code → returns `{ code: "INTERNAL", message: "Something went wrong. Please try again." }`.

**Route not found:** Any unmatched URL renders a simple "Page not found" stub with a link to `/dashboard` (or `/` if unauthenticated).

## Non-goals

- **Page implementations:** All route targets except TopNav and the guards are placeholder stubs (`<h1>Page Name</h1>` + `data-testid`). Real content comes from later tickets.
- **Toast UI:** This ticket provides `handleConvexError` (the mapping logic), not a toast notification system. A future ticket will add the toast component and wire it to this utility.
- **Solo mode toggle:** `PartyToggle` in TopNav is a separate ticket. TopNav in this ticket does not include it.
- **Auth provider configuration:** The Convex Auth setup (magic link + Google OAuth providers in `convex/auth.ts`) is assumed to exist from prior scaffolding work. This ticket consumes it, doesn't create it.
- **Dark mode toggle in TopNav:** Theme is handled by WOR-103. TopNav inherits whatever theme is active via CSS custom properties.

## Test coverage

**E2E tests (`e2e/app-shell.spec.ts`)** cover six of the eight ACs. These require a running app with Convex connection because they test provider initialization, real auth flows, role-based redirects, and browser navigation history. The test file covers:
- App boots without console errors (providers AC)
- Each route renders its stub (routes AC)
- Unauthenticated → redirected to /login (protected routes AC)
- Regular user at /admin/* → redirected to /dashboard (admin routes AC)
- TopNav variants visible in appropriate contexts (TopNav AC)
- Back/forward browser navigation works (navigation AC)

**Unit tests (`tests/unit/vite-config.test.ts`)** verify the Vite config is parseable and valid (a lightweight structural check).

**Unit tests (`tests/unit/errorHandler.test.ts`)** exhaustively test `handleConvexError`:
- Each of the 9 `ErrorCode` values returns a non-empty user-friendly string that differs from the raw code.
- An unknown/malformed error returns the generic fallback.
- A non-ConvexError (e.g., plain Error) returns the generic fallback.
