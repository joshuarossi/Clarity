# Contract: WOR-102 — App shell — Vite + React + ConvexProvider + AuthProvider + routing

## Why this work exists

Without this app shell, no frontend page can render, authenticate, or subscribe to live Convex data. Every subsequent UI task (dashboard, case detail, chat views, admin pages) depends on the provider tree, route definitions, and auth guards established here. This task also provides the TopNav (the only persistent layout element across all authenticated pages) and the frontend error handler utility that all future Convex-calling code will use to surface errors to users.

## Files and exports

### `src/main.tsx` (create)

The React 18 entry point. Calls `createRoot(document.getElementById("root")!)` and renders the full provider tree in strict mode. The nesting order is critical: `<StrictMode>` → `<ConvexProvider client={convex}>` → `<ConvexAuthProvider>` → `<BrowserRouter>` → `<App />`. ConvexProvider must be outermost (excluding StrictMode) because the auth provider depends on the Convex client connection. BrowserRouter sits inside auth so that route guard components can access auth state via hooks. This file has no named exports — it is a side-effect module that bootstraps the app.

### `src/App.tsx` (create)

Defines the complete route tree from TechSpec §9.2 using React Router v6 `<Routes>` and `<Route>`. The route structure uses layout routes for auth guards: public routes (`/`, `/login`, `/invite/:token`) are unguarded; protected routes (`/dashboard`, `/cases/*`) are wrapped in `<ProtectedRoute>`; admin routes (`/admin/*`) are wrapped in `<AdminRoute>`. TopNav is rendered inside the route tree as a layout element — present on all routes except possibly the landing page when logged out. Case subroutes (`/cases/:caseId/private`, `/cases/:caseId/joint`, `/cases/:caseId/closed`) are nested under `/cases/:caseId` with `CaseDetail` as the layout component rendering `<Outlet />`.

### `src/lib/convex.ts` (create)

Creates and exports the `ConvexReactClient` instance, initialized with `import.meta.env.VITE_CONVEX_URL`. This is a separate file (rather than inline in main.tsx) because the TechSpec §9.1 structure lists it and other modules may need the client reference.

### `src/lib/errorHandler.ts` (create)

The frontend counterpart to `convex/lib/errors.ts`. Exports `ERROR_MESSAGES`, a `Record<ErrorCode, string>` mapping each of the 9 Convex error codes to a user-friendly message suitable for toast display. Exports `handleConvexError(error: unknown): string`, which checks if the error is a `ConvexError` with the expected `AppErrorData` shape, extracts the code, and returns the corresponding message. For non-ConvexError errors or unrecognized codes, returns a generic fallback message ("Something went wrong. Please try again."). The function imports the `ErrorCode` and `AppErrorData` types from `convex/lib/errors` — it does not duplicate the type definitions.

The 9 user-friendly messages should be non-technical. Examples:
- `UNAUTHENTICATED` → "Please sign in to continue."
- `FORBIDDEN` → "You don't have permission to do that."
- `NOT_FOUND` → "We couldn't find what you're looking for."
- `CONFLICT` → "This action can't be performed right now. The state may have changed."
- `INVALID_INPUT` → "Please check your input and try again."
- `TOKEN_INVALID` → "This invite link is no longer valid."
- `RATE_LIMITED` → "Too many requests. Please wait a moment and try again."
- `AI_ERROR` → "The AI service encountered an issue. Please try again."
- `INTERNAL` → "Something went wrong on our end. Please try again."

### `src/hooks/useCurrentUser.ts` (create)

A thin hook wrapping `useQuery(api.users.me)`. Returns `{ user, isLoading }` where `user` is `Doc<"users"> | null | undefined` (undefined while loading, null if unauthenticated, Doc if authenticated). `isLoading` is derived from `user === undefined`. This hook is consumed by `AdminRoute` (to check role) and `TopNav` (to display user name/email in the user menu).

### `src/components/layout/TopNav.tsx` (create)

A **connected** component — the single most important interface decision on this ticket. TopNav calls hooks internally (`useConvexAuth()`, `useCurrentUser()`, React Router's `useLocation()` and `useParams()`) and requires no props. This is the right call because TopNav appears on every page and must reactively update when auth state changes or the user navigates — passing these as props from every parent would be redundant and fragile.

Three rendering variants per DesignDoc §3.2:

1. **Logged-out** (on `/`, `/login`, `/invite/:token`): Minimal — just the "Clarity" wordmark. No nav links.
2. **Logged-in** (on `/dashboard`, `/cases/new`, and other non-case authenticated routes): "Clarity" wordmark on the left, "Dashboard" link and user menu (showing user displayName or email) on the right.
3. **Case-detail** (on `/cases/:caseId/*`): "← Back to Dashboard" link on the left, case identifier and phase name in the center. The back arrow uses the ArrowLeft icon at 14px. Phase display shows the human-readable case status (e.g., "Private Coaching", "Joint Session", "Closed"). Since case data queries are not yet implemented, the phase display shows a loading placeholder until case detail tasks provide the data.

TopNav renders inside a `<nav>` element with appropriate ARIA attributes.

### `src/components/auth/ProtectedRoute.tsx` (create)

A **connected** route guard component. Uses `useConvexAuth()` from `convex/react`. When `isLoading` is true, renders a centered spinner. When `!isAuthenticated`, renders `<Navigate to="/login" replace />`. When authenticated, renders `<Outlet />` so child routes can render. The `replace` on Navigate ensures the login redirect doesn't create a back-button loop.

### `src/components/auth/AdminRoute.tsx` (create)

A **connected** route guard that composes the auth check with a role check. First verifies authentication (same as ProtectedRoute — spinner while loading, redirect to /login if unauthenticated). Then uses `useCurrentUser()` to query the user's role. If the user query is loading, shows spinner. If `user.role !== "ADMIN"`, renders `<Navigate to="/dashboard" replace />`. Otherwise renders `<Outlet />`. The role check is based on server-side data (the `users.me` query), not a client-side flag.

### `convex/users.ts` (create)

A Convex query file. Exports a single `me` query with no arguments. Uses `ctx.auth.getUserIdentity()` to get the current session's identity, then looks up the user record by email via the `by_email` index. Returns `null` if no authenticated session or no user record found (does NOT throw — this is a query, not a mutation guard). This differs from `convex/lib/auth.ts:requireAuth` which throws on unauthenticated access; the `me` query is designed for the frontend where unauthenticated is a valid state (the route guard handles the redirect).

## Data dependencies

### `api.users.me` (created by this task)

Returns the current authenticated user's document from the `users` table, or `null` if no authenticated session. Fields consumed by this task:
- `displayName` — rendered in TopNav user menu
- `email` — fallback display in TopNav user menu if displayName is empty
- `role` — checked by AdminRoute guard (`"ADMIN"` vs `"USER"`)

No other Convex queries are used. The auth state (isAuthenticated, isLoading) comes from `useConvexAuth()` which is a Convex React hook, not a user-defined query.

## Invariants

**Provider nesting order.** `ConvexProvider` → `ConvexAuthProvider` → `BrowserRouter` → `Routes`. This order is load-bearing: ConvexAuthProvider depends on the Convex client connection (needs ConvexProvider above it), and route guard components use `useConvexAuth()` which needs ConvexAuthProvider above them. Reversing any pair breaks the app.

**No content flash on protected routes.** When an unauthenticated user navigates to `/dashboard` (or any protected route), they must see either a loading spinner or the `/login` page — never the protected page content, even briefly. ProtectedRoute handles this by checking auth state before rendering children.

**Admin role check is server-authoritative.** The AdminRoute guard queries `api.users.me` and reads `user.role` from the server response. It does NOT read role from localStorage, a JWT claim, or any client-side cache. If the user's role changes server-side, the reactive query updates and the guard responds.

**Error handler covers all 9 codes.** `ERROR_MESSAGES` must have an entry for every value in the `ErrorCode` union type. If a new code is added to `convex/lib/errors.ts`, TypeScript should flag the missing key (the Record type enforces exhaustiveness).

**Focus management on navigation.** When the route changes, focus moves to the new page's `<h1>` element. This is an accessibility requirement (NFR-A11Y) that ensures screen reader users know they've navigated. Implementation: a small effect in App.tsx or a wrapper component that listens to location changes.

**Landmarks.** TopNav renders as `<nav>`. Each page's content is wrapped in `<main>`. These are required for screen reader landmark navigation.

## Edge cases

**Loading state.** When the app first loads, `useConvexAuth()` returns `isLoading: true`. During this time, ProtectedRoute and AdminRoute render a spinner. TopNav renders the logged-out variant (or nothing) until auth state resolves. The loading spinner should be simple (CSS-only, centered) — no need for a complex skeleton.

**Unauthenticated on protected route (deep link).** A user who opens a direct link to `/cases/abc123/joint` without being logged in should be redirected to `/login`. After login, ideally they return to the intended page — but implementing "return to" redirect is a non-goal for this task (can be added in the auth flow task).

**Invalid route (404).** The route tree should include a catch-all `*` route that renders a simple "Page not found" message with a link to `/dashboard` (if authenticated) or `/` (if not).

**VITE_CONVEX_URL not set.** If the env var is missing, `new ConvexReactClient(undefined)` will throw at startup. This is the correct behavior — the app cannot function without a Convex backend URL. No special error handling needed; the developer sees the error in console.

**ConvexError vs. non-ConvexError.** `handleConvexError` must handle both `ConvexError` instances (which have `.data` with the `{code, message, httpStatus}` shape) and arbitrary errors (network failures, unexpected exceptions). For non-ConvexError inputs, it returns the generic fallback message.

## Non-goals

**No actual page implementations.** Every route renders a stub. Case creation, dashboard content, chat UIs, and admin features are separate tasks (T10+).

**No solo mode party toggle.** The PartyToggle component in TopNav is part of the solo mode task, not the app shell.

**No dark mode toggle in TopNav.** Theme switching is handled by the inline script in `index.html` and localStorage (WOR-103). A visible toggle button is a future layout concern.

**No toast rendering component.** The error handler returns strings. A toast/notification UI component that displays these strings is a downstream dependency.

**No Convex Auth configuration.** This task wires `<ConvexAuthProvider>` into the tree but does NOT create `auth.config.ts`, implement signIn/signOut actions, or build the login form. Those belong to the authentication task. The auth provider will be present but non-functional until that task lands.

**No "return to" redirect after login.** When ProtectedRoute redirects to `/login`, it does not preserve the originally-requested URL for post-login redirect. This is a login flow concern.

**AC: TopNav variants** — `e2e/app-shell.spec.ts` (e2e only). When authenticated, verify the logged-in TopNav shows "Dashboard" link and user display name. On a `/cases/:caseId` route, verify the back arrow link and "Back to Dashboard" text are present. Requires real browser rendering and auth context.

**AC: Browser back/forward** — `e2e/app-shell.spec.ts` (e2e only). Navigate `/dashboard` → `/cases/new` → `page.goBack()` → assert URL is `/dashboard` → `page.goForward()` → assert URL is `/cases/new`. Requires real browser history API.

**AC: Vite config** — `tests/unit/vite-config.test.ts` (unit). Import or read `vite.config.ts`, assert it is valid and exports a config object. Verify that `build.target` is set to `"es2020"`. The `VITE_CONVEX_URL` env var is a runtime concern verified implicitly by the e2e tests.

**AC: Error handler utility** — `tests/unit/errorHandler.test.ts` (unit). Import `handleConvexError` and `ERROR_MESSAGES`. For each of the 9 error codes, construct a `ConvexError` with the matching `AppErrorData` shape, pass it to `handleConvexError`, assert the returned string matches `ERROR_MESSAGES[code]` and is a non-empty, non-technical string (does not contain the raw code). Test that an unknown error code returns the generic fallback. Test that a plain `Error("network failure")` (not a ConvexError) returns the generic fallback. Test that `ERROR_MESSAGES` has exactly 9 entries.
