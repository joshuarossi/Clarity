# Contract: WOR-107 — Playwright infrastructure — config, fixtures, Claude mock mode, CI wiring

## Why this work exists

Before any feature-level E2E tests can run, the project needs shared Playwright infrastructure: a configuration targeting all three browser engines, reusable auth fixtures that avoid slow magic-link flows, and a deterministic Claude mock mode so AI-dependent features can be tested without hitting the real Anthropic API. This ticket stands up that foundation so every subsequent Playwright suite has a consistent, fast base.

## Files and exports

### `playwright.config.ts` (modify)

The existing config only defines a single `chromium` project. This ticket adds `firefox` and `webkit` projects using Playwright's `devices` presets. The `baseURL` stays at `http://localhost:5173` (Vite dev server). CI-aware settings are already partially present (`forbidOnly`, `retries`, `workers`) and remain as-is. No new exports — this is a config file.

### `e2e/fixtures.ts` (create)

The central fixture file that extends `@playwright/test` with two custom fixtures:

- **`authenticatedPage`**: A `Page` instance inside a `BrowserContext` that has been programmatically authenticated as a test user via `loginAs`. This is the primary fixture most future E2E specs will import.
- **`twoUserContexts`** (exposed as `pageA` and `pageB`): Two separate `BrowserContext` instances, each authenticated as a different test user. Used by invite-flow and two-party tests.

All future E2E test files should `import { test, expect } from "./fixtures"` instead of from `@playwright/test` directly, so they get the auth fixtures for free.

### `e2e/helpers.ts` (create)

Test utility functions that interact with the Convex dev backend programmatically:

- **`createTestUser`**: Inserts a user record into the Convex dev deployment (via HTTP API or convex-test client). Accepts optional email and role. Returns `{ userId, email }`.
- **`createTestCase`**: Creates a case with the given initiator (and optionally an invitee). Returns `{ caseId }`. Uses a default category and template version.
- **`loginAs`**: Given a Playwright `Page` and an email, programmatically sets up an authenticated session. The exact mechanism depends on Convex Auth's test mode — likely inserting a session token directly or using a test-only endpoint. This is the function the `authenticatedPage` fixture calls internally.

### `e2e/claude-mock.ts` (create)

Client-side reference copy of the canned mock responses. This file mirrors what `convex/lib/claudeMock.ts` returns server-side, so E2E test assertions can compare against known expected output. Exports `CLAUDE_MOCK_RESPONSES` (a `Record<PromptRole, string>`) and a `getMockResponse` helper that wraps the lookup with an optional delay parameter.

### `convex/lib/claudeMock.ts` (create)

Server-side mock module. When a Convex action that calls Claude checks `isClaudeMockEnabled()` and gets `true`, it calls `getMockClaudeResponse(role)` instead of the real Anthropic SDK. The function returns a deterministic canned string per role:

- `PRIVATE_COACH`: Multi-paragraph empathetic markdown with reflective questions.
- `COACH`: Facilitative joint-session markdown with a summary and prompt for both parties.
- `DRAFT_COACH`: A draft suggestion in markdown with tone coaching.
- `SYNTHESIS`: Valid JSON string `{ "forInitiator": "...", "forInvitee": "..." }` with realistic synthesis content.

The `MOCK_DELAY_MS` constant reads `process.env.CLAUDE_MOCK_DELAY_MS` (default 100ms) for simulating streaming latency in tests.

### `e2e/smoke.spec.ts` (create)

A minimal placeholder test that opens `http://localhost:5173/`, waits for the page to load, and asserts that the landing page heading is visible. This test does NOT require authentication — it just proves the Playwright runner, config, and dev server are wired up correctly. It also includes a secondary test that uses the `authenticatedPage` fixture to verify the fixture itself works (navigates to `/dashboard` and asserts no redirect to `/login`).

## Data dependencies

This ticket introduces no new Convex queries or mutations. The test helpers (`createTestUser`, `createTestCase`) will call existing or soon-to-exist Convex mutations programmatically (e.g., inserting into `users` and `cases` tables), but the helpers abstract this away. The `convex/lib/claudeMock.ts` module is a pure helper — it reads environment variables and returns strings; it does not call the database.

## Invariants

**Deterministic mock responses**: `getMockClaudeResponse(role)` must be a pure function of `role` — no randomness, no timestamps, no varying content. This enables tests to assert exact string matches or substring checks against known canned output.

**Auth bypass is programmatic, never UI-driven**: The `loginAs` helper must set auth state directly (e.g., via Convex's internal test session API or direct token insertion), never by automating the magic-link UI flow. This keeps fixture setup fast (<500ms) and avoids flakiness from email delivery simulation.

**Two-user isolation**: The `twoUserContexts` fixture creates two fully independent `BrowserContext` instances. They do not share cookies, localStorage, or session state. Each is authenticated as a different user with a different email. This is critical for invite-flow tests where party isolation must be real.

**Mock shape matches real shape**: The SYNTHESIS mock response must be valid JSON matching `{ "forInitiator": string, "forInvitee": string }` because the UI will `JSON.parse()` it. The coaching role mocks must be valid markdown because the UI renders them with a markdown renderer.

**Config covers all three engines**: `playwright.config.ts` must list `chromium`, `firefox`, and `webkit` as separate `projects[]` entries. The AC is explicit about this — a config with only one browser does not satisfy the requirement.

## Edge cases

**Dev server not running**: The smoke test assumes `http://localhost:5173` is reachable. If the dev server is not running, Playwright will fail with a connection error. The config does not include `webServer` auto-start (that can be added later or by CI); the test failure message should be clear about the cause.

**Convex dev deployment not available**: `createTestUser` and `createTestCase` require a running Convex dev deployment. If unavailable, these helpers should throw a clear error rather than hanging. The helpers should have a reasonable timeout (e.g., 5s).

**CLAUDE_MOCK not set**: When `CLAUDE_MOCK` is not set (or set to any value other than `"true"`), `isClaudeMockEnabled()` returns `false` and AI actions proceed to the real Anthropic SDK. This is the default production behavior — no mock code path is active unless explicitly opted in.

**Empty/missing CLAUDE_MOCK_DELAY_MS**: If the env var is absent or not a valid number, `MOCK_DELAY_MS` defaults to 100ms. No error is thrown for a missing delay config.

## Non-goals

**CI pipeline wiring (WOR-108)**: This ticket does NOT create the `.github/workflows/ci.yml` file or configure GitHub Actions. It only ensures the Playwright config and fixtures are CI-ready (respecting `process.env.CI` for retries/workers).

**Feature-level E2E tests**: The individual test suites listed in TechSpec §10.1 (`auth.spec.ts`, `invite-flow.spec.ts`, `solo-full-flow.spec.ts`, etc.) are NOT part of this ticket. This ticket provides the infrastructure those tests will import.

**AI action integration**: This ticket creates the `convex/lib/claudeMock.ts` module but does NOT modify any Convex action to call it. The actual `if (isClaudeMockEnabled()) { ... }` check inside AI actions will be added when those actions are implemented by their respective tickets.

**Server-side privacy filter**: The response filter (regex + embedding similarity check) is a separate concern, not part of mock infrastructure.
