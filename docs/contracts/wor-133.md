# Contract: WOR-133 — Admin templates list page

## Why this work exists

Admins need a dedicated page to see all mediation templates at a glance — their category, current version, active/archived status, and how many cases are pinned to each. This is the primary entry point for template management (US-14) and must be role-gated so only admins can reach it.

## Files and exports

### `src/routes/AdminTemplatesPage.tsx` (create, connected)

The main page component. It is a **connected component** — it calls `useQuery(api.admin.listAll)` internally, takes no props, and is wrapped by `<AdminRoute>` in `src/App.tsx` (which handles auth/role gating). This matches the established pattern: every page in `src/routes/` is a connected component that owns its own data fetching (see `DashboardPage`, `ClosedCaseView`, etc.).

Exports a single named export: `AdminTemplatesPage`.

The component renders:

- A page header with an `<h1>` ("Templates") and a `+ New Template` button.
- A semantic `<table>` with columns: Category, Name, Current Version, Status, Pinned Cases.
- Each row is clickable (navigates to `/admin/templates/:id`).
- Archived rows receive muted/grayed styling (e.g. `opacity: 0.6` or a CSS class).
- Status column uses a badge/pill — "Active" or "Archived".
- Loading state: skeleton rows (consistent with `DashboardPage` pattern).
- Empty state: the verbatim message from the AC.

### `src/App.tsx` (modify, route)

Remove the inline `AdminTemplatesPage` stub function and replace it with an import from `src/routes/AdminTemplatesPage`. The route registration at `/admin/templates` already exists and wraps with `<AdminRoute>` — no route changes needed.

### `convex/admin.ts` (modify, query)

Enhance the `listAll` query to return enriched rows. Currently it does `ctx.db.query("templates").collect()` — it needs to:

1. For each template, resolve `currentVersionId` to get the `version` number from `templateVersions`.
2. For each template, count the number of cases whose `templateVersionId` points to any version belonging to that template.

The return shape adds two fields per row: `currentVersion: number | null` and `pinnedCasesCount: number`. The query remains admin-gated via `requireAdmin`.

## Data dependencies

### `api.admin.listAll` (query, admin-only)

Returns an array of `TemplateRow` objects. Each row contains all fields from the `templates` table plus:

- `currentVersion` — the `version` number from the `templateVersions` doc pointed to by `currentVersionId`. `null` if `currentVersionId` is not set (shouldn't happen in practice, but defensive).
- `pinnedCasesCount` — the number of `cases` documents whose `templateVersionId` belongs to any `templateVersions` doc with `templateId` equal to this template's `_id`.

Fields consumed by the UI:

- `_id` — used for row click navigation (`/admin/templates/${_id}`)
- `category` — rendered in Category column
- `name` — rendered in Name column
- `currentVersion` — rendered in Current Version column
- `archivedAt` — determines Active vs Archived badge and muted row styling
- `pinnedCasesCount` — rendered in Pinned Cases column

## Invariants

**Admin-only access.** The `/admin/templates` route is wrapped in `<AdminRoute>` which checks `user.role === "ADMIN"` and redirects non-admins to `/dashboard`. The `listAll` query independently enforces `requireAdmin(ctx)` server-side, so even if the UI guard were bypassed, the data call would fail.

**Verbatim empty state.** When the `listAll` query returns an empty array, the page must render exactly: "No templates yet. The app will use a built-in default baseline. Create a template when you want to tune the Coach's behavior per category." No paraphrasing.

**Archived rows visible but muted.** Archived templates (`archivedAt !== undefined`) remain in the table but with visually muted styling. They are not filtered out or hidden.

**Navigation by `_id`.** Row clicks navigate to `/admin/templates/${template._id}`. The route parameter is the Convex document ID, not a slug or numeric ID.

## Edge cases

**Loading state.** While `useQuery(api.admin.listAll)` returns `undefined`, render skeleton rows (3 placeholder rows matching the DashboardPage skeleton pattern). The `+ New Template` button and page header are visible during loading.

**Empty state.** When the query returns an empty array, render the verbatim empty-state message. The `+ New Template` button remains visible above it.

**Error state.** If the user somehow reaches the page without admin role (e.g. role downgraded after page load), the query will throw a `ConvexError` with code `FORBIDDEN`. The component should handle `useQuery` returning `undefined` indefinitely in this case — the `AdminRoute` wrapper will redirect on next render cycle. No special error UI is needed beyond the loading state.

**Mixed active and archived.** Active and archived templates appear in the same table. There is no grouping or separate sections — archived rows are simply muted. The query returns them intermixed.

## Non-goals

**Template creation form.** The `+ New Template` button's click target is out of scope. For this ticket, it navigates to a creation route (e.g. `/admin/templates/new`) or shows a placeholder. The actual creation form is a separate ticket.

**Template edit/detail view.** The `/admin/templates/:id` route is registered but its implementation is a separate ticket.

**Table interactivity.** No sorting, filtering, pagination, or search. The table renders all templates. Given the expected volume (tens, not thousands), this is acceptable for v1.

**Audit log.** The `/admin/audit` page is a separate ticket.

This requires the real auth + routing stack; a unit test with mocked `useConvexAuth` would only test `AdminRoute` in isolation (which has its own tests).

**AC 3 — + New Template button → `e2e/admin-templates.spec.ts` (e2e) + `tests/unit/admin-templates-page.test.tsx` (unit).**
Unit test: assert the button renders with accessible label. E2e test: click the button, verify navigation occurs.

**AC 4 — Row click navigation → `e2e/admin-templates.spec.ts` (e2e).**
