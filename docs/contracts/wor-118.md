# Contract: WOR-118 ‚Äî Private coaching AI action ‚Äî generateAIResponse with streaming

## Why this work exists

When a user sends a message during private coaching, `sendUserMessage` (WOR-117) schedules `generateAIResponse` ‚Äî currently a stub. This ticket replaces that stub with the real AI interaction loop: calling Claude with the correct prompt context, streaming tokens back to the database in real time so the UI can render progressively, and handling errors with retries. This is the core AI integration for the private coaching phase and the first Convex action in the codebase that calls the Anthropic API with streaming.

## Files and exports

### `convex/privateCoaching.ts` (modify, action-module)

This file already exports `myMessages`, `sendUserMessage`, `markComplete`, and a stub `generateAIResponse`. This ticket replaces the stub `generateAIResponse` handler with the full implementation and adds three new internal mutations that the action calls to write to the database (Convex actions cannot write directly ‚Äî they must call mutations).

**`generateAIResponse`** ‚Äî The existing `internalAction` stub is replaced with the full implementation. The handler:

1. Reads the acting user's `partyStates` row (via `by_case_and_user` index) to get form fields (`mainTopic`, `description`, `desiredOutcome`).
2. Reads the acting user's prior `privateMessages` (via `by_case_and_user` index with `[caseId, userId]`) to build `recentHistory`.
3. Calls `assemblePrompt({ role: "PRIVATE_COACH", caseId, actingUserId, recentHistory, context: { formFields } })` ‚Äî no `templateVersion` argument.
4. Inserts a `privateMessages` row with `{ caseId, userId, role: "AI", content: "", status: "STREAMING", createdAt: Date.now() }` via an internal mutation.
5. If `CLAUDE_MOCK=true`: uses `getMockClaudeResponse("PRIVATE_COACH")` and simulates streaming by chunking the canned response and calling `updateStreamingMessage` at `MOCK_DELAY_MS` intervals.
6. If real API: creates an Anthropic client with `process.env.ANTHROPIC_API_KEY`, calls `client.messages.stream({ model: "claude-sonnet-4-5", max_tokens: 4096, system: prompt.system, messages: prompt.messages })`, and listens for text events. Tokens are buffered and flushed to `updateStreamingMessage` at ~50ms intervals.
7. On stream completion: calls `finalizeStreamingMessage` with the full content and total token count (input + output from the API response usage).
8. On API error (non-429): waits 2s, retries once. On second failure: calls `markMessageError`.
9. On 429 rate limit: waits `2s * 2^attempt` (2s first retry), retries once. On second 429: calls `markMessageError`.

**`updateStreamingMessage`** ‚Äî A new `internalMutation` (not client-accessible). Takes `{ messageId, content }` and patches the `privateMessages` row's `content` field. This is called repeatedly during streaming. No auth checks ‚Äî it's internal-only.

**`finalizeStreamingMessage`** ‚Äî A new `internalMutation`. Takes `{ messageId, content, tokens }` and patches the row with `{ content, status: "COMPLETE", tokens }`.

**`markMessageError`** ‚Äî A new `internalMutation`. Takes `{ messageId }` and patches the row with `{ status: "ERROR" }`.

### `tests/unit/generateAIResponse.test.ts` (create, test-infrastructure)

Unit/integration tests using `convex-test` following the established pattern in `tests/unit/privateCoaching.test.ts`. Since `generateAIResponse` is an `internalAction` that calls external APIs, tests will need to mock the Anthropic SDK and/or rely on `CLAUDE_MOCK=true`. The test file uses the same `seedTwoPartyEnv` fixture pattern as `tests/unit/privateCoaching.test.ts` but extended with pre-seeded private messages for context assembly tests.

Tests are in a separate file from `privateCoaching.test.ts` because that file covers the queries/mutations (WOR-117) and is already substantial. The AI action has fundamentally different test concerns (mocking external APIs, streaming behavior, retry logic).

## Data dependencies

**Read (within the action, via `ctx.runQuery` or direct `ctx.db` access in internal context):**

- **`partyStates`** ‚Äî Read via `by_case_and_user` index with `[caseId, userId]` to get the acting user's form fields: `mainTopic`, `description`, `desiredOutcome`. These are injected into the prompt as `[Context ‚Äî my situation]`.
- **`privateMessages`** ‚Äî Read via `by_case_and_user` index with `[caseId, userId]` to get the acting user's full prior message history. Messages are mapped to `PromptMessage[]` format (`{ role: "user" | "assistant", content }` where `USER` ‚Üí `"user"` and `AI` ‚Üí `"assistant"`). Only `COMPLETE` messages are included (skip `STREAMING` and `ERROR`).

**Write (via internal mutations called from the action):**

- **`privateMessages`** ‚Äî Insert one row (the AI response) with initial `status: "STREAMING"`, `content: ""`. Patch `content` repeatedly during streaming. Final patch sets `status: "COMPLETE"`, `tokens: <count>`. On failure, patch sets `status: "ERROR"`.

**External API:**

- **Anthropic API** ‚Äî `POST /v1/messages` with `stream: true`, model `claude-sonnet-4-5`. The API key is `process.env.ANTHROPIC_API_KEY`. The SDK is `@anthropic-ai/sdk` (already in package.json dependencies per TechSpec ¬ß1.2).

## Invariants

### Privacy isolation at the action level

The action reads messages only via the `by_case_and_user` index with the acting user's `userId`. It never uses the `by_case` index (which would return both parties' messages). It never receives or fetches the other party's `userId`. The `assemblePrompt` function with `role: "PRIVATE_COACH"` further enforces this ‚Äî it only uses `formFields` and `recentHistory` from the acting party, ignoring `otherPartyPrivateMessages` and `otherPartySynthesis` even if they were somehow present in the context.

### System prompt is hardcoded, not templated

`assemblePrompt` with `role: "PRIVATE_COACH"` returns `PRIVATE_COACH_SYSTEM_PROMPT` as the system message and ignores `templateVersion`. The action must not pass a `templateVersion` argument. This matches PRD US-06 AC: "No template content is applied to the Private Coach."

### STREAMING row inserted before API call

The action inserts a `privateMessages` row with `status: "STREAMING"` and `content: ""` before initiating the Claude API call. This ensures the UI can show a typing indicator immediately via the reactive `myMessages` query. If the action crashes before calling Claude, the row remains in `STREAMING` state ‚Äî the UI should handle this gracefully (show as loading, eventually timeout).

### Retry semantics

Retries are limited to exactly one retry per failure type. The action does not retry indefinitely. On persistent failure (two consecutive failures), the message row is set to `ERROR` status. The retry delay for non-429 errors is a flat 2s. For 429 errors, the delay uses exponential backoff: `2000 * 2^attempt` ms (2s on first retry). After the second failure, `markMessageError` is called.

### Token batching interval

During streaming, tokens are buffered and flushed to the database at ~50ms intervals (not on every single token event). This prevents excessive mutation calls while keeping the UI responsive. The exact interval may vary slightly due to async scheduling; the contract specifies ~50ms as the target, not a hard guarantee.

## Edge cases

### Loading state

When `generateAIResponse` starts, it inserts a `STREAMING` row with empty content. The `myMessages` query will return this row immediately. The UI renders it as a typing/loading indicator. Content grows as tokens stream in.

### Empty message history

If the user's first message triggers `generateAIResponse`, the `recentHistory` passed to `assemblePrompt` contains only that one user message. The form fields (if present) provide additional context. The AI responds based on whatever context is available.

### Mock mode behavior

When `CLAUDE_MOCK=true`, the action skips the Anthropic SDK entirely. It reads the canned response from `getMockClaudeResponse("PRIVATE_COACH")`, splits it into chunks, and calls `updateStreamingMessage` at `MOCK_DELAY_MS` intervals to simulate streaming. The final call to `finalizeStreamingMessage` uses a synthetic token count (e.g., word count of the mock response). This enables deterministic E2E testing.

### API key missing

If `process.env.ANTHROPIC_API_KEY` is undefined and `CLAUDE_MOCK` is not `true`, the action should call `markMessageError` immediately rather than attempting an API call that will fail with an auth error. This is a deployment configuration issue, not a retryable error.

### Race condition: user sends another message while AI is streaming

This is safe. `sendUserMessage` schedules a new `generateAIResponse` call. Two actions may run concurrently, each writing to their own `privateMessages` row. The `myMessages` query returns all rows sorted by `createdAt`, so the interleaving is deterministic.

### Action timeout

Convex actions have a 10-minute timeout. Claude streaming responses for private coaching should complete well within this limit. If the stream hangs (network issue), the Anthropic SDK's built-in timeout (configurable, default 10min) will fire, triggering the retry logic.

## Non-goals

- **No UI changes** ‚Äî The private coaching chat UI already renders `STREAMING`, `COMPLETE`, and `ERROR` message states via `myMessages`. No frontend work is needed.
- **No transcript compression** ‚Äî For v1 private coaching, the full message history is passed to Claude. Compression (TechSpec ¬ß6.4) is a separate concern that applies when transcripts grow past the context budget.
- **No cost tracking** ‚Äî Token counts are recorded in the `tokens` field for future budget enforcement, but no per-case cost cap logic is implemented in this ticket.
- **No privacy response filter** ‚Äî The PRIVATE_COACH role never has access to the other party's data, so the post-generation substring filter (TechSpec ¬ß6.3.2) does not apply here. It applies to Synthesis and Coach roles only.
- **No changes to existing exports** ‚Äî `myMessages`, `sendUserMessage`, and `markComplete` are unchanged from WOR-117.
