# Contract: WOR-130 — Closure UI — proposal modal + confirmation banner

## Why this work exists

Case closure is the terminal lifecycle gate in Clarity's mediation flow. Currently the JointChatView has a minimal inline ClosureModal stub (just a textarea + Walk Away / Propose Resolution). This ticket replaces it with the full three-path closure modal per DesignDoc §4.11 (Resolved, Not resolved, Take a break) and adds the confirmation banner that the other party sees when closure has been proposed. Without this work, cases cannot be closed through the intended bilateral confirmation flow.

## Files and exports

### `src/components/chat/ClosureModal.tsx` (create, presentational)

A **presentational** component — it receives callback props for the two mutations (`onProposeClosure`, `onUnilateralClose`) and the other party's name. It does not call Convex hooks directly. The parent (`JointChatView`) wires it up to the mutations.

The component uses internal state to track which closure path the user has selected (`idle` | `resolved` | `not-resolved`). When `idle`, it shows the three option buttons. Selecting "Resolved" or "Not resolved" transitions to the appropriate sub-view. "Take a break" calls `window.close()` (closing the browser tab) and leaves the case in JOINT_ACTIVE.

The Dialog wraps the existing `src/components/ui/Dialog.tsx` Radix primitive. The `DialogContent` receives a CSS class that sets `max-width: 480px` and `border-radius: 20px` per StyleGuide §6.12.

This replaces the inline `ClosureModal` function currently defined inside `JointChatView.tsx` (lines 58-155).

### `src/components/chat/ClosureConfirmBanner.tsx` (create, presentational)

A **presentational** banner component. It receives the proposer's summary text, proposer name, and `onConfirm`/`onReject` callbacks. It renders:

- An icon/indicator (the 📬 glyph from DesignDoc §4.11)
- Text: `"{proposerName} has proposed resolving this case."`
- The summary text quoted
- Two buttons: "Confirm" and "Reject and keep talking"

The banner is positioned above the message input, below the message list — inserted into JointChatView's layout between `ChatWindow` and the input area.

### `src/routes/JointChatView.tsx` (modify, connected)

Modifications:

1. **Remove** the inline `ClosureModal` function (lines 58-155).
2. **Import** `ClosureModal` from `../components/chat/ClosureModal` and `ClosureConfirmBanner` from `../components/chat/ClosureConfirmBanner`.
3. **Add** `useMutation(api.jointChat.confirmClosure)` and `useMutation(api.jointChat.rejectClosure)`.
4. **Add** `useQuery(api.cases.partyStates, queryArgs)` to read `other.closureProposed` for banner visibility and `self` role to determine who is the proposer.
5. **Derive** `showConfirmBanner` from `partyStates.other?.closureProposed === true`.
6. **Render** `<ClosureConfirmBanner>` conditionally between `ChatWindow` and the input area when `showConfirmBanner` is true, passing `caseDoc.closureSummary` and the other party's name.
7. **Pass** `otherPartyName` to `ClosureModal` (from case data or partyStates context).
8. After `confirmClosure` succeeds, navigate to `/cases/${caseId}/closed`.

### `convex/cases.ts` (modify, query)

The `partyStates` query's `other` return object currently has shape `{ role, hasCompletedPC }`. Extend it to include `closureProposed: boolean` so the client can detect when the other party has proposed closure:

```typescript
const other = otherPartyState
  ? {
      role: otherPartyState.role,
      hasCompletedPC: Boolean(otherPartyState.privateCoachingCompletedAt),
      closureProposed: Boolean(otherPartyState.closureProposed),
    }
  : null;
```

This is the only backend change. All closure mutations (`proposeClosure`, `confirmClosure`, `unilateralClose`, `rejectClosure`) already exist in `convex/jointChat.ts`.

## Data dependencies

### `api.cases.get` ({ caseId })

Returns the full case document. This work reads: `status` (route guard + banner logic), `closureSummary` (displayed in the confirmation banner), `isSolo`, `initiatorUserId`.

### `api.cases.partyStates` ({ caseId, viewAsRole? })

Returns `{ self, other }`. This work reads: `other.closureProposed` (boolean — when true, show the confirmation banner). The `self.role` field determines whether the current user is the proposer or the responder.

### `api.jointChat.proposeClosure` ({ caseId, summary, viewAsRole? })

Called when user clicks "Propose Resolution" in the Resolved path. Sets `closureProposed = true` on caller's partyState and writes `closureSummary` on the case doc.

### `api.jointChat.unilateralClose` ({ caseId, viewAsRole? })

Called when user clicks "Close without resolution" in the Not resolved path. Transitions case to `CLOSED_UNRESOLVED` immediately.

### `api.jointChat.confirmClosure` ({ caseId, viewAsRole? })

Called when the other party clicks "Confirm" on the banner. Sets both parties' `closureConfirmed = true` and transitions case to `CLOSED_RESOLVED`.

### `api.jointChat.rejectClosure` ({ caseId, viewAsRole? })

Called when the other party clicks "Reject and keep talking" on the banner. Clears the proposer's `closureProposed` flag, causing the banner to disappear reactively.

## Invariants

**Dialog, not browser confirm:** The modal must use the Radix Dialog primitive from `src/components/ui/Dialog.tsx`. Using `window.confirm()` or `window.alert()` is forbidden. The E2E tests should verify the `cc-dialog-content` class is present.

**Max-width 480px, radius 20px:** The DialogContent for the closure modal must have `max-width: 480px` and `border-radius: 20px` applied via a CSS class (e.g. `cc-closure-modal`). The unit test should assert these computed styles.

**Required textarea for Resolved path:** The "Propose Resolution" button must be `disabled` when the textarea value is empty or whitespace-only. The textarea has `rows={5}` (not 3 as in the current stub).

**Optional textarea for Not resolved path:** The "Close without resolution" button is always enabled regardless of textarea content. The textarea reason is passed to `unilateralClose` if the mutation supports it (current implementation ignores it — that's acceptable for v1).

**Take a break is a no-op:** Clicking "Take a break" calls `window.close()` and does nothing to the backend. If the browser prevents `window.close()` (common when the page was not opened by script), the modal simply closes. No mutation fires.

**Reactive banner:** The confirmation banner appears and disappears reactively. When Party A proposes closure, Party B's `useQuery(api.cases.partyStates)` fires with `other.closureProposed === true`, and the banner renders. When Party B rejects, `closureProposed` is cleared, the query updates, and the banner disappears — no page refresh needed.

**Focus management (NFR-A11Y):** The Radix Dialog already handles focus trapping and return. The confirmation banner's "Confirm" button should call `ref.focus()` when the banner mounts, using a `useEffect` + ref pattern.

## Edge cases

**Loading state:** While `partyStates` query is `undefined`, do not render the banner. The existing loading spinner gate in JointChatView covers this — add `partyStates` to the undefined check.

**Take a break + window.close() blocked:** If the browser blocks `window.close()` (the page was navigated to directly, not opened via `window.open()`), fall back to closing the modal. The user sees the modal close but stays on the page. This is the expected browser behavior and needs no special error handling.

**Concurrent closure proposals:** If both parties click "Propose Resolution" simultaneously, both partyStates get `closureProposed = true`. The confirmation banner logic checks `other.closureProposed` — both parties would see each other's banner. Clicking "Confirm" on either side calls `confirmClosure`, which validates both parties have proposed and transitions to `CLOSED_RESOLVED`. This is correct behavior — no special handling needed.

**Modal state reset on close:** When the modal is dismissed (Cancel or overlay click), the internal state (`idle` | `resolved` | `not-resolved`) and textarea values should reset so reopening the modal shows the initial three-option view.

**Case closed while modal is open:** If the case transitions to `CLOSED_RESOLVED` or `CLOSED_UNRESOLVED` while the modal is open (e.g., the other party unilaterally closed), the route guard in JointChatView detects `status !== "JOINT_ACTIVE"` and redirects to `/cases/${caseId}`. The modal disappears as part of the redirect.

## Non-goals

**Closed case view UI** — The page at `/cases/:caseId/closed` is a separate ticket. After `confirmClosure`, this work navigates there, but the destination page is out of scope.

**Email notifications** — Notifying parties of closure events via email is deferred to v1.1 (US-16).

**Cooling-off period for unilateral close** — The TechSpec mentions a 24h cooling-off for `CLOSED_UNRESOLVED` but flags it as v1.1. This ticket implements immediate unilateral close.

**"Optionally posts a message to the other party" on reject** — The AC mentions this but the current `rejectClosure` mutation does not insert a joint message. This ticket clears the proposal flag only. Posting a message on reject is a v1.1 enhancement.

- Selecting "Not resolved" shows warning styling, optional textarea, warning message, Close without resolution and Cancel buttons

- Modal uses `cc-dialog-content` class with max-width 480px and border-radius 20px

- Not resolved path: select Not resolved → Close without resolution → assert case status is `CLOSED_UNRESOLVED`
- Take a break: select Take a break → assert modal closes → assert case remains `JOINT_ACTIVE`
- Reject: propose closure → toggle → click "Reject and keep talking" → assert banner disappears → assert case remains `JOINT_ACTIVE`
