# Contract: WOR-123 — ReadyForJointView page — synthesis display + Enter Joint Session CTA

## Why this work exists

After both parties complete private coaching, the system generates a per-party synthesis — personalized guidance for the upcoming joint session. The ReadyForJointView is the critical pause-and-prepare screen that shows this synthesis and gates entry into the joint session. It ensures each party absorbs the Coach's guidance before the high-stakes joint conversation begins.

## Files and exports

### `src/routes/ReadyForJointView.tsx` (create, connected)

The page component for the `/cases/:caseId/ready` route. It is a **connected component** — it calls `useQuery(api.jointChat.mySynthesis)`, `useQuery(api.cases.get)`, `useQuery(api.cases.otherPartyName)`, `useMutation(api.jointChat.enterSession)`, and `useSoloActingParty` directly. It takes no props from the route (reads `caseId` from `useParams`). This follows the exact pattern of `CasePrivatePage.tsx`.

The component renders:
1. An intro paragraph: "You've both completed private coaching. Here's what the Coach has prepared for you before the joint session:"
2. A `PrivacyBanner` (existing component from `src/components/ui/PrivacyBanner.tsx`) with copy `"🔒 Private to you — [Name] has their own version"` where `[Name]` is the other party's display name.
3. A synthesis card `<div>` with class `cc-synthesis-card` styled with `background: var(--private-tint)`, `padding: 32px`, `border-radius: 14px`. The banner and card share a border (no gap) per StyleGuide §6.8.
4. Inside the card: the synthesis text rendered as markdown (using the same markdown renderer used elsewhere in the project, e.g. `react-markdown`). The synthesis contains three H3 sections: **Areas of likely agreement**, **Points that will need real discussion**, **Suggested approach**.
5. A transition prompt: "Take your time reading this. When you're ready:"
6. A primary CTA button: "Enter Joint Session →" — using `cc-btn cc-btn-primary` classes, `size="lg"`. This is the **sole primary action** on the page.
7. Below the CTA: "[Name] will see you've entered when they enter too." where `[Name]` is the other party's display name.

On CTA click: calls `enterSession` mutation, then navigates to `/cases/${caseId}/joint`. Uses `useNavigate` from react-router-dom.

Loading state: renders `<LoadingSpinner />` while queries return `undefined`.

Guard: if `caseDoc.status` is not `READY_FOR_JOINT`, redirects to `/cases/${caseId}` (the orchestrator will handle the correct phase).

### `convex/jointChat.ts` (create, query module)

Creates the `jointChat` Convex module. For this ticket, it contains only two functions; T30 will extend this file with `messages`, `sendUserMessage`, `proposeClosure`, `confirmClosure`, and `unilateralClose`.

**`mySynthesis` query:** Reads the caller's `partyStates.synthesisText`. Accepts `caseId` and optional `viewAsRole` (for solo mode). Returns `{ text: string } | null`. If `synthesisText` is not set on the party state, returns `null`. Auth: requires authenticated user who is a party to the case (uses `requireAuth` + `requirePartyToCase` from `convex/lib/auth.ts`). For solo mode with `viewAsRole`, finds the partyState by `caseId + userId` filtered by `role` matching `viewAsRole`.

**`enterSession` mutation:** Transitions the case from `READY_FOR_JOINT` to `JOINT_ACTIVE`. Calls `validateTransition(caseDoc.status, "START_JOINT")` from `convex/lib/stateMachine.ts`. Patches the case's `status` and `updatedAt`. Accepts optional `viewAsRole` for solo mode consistency (the mutation is user-scoped so `viewAsRole` is only used for auth validation, not for different behavior). Auth: requires authenticated user who is a party to the case.

## Data dependencies

### `api.jointChat.mySynthesis`
Returns `{ text: string } | null`. The `text` field is the full synthesis markdown generated by `convex/synthesis.ts` (T28, already merged). It contains three H3 headings: "Areas of likely agreement", "Points that will need real discussion", "Suggested approach", each followed by prose paragraphs. The component renders this markdown directly — it does not parse or restructure the headings.

### `api.cases.get`
Returns the full case document. This work consumes `status` (to guard that the page is only shown for `READY_FOR_JOINT`) and `category` (passed to `PhaseHeader` as `caseName`).

### `api.cases.otherPartyName`
Returns `{ displayName: string | null }`. The `displayName` is substituted into the privacy banner ("Private to you — [displayName] has their own version") and the below-CTA message ("[displayName] will see you've entered when they enter too."). If `displayName` is null (edge case: other party has no name set), fall back to "the other party".

### `api.cases.partyStates`
Returns `{ self: PartyState, other: { role, hasCompletedPC } | null }`. Used to pass `viewAsRole` in solo mode. The `self` object is not directly rendered on this page (synthesis data comes from `mySynthesis` query instead).

### `api.jointChat.enterSession` (mutation)
Takes `{ caseId, viewAsRole? }`. Transitions case to `JOINT_ACTIVE`. No return value needed — on success the component navigates away.

## Invariants

### Synthesis is per-party private data
The `mySynthesis` query reads only the authenticated caller's `partyStates.synthesisText`. It never returns the other party's synthesis. This is enforced at the database query level (filter by `userId`). The UI never has access to the other party's synthesis text.

### Page only reachable in READY_FOR_JOINT status
The `CaseDetailPage` orchestrator redirects to `/cases/:caseId/ready` only when `status === "READY_FOR_JOINT"`. The `ReadyForJointView` itself also guards: if `caseDoc.status !== "READY_FOR_JOINT"`, it redirects back to `/cases/${caseId}`. This prevents direct URL navigation to `/ready` when the case is in another phase.

### State machine transition via validateTransition
The `enterSession` mutation uses `validateTransition(status, "START_JOINT")` — the same state machine used by all other transitions. If the case is not in `READY_FOR_JOINT`, the mutation throws a `CONFLICT` error (409). This prevents double-entry or race conditions.

### Read-only page
No input controls, text fields, or other mutations exist on this page. The only interactive element is the "Enter Joint Session →" button.

### Solo mode support
Both `mySynthesis` and `enterSession` accept `viewAsRole` to support solo mode, following the established convention from `CasePrivatePage` and `cases.partyStates`. The `useSoloActingParty` hook manages this via URL query params (`?as=initiator` / `?as=invitee`).

## Edge cases

### Loading state
While `caseDoc`, `synthesis`, or `otherPartyName` queries return `undefined`, render `<LoadingSpinner />`. No partial rendering — all data must be present before showing the page.

### Synthesis is null
If `mySynthesis` returns `null` (synthesis not yet generated, race condition), show a message like "Your synthesis is being prepared…" with a loading indicator instead of the synthesis card. This should be rare since the case transitions to `READY_FOR_JOINT` only after synthesis writes succeed, but network timing could cause a brief null window.

### Other party name is null
If `otherPartyName.displayName` is `null`, substitute "the other party" in the privacy banner and below-CTA message. E.g. "🔒 Private to you — the other party has their own version".

### CTA click error
If `enterSession` mutation fails (e.g. another tab already transitioned the case), catch the error and display it inline. Do not navigate. The user can reload to see the updated state.

### Direct URL access in wrong phase
If a user navigates directly to `/cases/:caseId/ready` when the case is not in `READY_FOR_JOINT` (e.g. already `JOINT_ACTIVE`), the component's status guard redirects to `/cases/${caseId}`, which the orchestrator resolves to the correct phase view.

## Non-goals

- **"View my guidance" link in joint chat top nav**: This ticket ensures synthesis data remains queryable via `mySynthesis` after the case transitions to `JOINT_ACTIVE`. The actual link in the joint chat header is part of the joint chat page work (T30/T31).
- **Joint chat page**: This ticket navigates to `/cases/:caseId/joint` after CTA click but does not build that page. The existing stub in `App.tsx` handles the route.
- **Full jointChat Convex module**: Only `mySynthesis` and `enterSession` are created here. The remaining functions (`messages`, `sendUserMessage`, `proposeClosure`, `confirmClosure`, `unilateralClose`, `generateCoachResponse`) belong to T30/T31.
- **PhaseHeader integration within ReadyForJointView**: The `ReadyForJointView` page renders its own `PhaseHeader` with the case name and "Ready for Joint Session" phase label, plus `PartyToggle` for solo mode — matching `CasePrivatePage`'s pattern. It does NOT reuse `CaseDetailPage`'s header since it's a separate route.
