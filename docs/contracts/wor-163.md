# Contract: WOR-163 — requireAuth resolves user by identity.email (always undefined)

## Why this work exists

With `@convex-dev/auth`, the `createOrUpdateUser` callback returns the user's `_id`, which Convex Auth stores as `identity.subject` on the identity token. The `email` field on the identity is **not populated** by Convex Auth — it remains `undefined`. The current `requireAuth` and `requireAdmin` functions look up the user via `identity.email` using the `by_email` index, which always fails because `identity.email` is `undefined`. This causes every authenticated query/mutation to 401.

## Files and exports

### `convex/lib/auth.ts` (modify)

The core auth helper module. The fix changes `requireAuth` and `requireAdmin` to:
1. Extract `identity.subject` (which is the user's `_id` as a string).
2. Use `ctx.db.get(userId)` for a direct document lookup by `_id` instead of querying the `by_email` index.

`getUserByEmail` remains unchanged — it is still called by the `createOrUpdateUser` callback in `convex/auth.ts` which receives the email from the OAuth/magic-link profile.

`requirePartyToCase` remains unchanged — it receives a `userId` parameter and never calls `getUserIdentity`.

### `convex/users.ts` (modify)

The `me` query currently extracts `identity.email` and queries `by_email`. It must be changed to extract `identity.subject`, cast it to `Id<"users">`, and use `ctx.db.get(userId)`. The null-safety behavior is preserved: returns `null` when unauthenticated or when the user doc doesn't exist.

## Data dependencies

- `ctx.auth.getUserIdentity()` — returns `UserIdentity | null`. With `@convex-dev/auth`, the `subject` field contains the Convex `_id` of the user (as returned by `createOrUpdateUser`). The `email` field is **not** populated.
- `ctx.db.get(id)` — direct document fetch by `_id`. Returns `Doc<"users"> | null`. This replaces the previous `ctx.db.query("users").withIndex("by_email", ...).unique()` pattern.

## Invariants

1. **`identity.subject` is the user's `_id`**: This is guaranteed by the `createOrUpdateUser` callback in `convex/auth.ts` which returns `user._id`. Convex Auth stores this as the `subject` claim on the identity token.

2. **Error shape preserved**: `requireAuth` must continue to throw `ConvexError({ code: "UNAUTHENTICATED", message: string, httpStatus: 401 })` on failure, matching TechSpec §7.4 and existing test expectations.

3. **`users.me` returns null, never throws**: For graceful client-side handling of loading states.

4. **`by_email` index retained**: Still needed by `getUserByEmail` (called from `createOrUpdateUser` during sign-in).

## Edge cases

### identity.subject is undefined or empty
This should not happen with `@convex-dev/auth` — the library always sets `subject` when `createOrUpdateUser` returns an ID. If it somehow occurs, `ctx.db.get()` will return `null` and `requireAuth` will throw UNAUTHENTICATED. No special handling needed.

### User row deleted after session issued
If a user's row is deleted but their session token is still valid, `ctx.db.get(userId)` returns `null`. `requireAuth` throws UNAUTHENTICATED. Same behavior as before (the `by_email` lookup would also return null).

### Existing tests use `withIdentity({ email: "..." })`
The `convex-test` library's `withIdentity` sets whatever fields you pass on the mock identity. Existing tests that pass `{ email: "..." }` will need to also pass `{ subject: "<user-id>" }` (or switch to subject-only). The test author will update the test helpers accordingly.

## Non-goals

- **Removing the `by_email` index** — still used by `getUserByEmail` in the auth callback.
- **Changing `convex/auth.ts`** — the `createOrUpdateUser` callback works correctly; the bug is only in the downstream resolution logic.
- **Adding new schema indexes** — `ctx.db.get(_id)` is O(1) and doesn't need an index.
- **Changing `requirePartyToCase`** — it already receives a resolved `userId` parameter.

## Test coverage

- **AC1 (requireAuth uses subject)**: Covered by `tests/unit/auth.test.ts`. Existing tests for `requireAuth` will be updated to use `withIdentity({ subject: userId })` pattern. Verifies that a user created with a known `_id` is returned when identity's `subject` matches that `_id`.
- **AC2 (requireAdmin uses subject)**: Covered by `tests/unit/auth.test.ts` (AC4 section). Same pattern — `withIdentity({ subject: adminUserId })`.
- **AC3 (users.me uses subject)**: Covered by `tests/unit/users-typecheck.test.ts`. Tests updated to pass `subject` on identity and verify user doc is returned.
- **AC4 (authenticated flows work end-to-end)**: Covered implicitly by AC1-AC3 unit tests which simulate the full identity→user resolution path.
- **AC5 (no regression on unauth)**: Covered by existing tests in `tests/unit/auth.test.ts` that call `requireAuth` with no identity and verify UNAUTHENTICATED error. These tests remain unchanged.
