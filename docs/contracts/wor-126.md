# Contract: WOR-126 — JointChatView page — shared conversation with Coach

## Why this work exists

The joint chat is the culmination of the mediation flow. After private coaching and synthesis, both parties enter a facilitated conversation with a neutral AI Coach. This page must render party-colored message bubbles, visually distinct Coach messages, and support the "Draft with Coach" workflow — all updating reactively in real time via Convex subscriptions.

## Files and exports

### `src/routes/JointChatView.tsx` (create, connected component)

This is the primary page component for the `/cases/:caseId/joint` route. It is a **connected** component — it calls Convex hooks internally (`useQuery`, `useMutation`) and the `useSoloActingParty` hook. It takes no props (route params come from `useParams`).

It composes existing shared components:
- `ChatWindow` for the message list with auto-scroll
- `MessageBubble` for individual messages (mapping `jointMessages` fields to `BubbleVariant`)
- `MessageInput` for the direct-send textarea
- `PhaseHeader` for the header
- `PartyToggle` for solo mode
- `LoadingSpinner` for loading state

It adds page-specific UI:
- "✨ Draft with Coach" button (renders a button + a mount-point div for `DraftCoachPanel`, does NOT import DraftCoachPanel itself — that's a sibling task)
- "My guidance" link that opens a side panel / modal showing synthesis text
- "Close" button that opens a closure modal (proposeClosure + unilateralClose)
- "Coach is thinking..." inline placeholder for STREAMING Coach messages
- Mapping logic: `jointMessages[].authorType` + `authorUserId` + case's `initiatorUserId` → correct `BubbleVariant`

### `src/App.tsx` (modify)

Replace the inline stub `CaseJointPage` with an import of `JointChatView` from `./routes/JointChatView`. The route path `/cases/:caseId/joint` is already defined.

## Data dependencies

### `api.cases.get` ({ caseId })
Returns the case document. This page reads: `status` (must be `JOINT_ACTIVE`), `isSolo`, `initiatorUserId`, `inviteeUserId`, `category`.

### `api.jointChat.messages` ({ caseId, viewAsRole? })
Returns `JointMessage[]` sorted by `createdAt` ascending. Each message has: `_id`, `caseId`, `authorType` ("USER" | "COACH"), `authorUserId` (null for Coach), `content`, `status` ("STREAMING" | "COMPLETE" | "ERROR"), `isIntervention` (boolean), `createdAt`.

The page maps these to `ChatMessage` for `ChatWindow`:
- `authorType === "COACH"` + `isIntervention === true` → variant `"coach-intervention"`
- `authorType === "COACH"` + `isIntervention !== true` → variant `"coach-joint"`
- `authorType === "USER"` + `authorUserId === case.initiatorUserId` → variant `"party-initiator"`
- `authorType === "USER"` + `authorUserId !== case.initiatorUserId` → variant `"party-invitee"`

For Coach messages with `status === "STREAMING"`, the rendered content is `"Coach is thinking..."` (not the partial streaming content — the ChatWindow will show the streaming indicator via the status prop).

### `api.jointChat.mySynthesis` ({ caseId, viewAsRole? })
Returns `{ text: string } | null`. Used for the "My guidance" side panel.

### `api.jointChat.sendUserMessage` ({ caseId, content, viewAsRole? })
Mutation called when user clicks Send. Returns the new message ID.

### `api.jointChat.proposeClosure` ({ caseId, summary, viewAsRole? })
Called from the closure modal when user proposes resolution.

### `api.jointChat.unilateralClose` ({ caseId, viewAsRole? })
Called from the closure modal when user walks away.

## Invariants

**Route guard:** The page checks `case.status === "JOINT_ACTIVE"`. If not, it renders a `<Navigate to={/cases/${caseId}} replace />`. This prevents accessing joint chat before the session starts or after closure.

**No auto-send:** The "✨ Draft with Coach" button opens a panel. No mechanism on this page auto-posts to the joint chat. The only path to send is the explicit Send button on `MessageInput`.

**Coach streaming placeholder:** When a Coach message has `status === "STREAMING"`, the bubble shows "Coach is thinking..." text. This avoids showing partial AI output that may be rejected by the privacy filter. The `StreamingIndicator` component (blinking cursor) is also shown.

**Auto-scroll:** Uses the existing `ChatWindow` scroll behavior (already implemented with `isAtBottomRef` pattern). The `ChatWindow` component already handles this — messages array changes trigger scroll-to-bottom only when user is at the bottom.

**Party isolation:** The page never fetches private messages. It only reads `jointChat/messages` (shared) and `jointChat/mySynthesis` (own party only).

## Edge cases

**Loading state:** While `caseDoc`, `messages`, or `synthesis` queries are `undefined`, render `<LoadingSpinner />`. This matches the pattern in `ReadyForJointView`.

**Empty state (no messages yet):** When `messages` returns an empty array, render the chat window with no bubbles. The Coach's opening message will appear reactively once generated server-side.

**Error state (Coach message with status ERROR):** Render with `variant="error"` and display the Retry button. The Retry button calls `sendUserMessage` with the last user message content to re-trigger Coach generation (or a dedicated retry mechanism if available).

**Synthesis unavailable:** If `mySynthesis` returns `null`, the "My guidance" panel shows a fallback message like "Synthesis not available."

**Solo mode:** When `case.isSolo`, pass `viewAsRole` to all queries/mutations. Show `PartyToggle` in the header. Messages from both "parties" render with their respective colors even though it's the same user.

## Non-goals

- **DraftCoachPanel implementation** — built in sibling task. This page renders a `<div id="draft-coach-mount" />` or similar mount point and the trigger button only.
- **Coach AI generation logic** — already fully implemented in `convex/jointChat.ts` (WOR-125).
- **Closure confirmation modal UI details** — the modal is minimal: a textarea for summary + Confirm/Cancel for proposeClosure, or a simple "Walk away" for unilateralClose.
- **Message propagation performance tuning** — Convex reactive queries already meet the <500ms target; no client-side optimization needed.
