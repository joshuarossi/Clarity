# Contract: WOR-120 — Solo mode — PartyToggle component + useSoloActingParty hook

## Why this work exists

Solo mode lets a single developer or demo user exercise the full two-party mediation flow in one browser session. Without a party toggle, there's no way to switch perspective between initiator and invitee — the user would see the same data regardless of which "hat" they're wearing. This ticket builds the toggle UI and the hook that drives it, plus the minimal backend changes to ensure private coaching messages are properly isolated per party role in solo mode.

## Files and exports

### `src/components/layout/PartyToggle.tsx` (create, presentational)

A pure presentational segmented control. It receives party names and the active selection as props and calls `onToggle` when the user clicks. It has no data dependencies — all state management is handled by the parent via `useSoloActingParty`. This is intentionally presentational (not connected) because: (a) the toggle is reused across multiple pages (private coaching, joint chat, synthesis — though only private coaching is wired in this ticket), and (b) the parent page already has the case data needed to determine `isSolo` and party names.

Uses the CSS classes `.party-toggle`, `.party-toggle-label`, `.party-toggle-btn`, `.party-toggle-btn[data-active="true"]` which are defined in `components.css` per StyleGuide §6.10. The "VIEWING AS" label is rendered as a `<span>` with class `.party-toggle-label` (11px uppercase, `--coach-accent` color, 0.05em letter-spacing).

### `src/hooks/useSoloActingParty.ts` (create, hook)

Reads the `?as=initiator|invitee` URL search param via React Router's `useSearchParams`. Reads the case doc via `useQuery(api.cases.get, { caseId })` to determine `isSolo`. Returns `{ actingRole, isSolo, setActingParty }`.

**Critical design note:** In solo mode, `initiatorUserId === inviteeUserId` (the same user plays both roles). The hook returns the _role_ ("INITIATOR" | "INVITEE") rather than a meaningful userId distinction, because the userId is the same for both. Consuming components use `actingRole` to pass the correct `partyRole` and `viewAsRole` args to Convex queries.

`setActingParty` is a thin wrapper around `setSearchParams` that updates `?as=` without a full navigation.

### `convex/schema.ts` (modify, config)

Adds to the `privateMessages` table definition:

- `partyRole: v.optional(v.union(v.literal("INITIATOR"), v.literal("INVITEE")))` — only set for solo cases. Non-solo messages omit this field entirely.
- New index: `by_case_user_role: ["caseId", "userId", "partyRole"]` — enables efficient filtering in solo mode.

### `convex/privateCoaching.ts` (modify, query + mutation)

**`myMessages`**: Adds optional `partyRole` arg. When provided, filters results to only messages matching that `partyRole`. When absent (non-solo callers), behaves exactly as before — returns all messages for the authenticated user on the case.

**`sendUserMessage`**: Adds optional `partyRole` arg. When provided, stores it on the inserted `privateMessages` row. The scheduled `generateAIResponse` action already receives the `userId` — the partyRole tag on the user's message means the AI response (which is inserted by `insertStreamingMessage`) must also receive and store the same `partyRole`. The internal mutations `insertStreamingMessage` and `finalizeStreamingMessage` are updated to accept optional `partyRole`.

### `convex/cases.ts` (modify, query)

**`partyStates`**: Adds optional `viewAsRole` arg. When provided and the case is solo, uses that role to determine which partyState is `self` and which is `other`. When absent, existing behavior is preserved (solo defaults to INITIATOR as self).

### `src/routes/CasePrivatePage.tsx` (modify, connected)

Imports and calls `useSoloActingParty(caseId)`. When `isSolo` is true:

- Passes `partyRole: actingRole` to `myMessages` and `sendUserMessage`
- Passes `viewAsRole: actingRole` to `partyStates`
- Renders `<PartyToggle>` inside the page header area (as a child slotted into the top of the page layout, visually positioned top-right per the AC)
- Resolves initiator/invitee names from case data and user doc for the toggle labels

When `isSolo` is false, the hook returns `isSolo: false` and the component behaves exactly as it does today.

### `src/styles/components.css` (modify, config)

Adds the four CSS rules from StyleGuide §6.10:

- `.party-toggle` — inline-flex container with `--coach-subtle` background and `--coach-accent` 1px border
- `.party-toggle-label` — 11px uppercase label with `--coach-accent` color and 0.05em letter-spacing
- `.party-toggle-btn` — 13px button with transparent background, secondary text color
- `.party-toggle-btn[data-active="true"]` — `--bg-surface` background, primary text color, `--shadow-1` box-shadow

## Data dependencies

**`api.cases.get`** (existing, read-only): Returns the full case document. This work consumes `isSolo` (to determine whether to show the toggle), `initiatorUserId`, and `inviteeUserId` (to resolve party names via user lookup — though in solo mode they're the same user, the display names come from the partyStates role labels "Initiator" / "Invitee" or the user's displayName).

**`api.cases.partyStates`** (modified): Returns `{ self: PartyState, other: { role, hasCompletedPC } | null }`. With the new `viewAsRole` param, solo mode returns `self` as the partyState matching the requested role. Fields consumed: `self.privateCoachingCompletedAt` (for read-only state), `self.role`.

**`api.privateCoaching.myMessages`** (modified): Returns `Doc<"privateMessages">[]` sorted by `createdAt` ascending. With the new `partyRole` param, solo mode filters to only messages tagged with that role. Fields consumed: `_id`, `role` (USER/AI), `content`, `status`, `createdAt`.

**`api.cases.otherPartyName`** (existing, read-only): Returns `{ displayName: string | null }`. In solo mode this returns the user's own displayName (since both parties are the same user). The toggle labels should use "Initiator" / "Invitee" role labels when the user has no displayName, or the user's displayName with a role suffix like "Alex (Initiator)" / "Alex (Invitee)" when both names would be identical.

## Invariants

**Privacy isolation in solo mode.** When toggled to Initiator, the `myMessages` query with `partyRole: "INITIATOR"` returns only messages tagged as INITIATOR. Invitee messages are not returned, not rendered, not accessible from the client. This mirrors the real two-user case where each user only sees their own messages. The enforcement is server-side in the Convex query — the client cannot bypass it by omitting the `partyRole` param (the query would then return ALL solo messages, but only if `partyRole` is absent, which doesn't happen when the hook is active).

**PartyToggle never renders on non-solo cases.** The `isSolo` check in the hook is the gate. The component itself has no knowledge of solo mode — it renders whatever props it receives. The parent component checks `isSolo` before rendering the toggle.

**URL param is the single source of truth.** The toggle value lives exclusively in `?as=initiator|invitee`. React Router's `useSearchParams` is the read/write mechanism. No React state, no context provider, no localStorage. This ensures refresh persistence and that copying the URL shares the toggle state.

**Default to initiator.** When `?as=` is absent or has an invalid value, the hook defaults to `"INITIATOR"`. This matches the natural flow: the user who created the solo case starts as the initiator.

**Backward compatibility.** The `partyRole` field on `privateMessages` is optional. Existing non-solo messages have `partyRole: undefined`. The modified `myMessages` query only filters by `partyRole` when the arg is explicitly provided. All existing behavior for non-solo cases is preserved.

## Edge cases

**Loading state.** The `useSoloActingParty` hook depends on `useQuery(api.cases.get)`. While the case data is loading (`undefined`), the hook returns `{ actingRole: "INITIATOR", isSolo: false, setActingParty: no-op }`. The PartyToggle is not rendered during loading. Once the case loads, if `isSolo` is true, the toggle appears and the hook becomes active.

**Invalid URL param.** If `?as=` contains a value other than "initiator" or "invitee" (e.g., `?as=foo`), the hook treats it as absent and defaults to "initiator". No error is shown.

**Party names when displayName is null.** If the user has no displayName set, the toggle buttons show the role labels "Initiator" and "Invitee" as fallback text. Since both parties in a solo case are the same user, showing the same displayName for both buttons would be confusing — when names would be identical, append the role: "Alex (Initiator)" / "Alex (Invitee)".

**Non-solo case with ?as= param.** If a URL somehow has `?as=initiator` on a non-solo case, the hook returns `isSolo: false` and the param is ignored. The toggle does not render. No error.

## Non-goals

**Joint chat / synthesis / closed view integration.** This ticket only wires the toggle into `CasePrivatePage`. Other pages (joint chat, synthesis view, closed case view) will consume `useSoloActingParty` when those pages are built in their respective tickets.

**Coach-accent full-page banner.** The AC mentions "Solo cases are visually distinct with coach-accent banner per DesignDoc." The party toggle's `--coach-accent` border and `--coach-subtle` background in the header satisfy this. A separate full-page solo banner is not in scope.

**Solo case creation.** Already exists in `NewCasePage` (merged in WOR-110).

**AI prompt modifications.** The `generateAIResponse` action already receives `userId` as a param and reads messages by `userId`. With the new `partyRole` tag on messages, the action's context assembly (`getPrivateMessagesForUser`) will need to filter by `partyRole` for solo cases — this is handled by passing `partyRole` through the internal query. The system prompt itself does not change.
