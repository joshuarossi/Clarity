# Contract: WOR-129 — DraftCoachPanel component — side panel with send gate

## Why this work exists

The Draft Coach Panel is the UX surface that enforces Clarity's core "send gate" principle: no message reaches the other party without explicit user approval. Users privately iterate with an AI coach to craft joint-chat messages in a right-side panel, and only a deliberate "Send this message" click posts the approved draft to the shared conversation. This component makes the Draft Coach backend (WOR-127, WOR-128) accessible through a visual interface.

## Files and exports

### `src/components/joint/DraftCoachPanel.tsx` (connected, create)

The primary panel component. It is **connected** — it calls Convex hooks (`useQuery`, `useMutation`) internally to manage the draft coaching session lifecycle. The parent (`JointChatView`) passes in only the `caseId`, the other party's name (for the privacy banner), an `onClose` callback, and an `onEditBeforeSending` callback that receives the draft text to populate the joint chat input.

The component renders:
1. An overlay panel (420px right-side sheet on desktop, full-screen bottom sheet on mobile < 768px)
2. A header with Sparkles icon, "Draft Coach" title, Lock icon with tooltip, and close button
3. A privacy banner: "This is private to you. {otherPartyName} can't see what you're discussing here."
4. A ChatWindow (reused from `src/components/chat/ChatWindow.tsx`) configured with 14px font size for the narrower surface
5. A "Draft it for me" button that sends the canonical `"Generate Draft"` message to trigger readiness detection
6. A DraftReadyCard when `session.finalDraft` is populated

The panel manages its own draft session: on mount, it calls `draftCoach.startSession` if no active session exists. It subscribes to `draftCoach.session` for reactive message updates.

### `src/components/joint/DraftReadyCard.tsx` (presentational, create)

A pure presentational component. Receives the draft text and four action callbacks via props. Renders:
- The polished draft in a quoted card with `.cc-draft-ready` CSS class and subtle background
- Four vertically stacked buttons: Send (primary) → Edit (secondary) → Keep refining (ghost) → Discard (ghost, danger text)

No hooks, no Convex calls. The parent `DraftCoachPanel` wires the callbacks to the appropriate mutations.

### `src/routes/JointChatView.tsx` (route, modify)

Replace the placeholder `<div id="draft-coach-mount" />` (line 378) with `<DraftCoachPanel>` rendered conditionally on `draftCoachOpen` state. Pass down `caseId`, the other party's display name (derived from case data), `onClose`, and `onEditBeforeSending` (which populates the joint chat input textarea and closes the panel). The `viewAsRole` prop is passed in solo mode.

## Data dependencies

### `api.draftCoach.session` (query)
Returns `{ session: DraftSession, messages: DraftMessage[] } | null`. The `session` object includes `status: "ACTIVE" | "SENT" | "DISCARDED"` and `finalDraft: string | undefined`. Messages have `role: "USER" | "AI"`, `content: string`, `status: "STREAMING" | "COMPLETE" | "ERROR"`, and `createdAt: number`.

The panel maps these messages to `ChatMessage[]` for `ChatWindow`, using variant `"user"` for USER role and `"coach"` for AI role.

### `api.draftCoach.startSession` (mutation)
Called on panel mount if `session` query returns null. Returns the new `sessionId`.

### `api.draftCoach.sendMessage` (mutation)
Args: `{ sessionId, content }`. Sends user text and triggers AI response generation. The "Draft it for me" button sends `content: "Generate Draft"` — the canonical readiness signal recognized by the backend.

### `api.draftCoach.sendFinalDraft` (mutation)
Args: `{ sessionId }`. Posts `session.finalDraft` to joint chat as a `jointMessages` row. This is the **only** path from draft to joint chat.

### `api.draftCoach.discardSession` (mutation)
Args: `{ sessionId }`. Marks session DISCARDED. No message sent.

### `api.draftCoach.retryLastDraftAIResponse` (mutation)
Args: `{ sessionId }`. Deletes the ERROR message and re-triggers generation.

## Invariants

### Send-gate hard rule
No draft text is ever posted to joint chat except through `draftCoach.sendFinalDraft`. The "Send this message" button is the sole trigger. The panel never calls `jointChat.sendUserMessage`. If this invariant is violated, the product's core privacy promise is broken.

### Privacy isolation
The panel only fetches `draftCoach.session` scoped to the current user. It never requests or renders another party's data. The backend enforces this via `requirePartyToCase` + user ID matching, but the UI layer must also never construct a query that could leak.

### Focus trap and accessibility
While the panel is open: `role="dialog"`, `aria-label="Draft Coach"`, keyboard focus trapped inside, Escape closes. Focus moves to the message textarea on open.

### Single active session
The backend enforces one ACTIVE session per case per user. The panel does not need to handle multi-session scenarios.

## Edge cases

### Loading state
While `draftCoach.session` query is loading (returns `undefined`), show a loading spinner within the panel body. The panel shell (header, close button) renders immediately.

### No active session on open
If the session query returns `null`, the panel calls `startSession` and shows a brief loading indicator until the session populates. The initial AI greeting message arrives via the scheduled `generateResponse` action.

### AI error
When a draft message has `status: "ERROR"`, it renders with the error bubble variant and a "Retry" button. Clicking Retry calls `retryLastDraftAIResponse`. No special empty state — the error message replaces the expected AI response inline.

### Session already SENT or DISCARDED
If the user opens the panel and the session query returns a non-ACTIVE session, start a fresh session (the existing non-ACTIVE one is inert).

### "Edit before sending"
The `onEditBeforeSending` callback passes `session.finalDraft` text up to `JointChatView`, which populates its `MessageInput` textarea programmatically and closes the panel. This requires `JointChatView` to expose a controlled value or ref on `MessageInput`.

## Non-goals

- **No drag-to-resize** — panel is fixed 420px wide on desktop.
- **No template management** — the backend applies template instructions server-side; the panel has no template awareness.
- **No multi-session** — only one ACTIVE session per case/user; enforced by backend `startSession`.
- **No panel state persistence** — opening/closing is ephemeral; page reload closes it.
