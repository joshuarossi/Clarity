# Contract: WOR-105 â€” Shared UI primitives â€” PrivacyBanner, StatusPill, PartyAvatar, PhaseHeader

## Why this work exists

Clarity's case screens reuse a small set of UI primitives that communicate privacy, status, identity, and navigation context. Building these four components as standalone presentational primitives prevents visual drift across phases and ensures the privacy visual shorthand (warm beige tint + lock icon) is identical everywhere it appears. Downstream tickets (Private Coaching, Joint Chat, Dashboard, Closure, Draft Coach) compose these rather than reimplementing layout or accessibility behavior.

## Files and exports

### `src/components/ui/PrivacyBanner.tsx` (presentational, create)

Exports a single `PrivacyBanner` function component. It uses the existing `.cc-banner-privacy` CSS class recipe from `components.css` for its container styling (flex layout, `--private-tint` background, gap, padding, border-bottom). The component renders a Lock icon (lucide-react, 16x16, `--text-secondary` color) as a button with `aria-label="Learn more about privacy"`. Clicking the lock icon opens a Dialog modal (max-width 480px, radius 20px per StyleGuide Â§6.12) explaining what's private: the conversation is private, only the user and the AI coach can see it, and none of it is shared with the other party. The `copy` prop is `React.ReactNode` to allow bold/link formatting as shown in the DesignDoc wireframe (e.g., `<strong>Private to you.</strong> Jordan will never see any of it.`).

The component is presentational â€” it takes a `copy` prop and renders it. It does not fetch party names or case data. The consuming screen (Private Coaching, Draft Coach panel, etc.) is responsible for interpolating the other party's name into the copy string.

### `src/components/ui/StatusPill.tsx` (presentational, create)

Exports a single `StatusPill` function component. Renders as an inline-flex element with a dot indicator and a text label. The `variant` prop determines both the dot shape and color:

- `pill-turn`: filled circle dot, `--accent` on `--accent-subtle` background
- `pill-waiting`: hollow circle dot (border only), `--text-secondary` color, outlined style
- `pill-ready`: filled circle dot, `--warning` color
- `pill-closed`: square dot, `--text-tertiary` color

Dot shape is the primary state indicator; color reinforces but never solely carries meaning (WCAG compliance). CSS class recipes `.cc-status-pill` and `.cc-status-pill-dot` are added to `components.css`.

### `src/components/ui/PartyAvatar.tsx` (presentational, create)

Exports a single `PartyAvatar` function component. Renders a circle with white initials centered on a party-colored background. Three sizes: `sm` (24px), `md` (32px, default), `lg` (40px) per StyleGuide Â§6.5. The `name` prop is used to derive initials (first character of each whitespace-separated word, max 2 characters, uppercased). Background color is set via CSS custom property per role:

- `initiator` â†’ `var(--party-initiator)`
- `invitee` â†’ `var(--party-invitee)`
- `coach` â†’ `var(--coach-accent)`

The coach variant may optionally render `âźˇ` instead of initials when `name` is not provided, per StyleGuide Â§6.5. Text color is always white. Border-radius is `--radius-full`. CSS class `.cc-avatar` is added to `components.css`.

This is a new component separate from the existing `TopNav`. `TopNav` is the app-level nav bar (dashboard, account menu). `PhaseHeader` is the in-case phase-level nav bar that sits below `TopNav` on case screens. They serve different purposes and do not share layout.

## Data dependencies

None. All four components are purely presentational. They receive data via props and render it. No Convex queries or mutations are called.

## Invariants

**PrivacyBanner uses `.cc-banner-privacy`:** The existing CSS class recipe in `components.css` (lines 226-237) defines the banner's layout, background color, border, typography, and gap. PrivacyBanner must use this class, not rebuild equivalent styles. This ensures visual consistency â€” if the token values change, all privacy banners update together.

**StatusPill dot shape encodes state independently of color:** Per StyleGuide Â§6.7 and the AC's explicit requirement, the dot shape (filled circle, hollow circle, square) must distinguish states even without color perception. A filled circle = active participation expected, hollow circle = passive waiting, square = terminal state. Color reinforces but does not solely carry the distinction.

**PartyAvatar colors from CSS custom properties only:** Background colors must reference `var(--party-initiator)`, `var(--party-invitee)`, or `var(--coach-accent)` â€” never hardcoded hex values. This ensures dark mode works automatically when `[data-theme="dark"]` switches the custom property values.

**PhaseHeader is exactly 56px tall:** 12px top padding + 32px content + 12px bottom padding = 56px. This is a normative constraint from the StyleGuide, not a suggestion.

**Aria-labels on all icon buttons:** The lock icon button in PrivacyBanner and the back arrow link in PhaseHeader must have `aria-label` attributes. Without them, screen readers announce the buttons as unlabelled, failing WCAG.

**Dark mode via `[data-theme="dark"]`:** The project uses a data-theme attribute on the root element, not Tailwind's `dark:` class prefix. All color values flow through CSS custom properties that are redefined under `[data-theme="dark"]` in `globals.css`. Components should not add any dark-mode-specific logic â€” the CSS custom properties handle it automatically.

**Dialog styling per StyleGuide Â§6.12:** Max-width 480px, padding 24px, radius 20px (softer than cards), shadow-3, overlay with `rgba(0,0,0,0.3)` + 2px backdrop blur, 150ms fade + 95%â†’100% scale animation on enter.

## Edge cases

**Loading state:** Not applicable. These are presentational primitives that render synchronously from props. The consuming screens handle loading states (e.g., skeleton rows on the dashboard before case data loads).

**Empty state â€” PartyAvatar with empty name:** If `name` is an empty string, PartyAvatar renders the circle with no initials (empty string). For the coach role specifically, it renders `âźˇ` when name is empty or not provided. This prevents a blank circle from appearing.

**Empty state â€” StatusPill:** All four variants always have a label. There is no "unknown" variant. The consuming component must map case status to one of the four pill variants.

**PhaseHeader with no children:** The right slot renders empty. The header still displays correctly with left and center content. No placeholder or empty-state content is needed.

**Privacy modal Esc/close:** The Dialog is built on Radix, which handles Esc-to-close and click-outside-to-close natively. The modal should also have an explicit close button (X icon, top-right) per standard Dialog behavior.

**Long case names in PhaseHeader:** Center content should truncate with ellipsis if the combined `caseName Â· phaseName` text would overflow the available center space. CSS `text-overflow: ellipsis` with `overflow: hidden` and `white-space: nowrap` on the center element.

## Non-goals

**No data fetching:** These components do not call Convex queries or mutations. They are not connected components. All data (party name, case name, status, etc.) is passed via props by the consuming screen. This keeps the primitives reusable across different data contexts (real cases, solo mode, admin preview, etc.).

**No responsive/mobile layout:** The AC does not specify mobile breakpoints for these components. PhaseHeader, for instance, may need a mobile adaptation (hamburger menu, collapsed center text), but that is out of scope for this ticket.

**No `avatar-you` variant:** The StyleGuide mentions an "avatar-you" variant using the sage accent color. This is not in the AC for this ticket. The three roles (initiator, invitee, coach) are sufficient for the consuming screens defined in the epic.

**No StatusPill click/interaction:** StatusPill is a read-only indicator. It does not have click handlers or navigation behavior. The dashboard case row handles click-to-navigate at the row level.
