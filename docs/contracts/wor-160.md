# Contract: WOR-160 — Auth broken in deployed app: convex/http.ts missing

## Why this work exists

Convex Auth requires an HTTP router file (`convex/http.ts`) to register its authentication endpoints — OAuth callbacks, magic-link verification URLs, session token refresh, and sign-out. Without this file, the Convex platform reports "HTTP actions not enabled" and all auth flows fail in the deployed app. The auth configuration (`convex/auth.ts` and `convex/auth.config.ts`) is already correct (fixed by WOR-156), but the HTTP routing layer that exposes auth endpoints to the browser was never created.

## Files and exports

### `convex/http.ts` (create, route)

This is the standard Convex HTTP router file. The Convex platform automatically discovers `convex/http.ts` (or `.js`) and uses its default export as the HTTP action router. The file creates an `httpRouter()` instance, calls `auth.addHttpRoutes(http)` to register all Convex Auth endpoints, and exports the router as default.

This file must NOT use the `"use node"` directive — it runs in the Convex JavaScript runtime (not Node.js). It must NOT import Node built-ins.

The file is intentionally minimal. Its only job is to wire up auth HTTP routes. Future tickets may add custom HTTP endpoints (webhooks, API routes), but this ticket's scope is auth enablement only.

### `tests/unit/convex-http.test.ts` (create, test-infrastructure)

Unit tests verifying:

1. `convex/http.ts` exists as a file.
2. The file imports `httpRouter` from `"convex/server"`.
3. The file imports `auth` from `"./auth"` (or `"./auth.js"`).
4. The file calls `auth.addHttpRoutes`.
5. The file exports a default export.
6. The file does NOT contain `"use node"` or Node.js built-in imports (`crypto`, `fs`, `path`, etc.).

These are structural/static tests (reading the source file) because Convex HTTP routers cannot be instantiated in a Vitest environment without the Convex runtime.

## Data dependencies

### `auth` (from `convex/auth.ts`)

The `auth` export from `convex/auth.ts` is the Convex Auth handler created by `convexAuth()`. It exposes an `addHttpRoutes(router)` method that registers the following HTTP routes on the router:

- `POST /.well-known/openid-configuration` — OpenID discovery
- `GET /api/auth/signin/*` — OAuth redirect initiation
- `GET /api/auth/callback/*` — OAuth callback handling
- `POST /api/auth/signin` — magic-link / credential sign-in
- `POST /api/auth/signout` — session invalidation
- `GET /api/auth/session` — session status

The exact route paths are managed by `@convex-dev/auth` internally — the contract does not hardcode them. The important thing is that `addHttpRoutes` is called on the router before export.

## Invariants

**File must exist at `convex/http.ts`.** The Convex platform looks for this exact file path to enable HTTP actions. Without it, all HTTP endpoints (including auth) return 404 or "HTTP actions not enabled."

**Default export must be an httpRouter.** The Convex platform expects the default export to be an `HttpRouter` instance created by `httpRouter()` from `"convex/server"`.

**`auth.addHttpRoutes(http)` must be called.** This registers all Convex Auth HTTP routes. Without this call, the router exists but has no auth endpoints, and auth still fails.

**No Node.js runtime.** The file must NOT use `"use node"` or import Node built-ins. HTTP routers run in the Convex JavaScript environment. (This was the cause of WOR-155 — Node imports in Convex files break deployment.)

## Edge cases

**Empty router (no custom routes).** The router only has auth routes. This is valid — `httpRouter()` with only `auth.addHttpRoutes` registered is the minimal correct configuration.

**Deployment ordering.** When `convex/http.ts` is added to an existing deployment, Convex automatically picks it up on the next `npx convex deploy` or `npx convex dev`. No migration or manual step is needed.

**Local development.** `npx convex dev` watches file changes. Adding `convex/http.ts` during local dev triggers a hot-reload that enables the HTTP routes immediately.

## Non-goals

**No custom HTTP endpoints.** This ticket adds only auth routes. Webhooks, REST APIs, or other HTTP endpoints are out of scope.

**No changes to auth configuration.** `convex/auth.ts` and `convex/auth.config.ts` are already correct (WOR-156). This ticket only adds the missing routing layer.

**No client-side changes.** The Convex React client already sends auth requests to the correct URLs. The problem is server-side: those URLs return 404 because the router doesn't exist.

**No CI workflow changes.** The existing `npx convex deploy` step will automatically pick up the new `convex/http.ts`.
