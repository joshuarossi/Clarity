# Contract: WOR-106 — Seed data script (convex/seed.ts)

## Why this work exists

A fresh Convex deployment has empty tables. Without seed data, every developer must manually create an admin user and templates via the Convex dashboard before they can exercise any case-creation or coaching flows. This script gives every developer a working baseline (one admin user, three default templates with initial versions) the moment they run `npx convex run seed:seed` after spinning up a local Convex deployment.

## Files and exports

### `convex/seed.ts` (create)

An `internalMutation` that bootstraps development data. Using `internalMutation` (rather than `mutation`) ensures the function is never callable from client code — it's a dev tool, not an API endpoint. It is callable via `npx convex run seed:seed` from the CLI and from other server-side Convex functions (e.g., test setup).

The file also exports two constants — `ADMIN_EMAIL` and `DEFAULT_CATEGORIES` — so that tests can import the canonical values rather than hardcoding them. This eliminates a class of test/implementation drift where tests check for one email and the seed creates another.

**Environment gating:** The function checks `process.env.IS_PRODUCTION` (or equivalent Convex environment variable). If the value is `"true"`, the function throws immediately before performing any database writes. In development and test environments, this variable is either unset or set to a non-`"true"` value. The implementation author should use whatever environment detection pattern Convex recommends — the key invariant is that the function refuses to run in production, not the specific mechanism.

**Idempotency strategy:**

1. Query users by `ADMIN_EMAIL` using the `by_email` index. If a matching user exists, skip admin user creation.
2. For each category in `DEFAULT_CATEGORIES`, query templates using the `by_category` index. If a template with that category already exists, skip that template and its version.

**Template version linkage:** After inserting each template, create a `templateVersion` with `version: 1`, a minimal `globalGuidance` string (e.g., `"Default guidance for {category} conflicts."`), and `publishedAt: Date.now()`. Then patch the template's `currentVersionId` to point to the new version's ID. The `publishedByUserId` on the version references the admin user (created or looked up in step 1).

## Data dependencies

None. This function only writes to the database — it does not consume any external queries or APIs. It writes to three tables: `users`, `templates`, and `templateVersions`.

## Invariants

**Production guard.** The seed function must throw before performing any database writes if running in a production environment. This is a hard safety requirement — seed data in production would create a backdoor admin account with a well-known email.

**Idempotency via index lookups.** The `by_email` index on `users` and `by_category` index on `templates` are the idempotency keys. The function must query these indexes before each insert. If the record exists, it skips that insert entirely (no upsert, no update). This means running the seed N times produces the same result as running it once.

**Template-version linkage integrity.** After creating a `templateVersion`, the parent `template.currentVersionId` must be updated to reference it. If this update is skipped, downstream code that reads `currentVersionId` to resolve the active template version will get `undefined` and break case creation.

**Admin role correctness.** The admin user must be created with `role: "ADMIN"`, not `"USER"`. Downstream admin-gated routes check this field server-side.

**internalMutation visibility.** Because the function is an `internalMutation`, it does not appear in the client-side `api` object. This is intentional — seed is a dev tool, not a user-facing feature.

## Edge cases

**Empty database (first run).** This is the primary use case. All four records (1 user + 3 templates) and three template versions are created. The function should succeed without errors.

**Partially seeded database.** If the admin user exists but templates do not (e.g., a previous run was interrupted), the function should skip the admin user and create only the missing templates. Each insert decision is independent.

**Fully seeded database (second run).** All index lookups find existing records; no inserts are performed. The function completes successfully with no side effects.

**Production environment.** The function throws an error (e.g., `"Seed function cannot run in production"`) before any database operations. The error message should be clear enough for a developer who accidentally targets a production deployment.

**createdByUserId for templates.** Templates require a `createdByUserId` field (a reference to the `users` table). The seed function must create the admin user first (or look up the existing one), then use that user's ID as the `createdByUserId` for all three templates and as `publishedByUserId` for all three template versions.

## Non-goals

**No CLI wrapper or npm script.** The seed is invoked directly via `npx convex run seed:seed`. Adding an npm script like `npm run seed` is convenient but out of scope — developers can add it themselves.

**No production seed data.** This script is exclusively for development bootstrapping. There is no "production seed" concept in v1.

**No case or message seeding.** Only `users`, `templates`, and `templateVersions` are populated. Seeding cases, party states, or messages for demo purposes is out of scope.

**No rich template content.** The `globalGuidance` text is minimal placeholder text (a sentence or two per category). Writing production-quality coaching prompts is a separate content task.

## Test coverage

All four ACs are covered by a single unit test file: `tests/unit/seed.test.ts`. All tests use `convex-test` to run the seed mutation against an in-memory Convex test deployment — no real Convex backend is needed.

**AC 1 (admin user creation)** — `tests/unit/seed.test.ts` (unit). Call the seed function against an empty test database, then query `users` by `ADMIN_EMAIL`. Assert exactly one user exists with `role === "ADMIN"` and the expected email.

**AC 2 (default templates)** — `tests/unit/seed.test.ts` (unit). After seeding, query `templates` by each category (`workplace`, `family`, `personal`). Assert each exists, has a non-null `currentVersionId`, and that the linked `templateVersion` has `version === 1` and a non-empty `globalGuidance`.

**AC 3 (idempotency)** — `tests/unit/seed.test.ts` (unit). Call the seed function twice in succession against the same test database. Query users by email and assert count === 1. Query templates and assert exactly 3 total (not 6). Assert template versions count === 3 (not 6).

**AC 4 (production guard)** — `tests/unit/seed.test.ts` (unit). Set the environment variable to indicate production (or mock the environment check), call the seed function, and assert it throws an error. Verify the `users` and `templates` tables remain empty afterward.
