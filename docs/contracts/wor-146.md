# Contract: WOR-146 — Coach never posts periodic / agreement summaries (US-10b)

## Why this work exists

The PRD (US-10b) requires: "Coach posts periodic summaries when new points of agreement emerge." Currently, the PROGRESS classification from the Haiku classifier bypasses the NORMAL_EXCHANGE suppression gate (allowing Coach to respond), but the Coach response is a generic facilitation message — not a deliberate agreement summary. Additionally, there is no periodic/timer mechanism that evaluates the transcript for emerging agreement independent of individual messages. This ticket closes both gaps: (1) when Haiku classifies a message as PROGRESS, the Coach response is instructed to produce a summary of agreed points, and (2) a periodic cron evaluates active joint sessions and triggers summary generation via the existing `triggerType: "timer"` path.

## Files and exports

### `convex/jointChat.ts` (modify)

The existing `generateCoachResponse` action already handles PROGRESS classification by allowing the Coach to respond (it bypasses the NORMAL_EXCHANGE gate at line 558). The modification adds **summary-mode prompt assembly** when classification is PROGRESS: the `assemblePrompt` call passes `summaryMode: true`, which appends the `COACH_SUMMARY_INSTRUCTION` to the system prompt. This instructs the model to produce a structured summary of points of agreement rather than a generic facilitation response.

A new `evaluateAndSummarize` internalAction is added. It is called by the cron job. It:
1. Queries all cases with status `JOINT_ACTIVE`.
2. For each, reads the joint messages and checks whether there has been meaningful activity since the last Coach summary (at least 6 user messages since the last COACH message, providing enough transcript to summarize).
3. If the threshold is met, it inserts a placeholder message (to serve as the `messageId` arg) and schedules `generateCoachResponse` with `triggerType: "timer"`. The timer path bypasses the NORMAL_EXCHANGE gate.

The Coach response under `triggerType: "timer"` also uses `summaryMode: true` in prompt assembly, ensuring the periodic evaluation produces a summary rather than a generic response.

### `convex/crons.ts` (modify)

Adds a second cron job: `"joint session summary evaluation"` running every 10 minutes via `crons.interval`. This calls `internal.jointChat.evaluateAndSummarize` with no args (it internally queries active cases). The 10-minute interval balances responsiveness with API cost.

### `convex/lib/prompts.ts` (modify)

Adds `COACH_SUMMARY_INSTRUCTION` — a constant string instructing the model to:
- Identify and summarize points of agreement that have emerged in the conversation so far.
- Note any areas where progress is being made but agreement is not yet reached.
- Keep the summary concise (2-4 bullet points max).
- Use neutral, encouraging language.
- Never quote or paraphrase either party's private content.

The `assemblePrompt` function is extended: when `opts.summaryMode === true` and `opts.role === "COACH"`, it appends `COACH_SUMMARY_INSTRUCTION` to the system prompt after the standard coach instructions.

### `convex/lib/claudeMock.ts` (modify)

The `getMockClaudeResponse` function is extended to accept an optional second argument `summaryMode?: boolean`. When `true`, it returns a deterministic mock summary response (e.g., "Here's a summary of where you both agree: ...") instead of the generic coach mock. This enables tests to verify that summary-mode generation produces distinguishable output.

## Data dependencies

- `internal.jointChat.getCaseForCoach` — returns the case document; we check `status === "JOINT_ACTIVE"`.
- `internal.jointChat.getJointMessagesForCoach` — returns all joint messages for the case; used to count recent activity and build prompt context.
- `internal.jointChat.getPartyStatesForCoach` — returns party states with synthesis texts; used for prompt context.
- `internal.jointChat.getTemplateVersionForCoach` — returns the pinned template version for prompt assembly.
- `internal.jointChat.insertCoachStreamingMessage` — inserts the initial STREAMING row.
- Existing streaming + finalization mutations handle the response lifecycle.

## Invariants

**Privacy filter applies unchanged.** Summary responses pass through the same 8-token consecutive window privacy filter as all other Coach responses. The `COACH_SUMMARY_INSTRUCTION` reinforces the anti-quotation rule, but enforcement remains in `filterResponse`.

**JOINT_ACTIVE gate.** Both the PROGRESS-triggered path and the cron-triggered path verify `caseDoc.status === "JOINT_ACTIVE"` before generating. If the case has closed between scheduling and execution, the action returns early.

**PROGRESS bypasses NORMAL_EXCHANGE gate.** This already works — the gate at line 558 only suppresses when `classification === "NORMAL_EXCHANGE"`. No change needed for PROGRESS messages; they always get a response.

**Timer path bypasses gate.** When `triggerType === "timer"`, the existing code at line 560 trusts the caller and skips the consecutive-message count. The cron's `evaluateAndSummarize` is responsible for verifying the activity threshold before scheduling.

**No schema changes.** Summary messages are regular `jointMessages` rows with `authorType: "COACH"`, `isIntervention: false`, and no new fields. They are indistinguishable in the UI from other Coach messages (which is the desired behavior — they appear naturally in the conversation flow).

## Edge cases

**No active cases.** The cron's `evaluateAndSummarize` queries for JOINT_ACTIVE cases. If none exist, it returns immediately with no work done.

**Insufficient activity since last summary.** If fewer than 6 user messages have been posted since the last COACH message, the cron skips that case. This prevents redundant summaries when parties haven't said enough to warrant one.

**Case closes between cron schedule and execution.** The `generateCoachResponse` action checks `caseDoc.status === "JOINT_ACTIVE"` at the top and returns early if the case has since closed. No error is thrown.

**PROGRESS classification on a very short exchange.** If Haiku classifies the second message of a session as PROGRESS, the Coach will still produce a summary. This is acceptable — a brief "It sounds like you're already finding common ground on X" is appropriate even early in the conversation.

**Mock mode.** When `CLAUDE_MOCK=true`, the `evaluateAndSummarize` action uses mock classification and mock responses. `CLAUDE_MOCK_CLASSIFICATION` can be set to `"PROGRESS"` to force summary behavior in tests.

## Non-goals

- **No UI changes.** Summary messages render identically to other Coach messages. There is no special "summary card" or visual treatment in v1.
- **No new schema fields.** We do not add `isSummary` or `messageType` to the `jointMessages` table. If future work needs to distinguish summaries, that's a separate ticket.
- **No configurable frequency.** The 10-minute cron interval is hardcoded. Admin-configurable intervals are out of scope.
- **No sophisticated NLP agreement detection.** The cron relies on message count heuristics (6+ user messages since last Coach response) as the trigger; the actual "agreement detection" is delegated to the LLM via the summary prompt. The Haiku classifier's PROGRESS label handles the per-message path.

## Test coverage

All tests live in `tests/unit/joint-chat-summary.test.ts` using `convex-test`:

- **AC: "Coach posts periodic summaries when new points of agreement emerge"** — Unit test. Seeds a JOINT_ACTIVE case with 6+ user messages, runs `evaluateAndSummarize`, verifies a new COACH message is inserted with summary content (mock mode).

- **AC: "Summary generation is triggered by PROGRESS classification"** — Unit test. Seeds a case, sends a message with `CLAUDE_MOCK_CLASSIFICATION=PROGRESS`, verifies `generateCoachResponse` is scheduled and produces a response (not suppressed by the gate). Verifies the response uses summary-mode prompt assembly by checking mock output matches summary variant.

- **AC: "Summary mechanism is independent of inflammatory-content interventions"** — Unit test. Seeds a case with an INFLAMMATORY message followed by a PROGRESS message. Verifies both produce Coach responses independently — the intervention sets `isIntervention: true`, the summary does not.

- **AC: "The triggerType 'timer' path is wired"** — Unit test. Directly invokes `generateCoachResponse` with `triggerType: "timer"` and `CLAUDE_MOCK_CLASSIFICATION=NORMAL_EXCHANGE`. Verifies the gate is bypassed and a Coach message is produced (confirming the cron path works end-to-end without hitting the 5-message suppression).
