# Contract: WOR-114 — Post-create invite sharing page

## Why this work exists

After creating a case, the initiator needs a dead-simple way to share the invite link with the other party. This is the critical handoff moment in the Clarity flow — if the initiator doesn't share the link, the case never progresses beyond private coaching. The page reduces friction by providing pre-written messages for email and text, suggested language for the awkward "hey, let's use an AI mediator" conversation, and a secondary path to start private coaching immediately without waiting.

## Files and exports

### `src/routes/InviteSharingPage.tsx` (create, connected)

The main page component. It is a **connected** component — it calls `useQuery` for case data, invite URL, and party states internally. It takes no props. This matches the convention established by `NewCasePage`, `LoginPage`, and `ProfilePage` in `src/routes/`.

The component reads `otherPartyName` from React Router location state (passed by `NewCasePage` on navigation). If location state is absent (e.g., direct URL navigation or page refresh), it falls back to `"the other party"` as the display name. This is a deliberate trade-off: `otherPartyName` is not persisted to the database (it's a UX convenience field from the create form, not a data model field), so the fallback is necessary.

The component contains an internal `useCopyWithFeedback` helper (not exported) that wraps `navigator.clipboard.writeText` and manages a 2-second `copied` boolean via `setTimeout`. This helper is used by all three copy/share actions.

A redirect guard at the top of the component checks `case.isSolo` and navigates to `/cases/:caseId/private` if true. This is defense-in-depth — the primary solo-mode redirect happens in `NewCasePage`'s onSuccess handler.

### `convex/invites.ts` (modify)

Adds a new **query** `getForCase` alongside the existing `redeem` mutation and helper functions. The query looks up the active invite token for a case via the `by_case` index, filters for `status === "ACTIVE"`, and returns `{ token, url }` or `null`. It enforces that the caller is the case initiator (not just any party) — the invitee should not see the raw invite token.

### `src/App.tsx` (modify)

Adds a new route entry: `/cases/:caseId/invite` → `<ProtectedRoute><InviteSharingPage /></ProtectedRoute>`. Imports `InviteSharingPage` from `src/routes/InviteSharingPage`. The inline stub for this route does not exist yet (unlike other case sub-routes), so this is purely additive.

### `src/routes/NewCasePage.tsx` (modify)

Modifies the `onSuccess` handler in the form submission. Currently, it always navigates to `/cases/${result.caseId}/invite`. After this change, it checks whether `isSolo` is true: if so, it navigates to `/cases/${result.caseId}/private` instead. This is a one-line conditional change.

## Data dependencies

### `api.cases.get` (existing query)

Returns the full case document. This page consumes:
- `isSolo: boolean` — for the redirect guard
- `status: string` — to confirm the case is in an appropriate state (DRAFT_PRIVATE_COACHING)
- `initiatorUserId: Id<"users">` — implicitly enforced by the query's `requirePartyToCase` check

### `api.cases.partyStates` (existing query)

Returns `{ self, other }`. This page consumes:
- `self.mainTopic: string` — interpolated into the mailto body and SMS text as `[topic]`

### `api.invites.getForCase` (new query)

Returns `{ token: string; url: string } | null`. This page consumes:
- `url: string` — displayed in the monospace field, used as the link in all copy/share actions
- Returns `null` if the token has already been consumed (invitee already joined), in which case the page should show a "Link already used" state instead of the copy field.

## Invariants

**Solo mode redirect.** If `case.isSolo` is true, `InviteSharingPage` must not render any UI. It redirects to `/cases/:caseId/private` via `useEffect` + `navigate`. The primary redirect happens in `NewCasePage`'s onSuccess handler (before the user ever reaches this route), but the page-level guard handles direct navigation and bookmarks.

**Initiator-only access.** The `getForCase` query checks that the authenticated user is the case's `initiatorUserId`. If a non-initiator (or the invitee after joining) navigates to this URL, the query returns an authorization error. The page should handle this gracefully (show an error or redirect to the case detail page).

**Read-only page.** No mutations are called from this page. The clipboard API, mailto links, and navigation are all client-side operations.

**Invite URL displayed verbatim.** The monospace field shows the full URL (e.g., `https://clarity.app/invite/aBcDeFgH...`). It is not shortened, masked, or truncated. The field is read-only (no `<input>` editing).

**Copy feedback timing.** After any copy action, the triggering button shows "Copied!" for exactly 2 seconds, then reverts to its original label. If the user clicks again during the 2-second window, the timer resets. An `aria-live="polite"` region announces "Copied!" for screen readers.

**Mailto template.** The "Copy for email" option opens a `mailto:` link with:
- Subject: `"Let's work through this together — Clarity"` (or similar)
- Body: `"Hey [name] — I found this thing called Clarity. It's a private tool that helps two people work through something difficult together with an AI mediator. I thought it might help us work through the [topic]. Here's a link to join: [url]. No pressure — let me know what you think."`

Where `[name]` is `otherPartyName` (from router state, fallback "there"), `[topic]` is `self.mainTopic` (from partyStates query), and `[url]` is the invite URL.

**SMS/text template.** The "Copy for text" option copies a shorter message to clipboard: `"Hey [name], I thought this might help us work through the [topic]: [url]"` — suitable for SMS length constraints.

## Edge cases

**Loading state.** While any of the three queries (`cases.get`, `cases.partyStates`, `invites.getForCase`) are loading (`undefined`), the page shows a loading spinner (using the existing `LoadingSpinner` component from `src/components/layout/LoadingSpinner.tsx`). No partial renders.

**Token already consumed.** If `getForCase` returns `null` (token consumed — the invitee has already joined), the page should display a message like "Your invite link has been used. [name] has joined the case." with a link to proceed to the case. This is a valid state if the user returns to this URL after the invitee has already accepted.

**Missing router state.** If `location.state?.otherPartyName` is undefined (direct navigation, page refresh, bookmark), the heading and message templates use `"the other party"` as the fallback name. This is acceptable degradation.

**Clipboard API unavailable.** If `navigator.clipboard.writeText` is not available (e.g., insecure context, older browser), the copy buttons should still be rendered but gracefully handle the error — either via a try/catch that shows a brief error message, or by falling back to a select-all prompt. The mailto link works independently of the clipboard API.

## Non-goals

**No invite resend or token regeneration.** This page displays the token created during `cases.create`. If the token is somehow lost or the initiator wants a new one, that is out of scope for this ticket.

**No delivery tracking.** There is no mechanism to know whether the invitee received or opened the link. The page is a sharing tool, not a delivery confirmation system.

**No native SMS integration.** "Copy for text" copies a string to the clipboard. It does not open the device's SMS app via `sms:` URI scheme.

**No waiting-room or real-time invitee-joined notification.** The page is a one-time sharing screen. The initiator shares the link and moves on (either to private coaching or to the dashboard). There is no WebSocket/reactive indicator showing "Jordan has joined!"
