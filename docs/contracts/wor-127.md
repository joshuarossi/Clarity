# Contract: WOR-127 — Draft coach Convex module — session queries + mutations

## Why this work exists

The Draft Coach is a private, short-lived AI conversation that helps a party craft a polished message before sending it into the joint chat. This task builds the core Convex domain module (`convex/draftCoach.ts`) with session lifecycle management — creating, querying, sending messages in, finalizing to joint chat, and discarding sessions. The module must enforce authentication, party-to-case authorization, and strict privacy isolation so that Draft Coach conversations remain invisible to the other party.

## Files and exports

### `convex/draftCoach.ts` (create)

Single domain module following the project's table-by-table convention (like `convex/jointChat.ts`, `convex/privateCoaching.ts`). Contains all public queries/mutations for the draftSessions and draftMessages tables, plus a stub internalAction for the AI generation that will be implemented in T34.

**Public exports:**

- **`session`** — query. Takes `{ caseId }`. Finds the caller's ACTIVE draftSession for the given case using the `by_case_and_user` index on `draftSessions`, filtering for `status === "ACTIVE"`. If found, joins draftMessages via `by_draft_session` index. Returns `{ session, messages }` or `null`. Privacy enforcement: filters by `userId === authenticatedUser._id` — this is the critical privacy gate.

- **`startSession`** — mutation. Takes `{ caseId }`. Creates a `draftSessions` row with `status: "ACTIVE"`, `createdAt: Date.now()`. Before creating, checks that no ACTIVE session already exists for this user+case (throws CONFLICT if one does). Schedules `internal.draftCoach.generateResponse` via `ctx.scheduler.runAfter(0, ...)`. Returns the new session ID.

- **`sendMessage`** — mutation. Takes `{ sessionId, content }`. Reads the session, verifies ownership (session.userId === caller), verifies session.status === "ACTIVE". Inserts a `draftMessages` row with `role: "USER"`, `status: "COMPLETE"`, `createdAt: Date.now()`. Schedules `internal.draftCoach.generateResponse`. Returns the new message ID.

- **`sendFinalDraft`** — mutation. Takes `{ sessionId }`. Reads the session, verifies ownership and ACTIVE status. Asserts `session.finalDraft` is non-null (throws ConvexError with code `CONFLICT` if null). Directly inserts a `jointMessages` row with `authorType: "USER"`, `authorUserId: session.userId`, `content: session.finalDraft`, `status: "COMPLETE"`. Schedules `internal.jointChat.generateCoachResponse` for the coach to respond. Patches the session to `status: "SENT"`, `completedAt: Date.now()`. Returns the new jointMessages ID.

  **Why direct insert instead of calling jointChat.sendUserMessage:** In Convex, mutations cannot call other mutations via `ctx.runMutation()` — that capability is reserved for actions. Since sendFinalDraft is a mutation (for transactional atomicity — the session status update and joint message insert must be atomic), it directly inserts into the `jointMessages` table and schedules the coach action. This is the same pattern used within `jointChat.sendUserMessage` itself.

- **`discardSession`** — mutation. Takes `{ sessionId }`. Reads the session, verifies ownership and ACTIVE status. Patches to `status: "DISCARDED"`, `completedAt: Date.now()`. No joint chat interaction.

- **`generateResponse`** — internalAction stub. Takes `{ sessionId, userId }`. This is the entry point for the Draft Coach AI action that T34 will implement. For this ticket, it should be defined as a no-op internalAction so that `internal.draftCoach.generateResponse` resolves as a valid function reference for the scheduler calls in startSession and sendMessage.

**Auth pattern (consistent with jointChat.ts and privateCoaching.ts):**

```typescript
const user = await requireAuth(ctx);
const caseDoc = await requirePartyToCase(ctx, caseId, user._id);
```

Both imported from `./lib/auth`. Every public function runs both checks before any DB operations.

## Data dependencies

**Tables read:**

- `draftSessions` — via `by_case_and_user` index for session lookup. Fields consumed: `_id`, `caseId`, `userId`, `status`, `finalDraft`, `createdAt`, `completedAt`.
- `draftMessages` — via `by_draft_session` index for message listing. Fields consumed: `_id`, `draftSessionId`, `role`, `content`, `status`, `createdAt`.
- `cases` — via `ctx.db.get(caseId)` through `requirePartyToCase`. Fields consumed: `status`, `initiatorUserId`, `inviteeUserId`.

**Tables written:**

- `draftSessions` — insert (startSession), patch (sendFinalDraft, discardSession).
- `draftMessages` — insert (sendMessage).
- `jointMessages` — insert (sendFinalDraft only). Fields written: `caseId`, `authorType: "USER"`, `authorUserId`, `content` (from session.finalDraft), `status: "COMPLETE"`, `createdAt`.

**Scheduled functions:**

- `internal.draftCoach.generateResponse` — scheduled by startSession and sendMessage.
- `internal.jointChat.generateCoachResponse` — scheduled by sendFinalDraft (the joint chat Coach should respond to the newly posted draft message).

## Invariants

**Privacy: session query enforces userId match.** The session query filters draftSessions by `caseId` + `userId` (the authenticated caller). This means User A can never see User B's draft sessions or messages, even if both are parties on the same case. This is the Draft Coach's core privacy guarantee per US-12 and NFR-PRIVACY.

**Auth gate on every public function.** `requireAuth` resolves the user from the session. `requirePartyToCase` verifies the user is initiator or invitee on the case. Both throw structured ConvexErrors (`UNAUTHENTICATED` / `FORBIDDEN`) before any data access. This matches the pattern in `convex/jointChat.ts` and `convex/privateCoaching.ts`.

**Session uniqueness.** `startSession` queries for an existing ACTIVE session for the user+case before creating one. If found, it throws CONFLICT. This prevents orphaned sessions and ensures `session` query returns at most one result.

**Final draft guard.** `sendFinalDraft` reads `session.finalDraft` and throws ConvexError with `{ code: "CONFLICT" }` if it's null or undefined. The draft must have been set by the AI action (T34) before the user can send it.

**Atomic joint chat bridge.** `sendFinalDraft` inserts into `jointMessages` and patches `draftSessions` in a single mutation transaction. If either fails, both roll back. This prevents a state where the draft is marked SENT but no joint message exists, or vice versa.

**No auto-send.** Only `sendFinalDraft` touches the `jointMessages` table. `startSession`, `sendMessage`, and `discardSession` never insert into `jointMessages`.

## Edge cases

**No active session.** `session` query returns `null`. This is the normal state before a user starts drafting — the UI uses this to show the "Draft with Coach" button.

**Session already exists.** `startSession` throws CONFLICT if the user already has an ACTIVE session for the case. The UI should call `session` first and route to the existing session.

**sendFinalDraft with null finalDraft.** Throws ConvexError with code `CONFLICT`. This happens if the user tries to send before the AI has produced a draft. The UI should disable the Send button until `session.finalDraft` is populated.

**sendMessage / sendFinalDraft / discardSession on non-ACTIVE session.** Each of these mutations reads the session and verifies `status === "ACTIVE"`. If the session is already SENT or DISCARDED, they throw CONFLICT. This prevents double-sends and post-discard mutations.

**Ownership mismatch.** `sendMessage`, `sendFinalDraft`, and `discardSession` all verify that `session.userId === authenticatedUser._id`. If a party somehow obtains another party's session ID, they get a FORBIDDEN error.

## Non-goals

**Draft Coach AI action implementation.** The `generateResponse` internalAction is a stub in this ticket. T34 will implement the full Haiku gate + Sonnet generation flow for the Draft Coach, following the same pattern as `jointChat.generateCoachResponse`.

**Draft Coach UI / side panel.** The frontend components for the Draft Coach side panel are a separate ticket. This ticket is backend-only.

**Solo mode viewAsRole.** The existing jointChat and privateCoaching modules support a `viewAsRole` arg for solo mode. This ticket does not add viewAsRole support to draftCoach; it can be added in a follow-up if needed for solo testing of the Draft Coach flow.

**Case status validation.** The implementation notes don't specify that draftCoach functions should check that the case is in `JOINT_ACTIVE` status. However, since Draft Coach is used during the joint chat phase, sendFinalDraft implicitly requires JOINT_ACTIVE because it schedules `generateCoachResponse` which validates this. The other mutations (startSession, sendMessage, discardSession) do not gate on case status — the Draft Coach can be started even if the case transitions away from JOINT_ACTIVE (the final draft just won't be sendable via sendFinalDraft if the coach response action fails).

## Test coverage

All ACs are covered by **unit tests** in `tests/unit/draft-coach.test.ts`, using the `convex-test` framework with the project schema — the same pattern as `tests/unit/joint-chat.test.ts`.

- **AC 1 (session query):** Test that `session` returns `{ session, messages }` for the owning user with an ACTIVE session + messages. Test that it returns `null` for a different user on the same case. Test that it returns `null` when no ACTIVE session exists. Test that it excludes SENT/DISCARDED sessions.

- **AC 2 (startSession):** Test that `startSession` inserts a draftSessions row with `status: "ACTIVE"` and `createdAt`. Assert that `internal.draftCoach.generateResponse` was scheduled via `_scheduled_functions`. Test that calling startSession twice for the same user+case throws CONFLICT.

- **AC 3 (sendMessage):** Test that `sendMessage` inserts a draftMessages row with `role: "USER"`, `status: "COMPLETE"`. Assert AI action scheduled. Test that it throws CONFLICT if session is not ACTIVE.

- **AC 4 (sendFinalDraft — happy path):** Create a session with `finalDraft` populated, call `sendFinalDraft`. Assert session status becomes `SENT` with `completedAt`. Assert a `jointMessages` row was created with the draft content and `authorType: "USER"`. Assert `generateCoachResponse` was scheduled.

- **AC 5 (sendFinalDraft — conflict):** Create a session with `finalDraft: undefined`, call `sendFinalDraft`, assert ConvexError with code `CONFLICT`.

- **AC 6 (discardSession):** Create active session, call `discardSession`. Assert `status: "DISCARDED"`, `completedAt` set. Assert no `jointMessages` rows created.

- **AC 7 (privacy):** Create session as User A, query `session` as User B (also a party on the case), assert `null` returned. This directly tests the userId filter.

- **AC 8 (auth):** Call each public function without auth — assert `UNAUTHENTICATED`. Call as a user not party to the case — assert `FORBIDDEN`.

All tests are unit-level using `convex-test` because the privacy and auth enforcement happens in the Convex function layer, which `convex-test` exercises fully (it runs the actual query/mutation handlers against an in-memory DB with the real schema). No e2e tests are needed for this backend-only ticket.
