# Contract: WOR-124 — Joint chat Convex module — queries + mutations including closure

## Why this work exists

The joint chat is the core real-time surface where both parties interact with each other and the AI Coach mediator. This ticket provides the Convex backend — reactive queries for the live message feed and per-party synthesis, the send mutation (routing through Coach response scheduling), and the three closure mutations (propose, confirm, unilateral). Without this module, no joint-session UI can function.

## Files and exports

### `convex/jointChat.ts` (modify)

This file already exports `mySynthesis` (query) and `enterSession` (mutation). This ticket adds six new exports to the same file, following the project's convention of a single module file per domain (see `convex/cases.ts`, `convex/privateCoaching.ts`).

**New exports:**

- **`messages`** — Reactive query. Returns all `jointMessages` rows for the given `caseId`, sorted ascending by `createdAt`. Enforces auth + party-to-case. The `viewAsRole` arg is passed for solo-mode consistency but does not change the query's behavior (joint messages are visible to both parties).

- **`sendUserMessage`** — Mutation. Validates case is `JOINT_ACTIVE` (throws CONFLICT otherwise). Inserts a `jointMessages` row with `{ caseId, authorType: "USER", authorUserId: caller._id, content, status: "COMPLETE", createdAt: Date.now() }`. Then schedules the `jointChat/generateCoachResponse` internal action (the action itself is out of scope — this ticket only schedules it). Returns the new message's `Id<"jointMessages">`.

- **`proposeClosure`** — Mutation. Sets the caller's `partyStates.closureProposed = true` and patches `cases.closureSummary` with the provided `summary` string. Case must be `JOINT_ACTIVE`.

- **`confirmClosure`** — Mutation. Reads both party states. If the OTHER party has `closureProposed === true`, sets both parties' `closureProposed = true` and `closureConfirmed = true`, then calls `validateTransition(caseDoc.status, "RESOLVE", { partyStates })` to get the new status (`CLOSED_RESOLVED`). Patches the case with `{ status: newStatus, closedAt: Date.now(), updatedAt: Date.now() }`. If the other party has NOT proposed, throws CONFLICT.

- **`unilateralClose`** — Mutation. Calls `validateTransition(caseDoc.status, "CLOSE_UNRESOLVED")` and patches the case with `{ status: newStatus, closedAt: Date.now(), updatedAt: Date.now() }`.

- **`rejectClosure`** — Mutation. Finds the OTHER party's `partyStates` row and clears `closureProposed` to `false` (or `undefined`). Does NOT transition case status.

All functions follow the established pattern: `requireAuth(ctx)` → `requirePartyToCase(ctx, caseId, user._id)` → business logic. Solo-mode support uses `viewAsRole` to resolve which party state to act on, matching the pattern in the existing `mySynthesis` query.

## Invariants

**Auth + party-to-case:** Every function calls `requireAuth` then `requirePartyToCase`. Unauthenticated callers get `{ code: "UNAUTHENTICATED", httpStatus: 401 }`. Non-party callers get `{ code: "FORBIDDEN", httpStatus: 403 }`.

**JOINT_ACTIVE guard:** `sendUserMessage`, `proposeClosure`, `confirmClosure`, and `unilateralClose` all require the case to be in `JOINT_ACTIVE` status. If not, they throw `{ code: "CONFLICT", httpStatus: 409 }`. For `confirmClosure` and `unilateralClose` this is enforced by `validateTransition` itself. For `sendUserMessage` and `proposeClosure`, an explicit check before proceeding.

**State machine usage:** All status transitions go through `validateTransition` from `convex/lib/stateMachine.ts`. `confirmClosure` uses the `RESOLVE` transition (which internally validates that both parties have `closureProposed=true && closureConfirmed=true`). `unilateralClose` uses `CLOSE_UNRESOLVED`.

**Message ordering:** The `messages` query returns rows sorted by `createdAt` ascending. It uses the `by_case` index on `jointMessages`.

**Privacy boundary:** `mySynthesis` only returns the calling party's synthesis. Joint messages are shared — both parties see all messages in the case.

## Edge cases

**Loading state:** `messages` returns an empty array if no messages exist yet. `mySynthesis` returns `null` if synthesis hasn't been generated.

**Calling confirmClosure when no one has proposed:** Throws CONFLICT. The confirming party cannot confirm unless the other party's `closureProposed === true`.

**Calling proposeClosure twice:** Idempotent — sets `closureProposed = true` again and updates `closureSummary` to the new value. No error.

**rejectClosure when no proposal exists:** No-op — sets `closureProposed = false` on the other party even if it was already falsy. No error thrown.

**Solo mode:** All mutations accept `viewAsRole` to disambiguate which party the caller is acting as. In solo mode, `requirePartyToCase` succeeds because the user is both initiator and invitee. The `viewAsRole` param is used to look up the correct `partyStates` row.

**sendUserMessage when action doesn't exist yet:** The `ctx.scheduler.runAfter` call references the internal action. If the action module doesn't exist at deploy time, Convex will fail the deploy. The implementation should use `internal.jointChat.generateCoachResponse` as the reference. If T31 hasn't been implemented yet, a placeholder no-op internal action should be defined in the same file.

## Non-goals

- **generateCoachResponse action logic** — T31 implements the actual AI call. This ticket only schedules it.
- **Draft Coach flow** — Separate domain (`draftSessions` / `draftMessages`), separate ticket.
- **UI components for joint chat** — This is backend-only.
- **24h cooling-off for unilateral close** — v1 simplification; immediate transition.
- **Coach opening message** — The Coach's first message in the joint session is triggered by `enterSession` or by the action; not part of this ticket's send/query surface.
