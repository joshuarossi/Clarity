# Contract: WOR-167 — Solo-mode create-case writes 2 partyStates rows for same (caseId,userId) → every by_case_and_user .unique() consumer 500s

## Why this work exists

Solo mode (US-13) intentionally creates two `partyStates` rows for the same user — one as INITIATOR, one as INVITEE — so the user can play both sides of a mediation. The `by_case_and_user` index covers only `[caseId, userId]`, so when any consumer calls `.unique()` on that index for a solo case, Convex throws because two rows match instead of one. This crashes the dashboard and at least 7 other query/mutation sites. The fix is to make every consumer solo-aware by disambiguating on `role`.

## Files and exports

### `convex/schema.ts` (modify)

Add a new index `by_case_user_role` on the `partyStates` table with fields `["caseId", "userId", "role"]`. This index allows `.unique()` to succeed when all three fields are specified, which is the correct lookup for solo cases. The existing `by_case_and_user` index is **not removed** — it is still used by `.collect()` callers and remains correct for non-solo lookups.

### `convex/cases.ts` (modify)

**`listForDashboard`** (line ~256): The `.unique()` call in the `BOTH_PRIVATE_COACHING` branch must be replaced. For solo cases, the caller's role can be determined from context (the dashboard knows if it's a solo case and can use the INITIATOR role as the canonical "caller" party state, since the purpose is just to check `privateCoachingCompletedAt`). Alternatively, use `.collect()` + `.find()` with a role filter. The key requirement: this must not 500 on solo cases.

**`updateMyForm`** (line ~318): Add an optional `viewAsRole` argument matching the pattern used by `markComplete` and `proposeClosure`. When `viewAsRole` is provided and the case is solo, use `.collect()` + `.find(role)`. When not provided (non-solo), `.unique()` is safe.

### `convex/privateCoaching.ts` (modify)

**`markComplete`** (line ~114): The `else` branch (non-solo or solo-without-viewAsRole) calls `.unique()`. This branch is reachable for solo cases when `viewAsRole` is not passed. Fix: always use the solo-aware `.collect()` + `.find()` pattern when the case is solo, regardless of whether `viewAsRole` was provided — default to INITIATOR if no role specified on a solo case, or throw a clear error.

**`getPartyState`** (internalQuery, line ~290): Add an optional `role` argument. When provided, use `by_case_user_role` index with `.unique()`. When omitted, fall back to `by_case_and_user` with `.unique()` (safe for non-solo callers). All internal callers of `getPartyState` that operate on solo cases must pass the role.

### `convex/jointChat.ts` (modify)

**`mySynthesis`** (line ~42): The `else` branch calls `.unique()`. Same fix pattern as `markComplete`: when the case is solo, use `.collect()` + `.find()`.

**`proposeClosure`** (line ~175): Same pattern — the `else` branch calls `.unique()`.

### `convex/draftCoach.ts` (modify)

**`getPartyStateForDraftCoach`** (internalQuery, line ~228): Add an optional `role` argument. The callers of this internal query know which party role they are operating on (the draft coach always operates in the context of a specific party). Use `by_case_user_role` when role is provided.

## Data dependencies

All affected functions query the `partyStates` table via the `by_case_and_user` index. The fix changes how they query (adding role disambiguation) but does not change what data they return or consume. The fields consumed are: `role`, `privateCoachingCompletedAt`, `synthesisText`, `closureProposed`, `closureConfirmed`, `mainTopic`, `description`, `desiredOutcome`, `formCompletedAt`.

## Invariants

**Solo cases always have exactly 2 partyStates rows per (caseId, userId).** This is by design. The fix must not attempt to deduplicate or prevent this. The rows differ only in `role` (INITIATOR vs INVITEE) and their form/coaching/synthesis fields.

**Every `.unique()` call on `by_case_and_user` must be audited.** There are exactly 7 call sites across 4 files. Each must be changed to either: (a) use the new `by_case_user_role` index when role is known, or (b) use `.collect()` + array `.find()` when the role needs to be resolved from context (e.g., `viewAsRole` arg).

**Non-solo cases are unaffected.** For non-solo cases, there is exactly 1 partyStates row per (caseId, userId), so both the old `.unique()` pattern and the new solo-aware pattern return the same result.

**The `by_case_and_user` index is retained.** It is still correct and useful. The new `by_case_user_role` index is additive.

## Edge cases

**Solo case with no `viewAsRole` provided.** Some UI code paths may not pass `viewAsRole` (e.g., `updateMyForm` currently has no such arg). The fix adds `viewAsRole` where needed. For any remaining edge case where role is unknown on a solo case, the implementation should default to INITIATOR (the canonical "self" role) rather than throwing.

**Internal queries called from actions.** `getPartyState` and `getPartyStateForDraftCoach` are `internalQuery` functions called from actions that process AI responses. These callers must be updated to pass the role. The role is available in the action context because the action knows which party it is generating a response for.

**Existing data.** No schema migration is needed. The new index `by_case_user_role` will be built automatically by Convex when the schema is pushed. Existing rows already have all three fields populated.

## Non-goals

**No synthetic second userId for solo mode.** The two-rows-same-userId design is intentional and matches US-13's requirement that "AI responses behave as if each party were separate." Changing to a synthetic user would require changes across the entire codebase.

**No unique constraint on the index.** Convex does not support unique index constraints at the schema level. `.unique()` is a query-time assertion. The fix is to ensure the query always narrows to exactly 1 row before calling `.unique()`.

**No data migration or cleanup.** Existing solo cases with 2 rows per (caseId, userId) are correct. No backfill needed.

## Test coverage

**`tests/unit/cases.test.ts`** — Extend the existing solo case creation tests to verify that `listForDashboard` and `updateMyForm` do not throw when called on a solo case. This file already has solo case tests (line ~412); the new tests add consumer-side verification.

**`tests/unit/solo-party-state-lookup.test.ts`** — New test file covering the cross-cutting concern: for each of the 7 affected consumers, verify that calling the function on a solo case (where 2 partyStates rows exist for the same userId) returns a valid result without throwing. Also verify that non-solo cases continue to work identically. This groups the tests by concern (solo-mode partyState disambiguation) rather than by ticket, which matches the project's convention of domain-grouped unit tests.

Both ACs are unit-testable because the bug is in the query logic (`.unique()` on an index that returns 2 rows), which can be fully reproduced with `convex-test` by inserting 2 partyStates rows and calling each affected function.
