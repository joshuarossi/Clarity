# Contract: WOR-115 — InviteAcceptPage — invite acceptance flow

## Why this work exists

The invite acceptance page is the front door for the second party in every Clarity case. When someone receives an invite link, this page must explain what Clarity is (if logged out), show just enough context to make an informed accept/decline decision (if logged in), and handle edge cases like consumed tokens. Getting this right is critical to the invite redemption funnel (target: >50% per PRD §7.2). A confusing or trust-eroding first impression will tank conversion.

## Files and exports

### `src/routes/InviteAcceptPage.tsx` (create, connected)

The main page component. It is a **connected** component — it reads auth state via `useConvexAuth()`, fetches invite preview data via `useQuery(api.invites.getByToken)`, and calls mutations internally. It takes no props, matching the convention of `LoginPage`, `NewCasePage`, `ProfilePage`, and `InviteSharingPage` in `src/routes/`.

The component renders one of three states based on auth status and token status:

1. **Logged-out + ACTIVE token**: Centered card with heading "[Name] has invited you to work through something together", body text explaining Clarity (exact copy from DesignDoc §4.3: "Clarity is a private mediation tool. You'll each talk with an AI coach privately before having a facilitated conversation together."), and a "Sign in to continue" button that navigates to `/login?redirect=/invite/:token`.

2. **Logged-in + ACTIVE token**: Same framing card plus the initiator's `mainTopic` (displayed as written — one sentence) and `category`. A privacy callout: "[Name] wrote this in the shared summary. You'll have your own private space to share your perspective." Two CTAs: "Accept invitation" (primary) and "Decline" (secondary/danger).

3. **CONSUMED or invalid token**: Error message explaining the invite has already been used or is invalid. Links/buttons: "Sign in" (if logged out) or "Go to dashboard" (if logged in).

The component does **not** wrap in `<ProtectedRoute>` because it must render for unauthenticated users in state 1.

### `convex/invites.ts` (modify)

Two new exports alongside existing ones:

**`getByToken` query** — Takes `{ token: string }`. Does NOT require authentication (critical: logged-out users need the initiator's name for the heading). Looks up the token via the `by_token` index. Returns:
- For ACTIVE tokens: `{ status: "ACTIVE", initiatorName, mainTopic, category, caseId }` — fetched by joining through `inviteTokens.caseId` → `cases` → `cases.initiatorUserId` → `users.displayName`, and `partyStates` (initiator's row) for `mainTopic`. The `category` comes from `cases.category`.
- For CONSUMED tokens: `{ status: "CONSUMED" }`.
- For invalid/missing tokens: `null`.

This query is the privacy boundary. It returns `mainTopic` and `category` (shared context the initiator chose to write) but NEVER `description` or `desiredOutcome` (the initiator's private reflections).

**`decline` mutation** — Takes `{ token: string }`. Requires authentication. Validates the token is ACTIVE. Loads the case. Prevents self-decline (initiator cannot decline their own invite — mirrors the self-redeem guard in `redeem`). Calls `validateTransition(case.status, "DECLINE_INVITE")` to enforce the state machine. Patches the case with `{ status: "CLOSED_ABANDONED", closedAt: Date.now(), updatedAt: Date.now() }`. Marks the token as CONSUMED with `{ status: "CONSUMED", consumedAt, consumedByUserId }`. Returns `null`.

### `src/App.tsx` (modify)

Replaces the inline `InvitePage` stub with an import of `InviteAcceptPage` from `src/routes/InviteAcceptPage`. The route `/invite/:token` continues to point to `<InviteAcceptPage />` — crucially NOT wrapped in `<ProtectedRoute>`, as it already isn't in the current stub.

## Data dependencies

### `api.invites.getByToken` (new query)

Returns invite preview data without requiring auth. Fields consumed by the UI:
- `initiatorName: string` — used in heading ("[Name] has invited you...") and privacy callout ("[Name] wrote this...")
- `mainTopic: string` — displayed in the logged-in view as the case topic
- `category: string` — displayed alongside mainTopic
- `caseId: Id<"cases">` — used for post-accept navigation to `/cases/:caseId/private`
- `status: "ACTIVE" | "CONSUMED"` — determines which view to render

The query joins across three tables: `inviteTokens` → `cases` → `users` (for initiator name) and `inviteTokens` → `cases` → `partyStates` (for initiator's mainTopic, filtered by `role === "INITIATOR"`).

### `api.invites.redeem` (existing mutation)

Called when the user clicks "Accept invitation". Returns `{ caseId }`. After success, the page navigates to `/cases/:caseId/private`. No changes to this mutation.

### `api.invites.decline` (new mutation)

Called when the user clicks "Decline". Returns `null`. After success, the page navigates to `/dashboard`. The mutation handles the state transition server-side.

## Invariants

**No ProtectedRoute wrapper.** The `/invite/:token` route must render for unauthenticated users. The `InviteAcceptPage` component handles auth state internally via `useConvexAuth()`.

**Privacy boundary on getByToken.** The query returns only `mainTopic`, `category`, and `initiatorName`. It must never return `description`, `desiredOutcome`, or any content from privateMessages. This is the invitee's first contact with the case — they should see only what the initiator intended to share (the topic framing), not the initiator's private reflections.

**Token persistence through auth.** The token lives in the URL path (`/invite/:token`). The "Sign in to continue" button navigates to `/login?redirect=/invite/:token`. After auth completes, the LoginPage redirects back to `/invite/:token`. This works because LoginPage already supports the `redirect` search param (via `safeRedirect`). No localStorage is needed — the URL is the persistence mechanism.

**Self-action prevention.** The `redeem` mutation already prevents self-redeem (initiator cannot accept their own invite). The `decline` mutation must also prevent self-decline. Both check `caseDoc.initiatorUserId === user._id` and throw `conflict()`.

**State machine enforcement.** The `decline` mutation calls `validateTransition(case.status, "DECLINE_INVITE")`, which only allows this transition from `DRAFT_PRIVATE_COACHING`. If the case is in any other state, the mutation throws a CONFLICT error.

**Post-accept navigation.** After a successful `redeem`, the page navigates to `/cases/:caseId/private`. This is where the invitee will fill their own case form (mainTopic, description, desiredOutcome) — that form is built by a separate ticket. The contract for this ticket only specifies the navigation target.

**Post-decline navigation.** After a successful `decline`, the page navigates to `/dashboard`.

**Privacy callout text.** The exact callout text pattern is: `"[Name] wrote this in the shared summary. You'll have your own private space to share your perspective."` where `[Name]` is the initiator's displayName from `getByToken`. This callout is rendered only in the logged-in unredeemed view, below the mainTopic/category display.

## Edge cases

**Loading state.** While `getByToken` is loading (`undefined`), the page shows `LoadingSpinner` (from `src/components/layout/LoadingSpinner.tsx`). While auth state is loading (`isLoading === true`), same behavior. No partial renders — the page shows either the full logged-out view, the full logged-in view, or the loading spinner.

**Invalid token.** If `getByToken` returns `null` (token string doesn't match any record), the page shows an error message: "This invite link is not valid." with a "Go to dashboard" link (if logged in) or "Sign in" button (if logged out). This is distinct from the consumed-token state.

**Consumed token.** If `getByToken` returns `{ status: "CONSUMED" }`, the page shows: "This invite has already been accepted." with navigation options: "Sign in" (logged out) or "Go to dashboard" (logged in).

**Redeem error handling.** If `redeem` throws (e.g., race condition where another session consumed the token, or self-redeem attempt), the error is displayed inline using the existing `handleConvexError` utility from `src/lib/errorHandler.ts`. The accept button should show a loading/disabled state while the mutation is in flight.

**Decline error handling.** Same pattern as redeem — inline error display via `handleConvexError`.

**Auth state transitions.** If the user is viewing the logged-out view and logs in via another tab, `useConvexAuth()` will reactively update `isAuthenticated` to true. The component should transition to the logged-in view without a page reload. This happens naturally with Convex's reactive auth.

## Non-goals

**No invitee form implementation.** The form the invitee fills after accepting (mainTopic, description, desiredOutcome) is part of the private coaching flow, handled by a separate ticket. This ticket only navigates to `/cases/:caseId/private` after accept.

**No push/email notification for decline.** The AC says "Decline... and notifies initiator." In v1, this notification is the case status change to CLOSED_ABANDONED appearing on the initiator's dashboard. Push notifications and email are P2 features (PRD US-16). The decline mutation updates the case status; the dashboard reactively shows the change.

**No confirmation dialog before decline.** The test-gen brief mentions asserting a "confirmation prompt" on decline, but the actual AC does not specify one. If the test author needs a confirmation step, the contract does not require it — the decline button directly calls the mutation. (If this is contentious, it's flagged as an open question below.)

**No token regeneration.** If a token is consumed or invalid, the page shows an error. There is no way to generate a new invite from this page.

**No real-time consumed-while-viewing detection.** If the invitee has the page open and the token is consumed by another session, the page does not reactively update. The user will see an error only when they attempt to accept/decline.

## Open questions

**Decline confirmation dialog.** The test-gen brief (AC 6) says to "assert confirmation prompt" on decline, but the acceptance criteria themselves do not mention a confirmation step. This contract does not require one. If a confirmation dialog is desired, the implementation author should add a simple "Are you sure?" dialog before calling the decline mutation, but this is not contracted.
