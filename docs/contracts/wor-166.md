# Contract: WOR-166 — Build the e2e test harness: test-only ConvexCredentials sign-in + implement e2e/helpers.ts

## Why this work exists

The project has ~147 Playwright tests across 20 spec files, but they all depend on `e2e/helpers.ts` functions that are currently stubs returning `crypto.randomUUID()`. No test can actually create users, sign in, or seed cases in the Convex backend. This ticket wires those helpers to real Convex mutations and adds a test-only `ConvexCredentials` provider so that `loginAs` can programmatically authenticate a Playwright page without automating the magic-link email flow.

## Files and exports

### `convex/auth.ts` (modify)

The existing file configures `convexAuth()` with `Email` (magic-link) and `Google` providers. This ticket adds a conditional third provider: `Credentials` from `@convex-dev/auth/providers/ConvexCredentials`, gated behind `process.env.ENABLE_TEST_AUTH === "true"`. The Credentials provider's `authorize` callback looks up (or creates) a user by email and returns `{ userId }` — no password is required because this is strictly for test automation. The gate ensures this provider never exists in production deployments.

The `Credentials` provider must use `id: "test-credentials"` so that `loginAs` can target it explicitly in the sign-in request (Convex Auth's `signIn` action accepts a `provider` field matching the provider's `id`).

### `convex/testSupport.ts` (create)

A new file containing `internalMutation` functions for test data seeding. These are internal-only (not exposed via `api.*`, only via `internal.*`), so they cannot be called from client code. The helpers in `e2e/helpers.ts` call them via `ConvexHttpClient` using the deployment's admin key.

Functions:

- **`createTestUser`**: Inserts a user into the `users` table with the given email, optional role (default `"USER"`), and auto-derived `displayName`. Uses `getUserByEmail` from `convex/lib/auth.ts` for idempotency — if the email already exists, returns the existing user's ID.

- **`createTestCase`**: Creates a fully consistent case at the specified status. This means: (1) look up or create the initiator user, (2) look up or create the first available `templateVersion` (or use the seed default for the given category), (3) insert a `cases` row, (4) insert `partyStates` for the initiator (and invitee if provided), (5) optionally insert `privateMessages` and `synthesisText` if the target status implies those phases are complete. The `status` parameter defaults to `"DRAFT_PRIVATE_COACHING"`.

- **`transitionCaseStatus`**: Force-patches `case.status` to the given value. No state-machine validation — this is a test shortcut for setting up preconditions.

- **`createTestCaseWithInvite`**: Calls `createTestCase` internally, then inserts an `inviteTokens` row with status `"ACTIVE"` and a randomly generated 32-character token string. Returns `{ caseId, token, initiatorEmail }`.

- **`consumeTestInvite`**: Looks up the invite token, patches it to `"CONSUMED"`, sets `consumedByUserId` to the invitee user, and patches the case's `inviteeUserId`.

### `e2e/helpers.ts` (modify)

Replace all stub implementations with real calls to the Convex backend. The helpers use `ConvexHttpClient` from the `convex/browser` package (or equivalent Node-compatible client), configured with:
- `CONVEX_URL` (the dev deployment URL, e.g. from `process.env.VITE_CONVEX_URL`)
- Admin auth token for calling internal mutations (via `process.env.CONVEX_ADMIN_KEY`)

**`loginAs(page, email)`** is the most complex helper. It must:
1. Ensure the user exists (call `createTestUser` if needed).
2. Navigate the page or use `page.evaluate()` / `page.request` to call the Convex Auth sign-in endpoint with `provider: "test-credentials"` and the user's email.
3. After sign-in, the page's Convex client must have a valid session token so that subsequent navigation to authenticated routes (e.g., `/dashboard`) succeeds without redirect to `/login`.

The exact mechanism: use `page.goto(baseURL)` to load the app, then `page.evaluate()` to call `signIn("test-credentials", { email })` via the Convex Auth React integration, or use `page.request.post()` to hit the Convex Auth HTTP endpoint directly and inject the resulting session token into the page's storage.

### `e2e/fixtures.ts` (modify)

No structural changes needed — the file already imports `createTestUser` and `loginAs` from `./helpers`. Once those are wired, the `authenticatedPage`, `pageA`, and `pageB` fixtures will work. The only potential change is ensuring the fixture waits for auth to settle (e.g., waiting for a navigation or selector that confirms the user is logged in) after `loginAs` returns.

## Data dependencies

### Internal mutations (via `internal.testSupport.*`)

All test support mutations write directly to Convex tables. They consume:
- `users` table (insert/query by email index)
- `cases` table (insert with status, templateVersionId, initiatorUserId)
- `partyStates` table (insert per party)
- `privateMessages` table (optional insert for pre-seeded messages)
- `inviteTokens` table (insert/patch for invite flow)
- `templateVersions` table (query for the first available version in category)
- `templates` table (query by category index to find template + version)

The `createTestCase` mutation depends on the seed data (WOR-107's `seed.ts`) having been run so that at least one `templateVersion` exists per category. If no template exists for the requested category, it should fall back to any available template version.

### Convex Auth HTTP endpoint

`loginAs` uses the Convex Auth HTTP routes registered in `convex/http.ts` (specifically the sign-in endpoint). The Credentials provider receives `{ email }` in the credentials object and returns a session token.

## Invariants

**Production safety**: The `Credentials` provider is wrapped in a runtime check: `if (process.env.ENABLE_TEST_AUTH === "true")`. In production, this env var is never set. If someone sets it in production, the only exposure is email-only sign-in (no password) — but since the test support mutations are `internalMutation`, they still require the admin key and are not callable from the client.

**Session validity**: After `loginAs` completes, `getAuthUserId(ctx)` in any Convex query/mutation called by that page must resolve to the correct user's `_id`. This is the core contract — if this doesn't work, no test works.

**Idempotent user creation**: `createTestUser` must be idempotent on email. Calling it twice with the same email returns the same userId. This is enforced by delegating to `getUserByEmail` which already has this behavior.

**Consistent case state**: `createTestCase` at status `"READY_FOR_JOINT"` must also set `privateCoachingCompletedAt` on both party states and generate synthesis text, because the UI and queries for that status expect those fields to be populated.

**Test mutations are internal-only**: `convex/testSupport.ts` must use `internalMutation` (not `mutation`). This ensures they appear under `internal.testSupport.*` and are not callable from the public Convex client.

## Edge cases

**No seed data**: If the `seed` mutation hasn't been run, `createTestCase` will fail when looking up a `templateVersion`. The helper should throw a clear error: "No template versions found — run `npx convex run seed:seed` first."

**Concurrent test runs**: Multiple Playwright workers may call `createTestUser` with overlapping emails. Because `getUserByEmail` is idempotent (upsert), this is safe — but `createTestCase` creating cases with the same initiator email concurrently is fine because each case is a separate document.

**loginAs on a page that already has a session**: If the page already has a Convex Auth session (e.g., from a previous `loginAs` call), calling `loginAs` again should sign out the current session first and sign in as the new user. Alternatively, fixtures always start with a fresh `BrowserContext`, making this a non-issue in practice.

**CONVEX_ADMIN_KEY not set**: The helpers that call internal mutations need an admin key. If `CONVEX_ADMIN_KEY` is not set, helpers should throw immediately with a clear error message rather than making a request that will 401.

**ConvexCredentials provider not available**: If tests run against a deployment where `ENABLE_TEST_AUTH` is not set, the `loginAs` sign-in call will fail because the `test-credentials` provider doesn't exist. The error should be caught and re-thrown with guidance: "Set ENABLE_TEST_AUTH=true in your Convex deployment env vars."

## Non-goals

**Spec file changes**: This ticket does NOT modify any `e2e/*.spec.ts` file. The specs were written to the `helpers.ts` interface in WOR-107; once the helpers are real, the specs should work as-is.

**Playwright config changes**: No changes to `playwright.config.ts`. The `webServer` auto-start is not in scope.

**Test data cleanup**: No teardown or cleanup logic. Tests use unique timestamped emails (`test-${Date.now()}@example.com`) for isolation. Full cleanup is out of scope for v1.

**Claude mock changes**: `convex/lib/claudeMock.ts` and `e2e/claude-mock.ts` already exist and work. This ticket does not touch them.

**Password validation**: The Credentials provider's `authorize` callback does not validate a password. It accepts any request with a valid email. This is acceptable because it is gated behind `ENABLE_TEST_AUTH` and only used in dev/test deployments.
