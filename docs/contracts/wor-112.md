# Contract: WOR-112 — Invite Convex module — token generation + redeem mutation

## Why this work exists

When a user creates a case, the system generates a secure invite link for the second party. The second party clicks the link, authenticates, and redeems the token to join the mediation. This ticket builds the `convex/invites.ts` module: the pure-function token generator (already used inline in `convex/cases.ts` — this ticket extracts and exports it), the URL builder, and the `invites/redeem` mutation that atomically binds the invitee to the case. Without this module, there is no way for a second party to join a case.

## Files and exports

### `convex/invites.ts` (create)

A new Convex module file following the project's one-domain-module-per-file convention (see `convex/cases.ts`, `convex/users.ts`). Contains:

**`generateToken()`** — A pure helper function (no database access) that produces a 32-character crypto-random URL-safe string. Uses `crypto.getRandomValues(new Uint8Array(32))` and maps each byte to the alphabet `ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz0123456789-_` (64 characters, so `byte % 64` has zero modulo bias for a 256-value byte). This is the same algorithm currently inlined as `generateToken()` in `convex/cases.ts:211-217`; the implementation should be moved here and `convex/cases.ts` should import it from `convex/invites.ts` (or, if modifying `cases.ts` is out of scope for this ticket, the function can be duplicated — the test author will test the export from `convex/invites.ts`). The existing implementation in `cases.ts` already matches this spec.

**`buildInviteUrl(token: string)`** — A pure helper that returns `${process.env.SITE_URL ?? "http://localhost:5173"}/invite/${token}`. Extracted so tests can verify the URL format without calling the full `cases/create` mutation.

**`redeem`** — A Convex mutation with `args: { token: v.string() }`. Steps:

1. Call `requireAuth(ctx)` to get the authenticated user.
2. Look up the `inviteTokens` row via the `by_token` index.
3. If no row found or `status !== "ACTIVE"`, throw `tokenInvalid("Invite token is invalid or has already been used")` using the helper from `convex/lib/errors.ts`.
4. Load the case via `ctx.db.get(inviteToken.caseId)`.
5. If `caseDoc.initiatorUserId === user._id`, throw `conflict("Cannot redeem your own invite — you are already the initiator of this case")` using the helper from `convex/lib/errors.ts`.
6. Patch the case: `ctx.db.patch(caseId, { inviteeUserId: user._id, updatedAt: Date.now() })`.
7. Insert a `partyStates` row: `{ caseId, userId: user._id, role: "INVITEE" }`. No form fields — the invitee fills their form after redeeming.
8. Patch the invite token: `ctx.db.patch(inviteToken._id, { status: "CONSUMED", consumedAt: Date.now(), consumedByUserId: user._id })`.
9. Return `{ caseId }`.

All of steps 6-8 happen within the same Convex mutation transaction, ensuring atomicity.

### `tests/unit/invites.test.ts` (create)

Unit and integration tests using `convex-test` following the exact same pattern as `tests/unit/cases.test.ts`: import schema from `../../convex/schema`, use `convexTest(schema)`, seed test data with direct `ctx.db.insert` calls. Also imports the pure helpers `generateToken` and `buildInviteUrl` directly for pure-function tests (no Convex harness needed for those).

## Data dependencies

### Tables read

- **`inviteTokens`** — looked up by `by_token` index in `redeem`. Fields consumed: `caseId`, `status`, `_id`.
- **`cases`** — loaded by `_id` (from `inviteToken.caseId`). Fields consumed: `initiatorUserId` (for the self-redeem check).
- **`users`** — read indirectly via `requireAuth`.

### Tables written

- **`cases`** — patched: `inviteeUserId` set to caller, `updatedAt` refreshed.
- **`partyStates`** — new row inserted with `{ caseId, userId, role: "INVITEE" }`.
- **`inviteTokens`** — patched: `status` set to `"CONSUMED"`, `consumedAt` and `consumedByUserId` set.

### Key field flows

- `inviteTokens.token` → looked up via `by_token` index → the primary key of the redeem operation.
- `inviteTokens.caseId` → used to load the case and to create the `partyStates` row.
- `cases.initiatorUserId` → compared against caller's `user._id` for the CONFLICT check.
- `user._id` → written to `cases.inviteeUserId`, `partyStates.userId`, and `inviteTokens.consumedByUserId`.

## Invariants

### Atomicity of redeem

The `redeem` mutation performs three writes (patch case, insert partyStates, patch inviteToken) within a single Convex mutation. Convex mutations are transactional — all writes commit together or none do. This guarantees no partial state: you cannot have `inviteeUserId` set without the `partyStates` row, or a consumed token without the binding.

### Token validity check

The mutation checks `status === "ACTIVE"` on the `inviteTokens` row. Tokens with status `CONSUMED` or `REVOKED` (or tokens that don't exist) all result in `TOKEN_INVALID`. The error message is clear and actionable: `"Invite token is invalid or has already been used"`.

### Self-redeem prevention

If the authenticated caller is the case's `initiatorUserId`, the mutation throws `CONFLICT` with message `"Cannot redeem your own invite — you are already the initiator of this case"`. This prevents a user from being both initiator and invitee in a non-solo case.

### No status transition at redeem

The `redeem` mutation does NOT change `cases.status`. After redeem, the case remains `DRAFT_PRIVATE_COACHING`. The transition to `BOTH_PRIVATE_COACHING` happens when the invitee completes their intake form (via whichever mutation handles form completion — likely an extension of `cases/updateMyForm` or a dedicated mutation in a future ticket). This is an explicit TechSpec decision (§5): "The transition from DRAFT_PRIVATE_COACHING → BOTH_PRIVATE_COACHING happens when the invitee completes their form (not at the moment of redeem)."

### Crypto-random token generation

`generateToken()` uses `crypto.getRandomValues()` (CSPRNG), not `Math.random()`. The output is exactly 32 characters from the URL-safe alphabet `[A-Za-z0-9_-]`. With 64 possible characters per position and 32 positions, this gives 192 bits of entropy — more than sufficient for unguessability.

### Auth enforcement

`redeem` calls `requireAuth` as its first operation. Unauthenticated calls throw `ConvexError({ code: "UNAUTHENTICATED", httpStatus: 401 })`.

## Edge cases

### Token not found

If no `inviteTokens` row matches the provided token string, the mutation throws `TOKEN_INVALID`. This is the same error as a consumed/revoked token — we do not distinguish "not found" from "already used" to avoid information leakage about token existence.

### Double redeem (consumed token)

If user B redeems a token, and then user C tries to redeem the same token, user C gets `TOKEN_INVALID`. The token's status is `CONSUMED` (not `ACTIVE`), so the check fails.

### Initiator redeems own token

If the case initiator tries to redeem their own invite token, the mutation throws `CONFLICT` before any writes occur. The initiator is already bound to the case.

### Case already has an invitee (race condition)

Two users could theoretically try to redeem the same token simultaneously. Convex's serializable transactions handle this: the first to commit wins and sets the token to `CONSUMED`; the second sees `status !== "ACTIVE"` and gets `TOKEN_INVALID`. No explicit locking is needed.

### PartyStates row created without form fields

The `partyStates` row created by `redeem` has only `{ caseId, userId, role: "INVITEE" }`. The form fields (`mainTopic`, `description`, `desiredOutcome`) are all `v.optional` in the schema and are left undefined until the invitee completes their intake form via `cases/updateMyForm`.

## Non-goals

- **No case status transition at redeem** — handled by the form-completion mutation (separate ticket or extension of `updateMyForm`).
- **No token revocation** — the `REVOKED` status exists in the schema but no mutation in this ticket transitions tokens to it. Revocation is a future concern.
- **No invite listing queries** — there is no `invites/list` or `invites/get` query. The invite token is generated by `cases/create` (WOR-111) and consumed by `invites/redeem` (this ticket). No query surfaces invite data.
- **No email notifications** — deferred to v1.1 (US-16).
- **No UI** — this is backend-only.
- **No token hashing** — tokens are stored in plaintext per TechSpec decision. Acceptable because tokens are single-use, consumed instantly, and have no value after consumption.
