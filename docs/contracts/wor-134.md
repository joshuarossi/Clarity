# Contract: WOR-134 — Admin template edit page — two-pane editor with version history

## Why this work exists

Admins need to modify coaching template content and publish new immutable versions that drive AI behavior across all case types. The current `/admin/templates/:id` route is a stub placeholder. This ticket replaces that stub with a full two-pane editor showing the form on the left (for editing) and the version history timeline on the right (for reference), reducing the risk of accidental regressions when publishing changes.

## Files and exports

### `src/routes/AdminTemplateEditPage.tsx` (create, connected)

The primary page component for the `/admin/templates/:id` route. It is a **connected** component — it calls `useParams()` to extract the template ID, `useQuery(api.admin.get, { templateId })` to load the template with its current version content, `useQuery(api.admin.listVersions, { templateId })` to load the version timeline, and `useMutation(api.admin.publishNewVersion)` / `useMutation(api.admin.archive)` for write operations. It takes no props — everything comes from the route param and Convex queries.

The two-pane layout uses a CSS grid or flex container: left pane contains the form, right pane contains the version history timeline. On narrow viewports, panes may stack vertically (responsive, not explicitly in AC but follows the 8-point grid convention).

The "View" button on each version entry shows that version's content in a read-only overlay or inline expansion — not a diff, just the full content of that version.

### `src/App.tsx` (modify, route)

Replace the inline `AdminTemplateDetailPage` stub function (currently at ~line 31) with an import of `AdminTemplateEditPage` from `src/routes/AdminTemplateEditPage.tsx`. The route registration at `/admin/templates/:id` already exists wrapped in `<AdminRoute>`.

### `convex/admin.ts` (modify, query)

Add a new `get` query that returns a single template by ID with:

- The template document fields
- `currentVersion`: the full `templateVersions` document for the current version (needed to pre-populate the form)
- `pinnedCasesCount`: count of cases pinned to any version of this template (needed for the archive confirmation modal)

The existing `listVersions`, `publishNewVersion`, and `archive` exports remain unchanged.

## Data dependencies

### `api.admin.get` (new query)

Returns the template document enriched with `currentVersion` (full version doc with `globalGuidance`, `coachInstructions`, `draftCoachInstructions`, `notes`) and `pinnedCasesCount`. The form pre-populates from `currentVersion.globalGuidance`, `currentVersion.coachInstructions`, `currentVersion.draftCoachInstructions`. The `category` and `name` fields come from the template document itself.

### `api.admin.listVersions` (existing)

Returns `TemplateVersion[]` sorted by `version` descending (newest first). Each entry's `publishedAt` is rendered as a formatted date, `publishedByUserId` is resolved to a display name (the component will need to either resolve user names client-side via a secondary query or the `listVersions` query should be enhanced to include the admin display name). **Decision:** Enhance `listVersions` to join the `publishedByUserId` to `users.displayName` so the timeline can show admin names without N+1 client queries.

### `api.admin.publishNewVersion` (existing mutation)

Called with `{ templateId, globalGuidance, coachInstructions, draftCoachInstructions, notes }`. On success, the component shows a success toast/feedback and the reactive queries automatically update (new version appears in timeline, form content reflects new current version).

### `api.admin.archive` (existing mutation)

Called with `{ templateId }`. On success, the page redirects to `/admin/templates` (the list page).

## Invariants

**Admin-only access:** The route is wrapped in `<AdminRoute>` which checks `user.role === "ADMIN"` via `api.users.me`. Non-admins are redirected to `/dashboard`. The `get` query also enforces `requireAdmin` server-side — even if the client somehow renders, it won't get data.

**Immutable versions:** `publishNewVersion` creates a new row in `templateVersions` with a monotonically increasing `version` number. It never modifies existing version rows. The form always shows the _current_ version's content as a starting point for the _next_ version.

**Archive confirmation requires pinned case count:** The archive modal must fetch and display `pinnedCasesCount` before the admin can confirm. This is provided by the `get` query — no additional fetch is needed.

**Form pre-population:** When the page loads, all editable fields are pre-populated with the current version's content. The `category` and `name` fields are displayed (read-only informational context) but are not part of `publishNewVersion` — they're set at template creation time.

**Newest-first timeline:** Version history is sorted descending by `version` number so the most recent publication is at the top.

## Edge cases

**Loading state:** While `api.admin.get` or `api.admin.listVersions` returns `undefined` (loading), show skeleton/spinner placeholders in both panes. Do not show the form with empty fields — that could be confused with a blank template.

**Template not found:** If `api.admin.get` returns `null` (invalid ID or deleted template), show a "Template not found" message with a link back to `/admin/templates`.

**Archived template:** If the template has `archivedAt` set, the form fields should be disabled/hidden and the page should show an "Archived" badge. The version timeline remains visible for historical reference. The "Publish New Version" and "Archive Template" buttons are not shown.

**Empty version history:** Impossible in practice — `admin.create` always creates a v1. But if somehow encountered, the timeline pane shows "No versions published yet."

**Publish validation:** The "Publish New Version" button should be disabled if `globalGuidance` is empty (the one required content field). Other fields are optional per the mutation schema.

**Concurrent edits:** Not a concern for v1 — single admin expected. Convex's reactive queries ensure the page reflects the latest state.

## Non-goals

- **Rich markdown editor** — plain textarea for all markdown fields in v1.
- **Side-by-side diff view** — "View" shows the full content of a historical version, not a comparison.
- **Undo/revert** — no mechanism to revert to a previous version (admin can manually copy content).
- **Template creation** — `/admin/templates/new` is separate scope (may reuse this component in the future but is not part of this ticket).
- **Category/name editing** — these are immutable after creation; displayed for context only.
