# Contract: WOR-143 — Admin "+ New Template" white-screens — no template-creation UI exists

## Why this work exists

The "+ New Template" button navigates to `/admin/templates/new`, but that URL is caught by the `/admin/templates/:id` route, which treats `"new"` as a Convex document ID. The resulting argument-validation error from `api.admin.get` causes a white screen. The backend `api.admin.create` mutation exists and works but has zero UI callers — no creation form was ever built.

## Files and exports

### `src/routes/AdminTemplateCreatePage.tsx` (create)

A new connected page component that renders the template creation form. It is the dedicated handler for `/admin/templates/new`. It owns its own state for the form fields (category, name, globalGuidance, coachInstructions, draftCoachInstructions), calls `useMutation(api.admin.create)` on submit, and navigates to `/admin/templates/<newId>` on success. This is a connected component — it calls hooks internally and takes no props.

### `src/App.tsx` (modify)

Add a new `<Route path="/admin/templates/new" ...>` **before** the existing `/admin/templates/:id` route. React Router matches routes in order; placing `/new` first ensures the literal path wins over the parameterized path. The route wraps `AdminTemplateCreatePage` in `<AdminRoute>`.

### `src/routes/AdminTemplateEditPage.tsx` (modify — minimal)

No functional changes required. The routing fix (dedicated `/new` route) means `AdminTemplateEditPage` will never receive `id === "new"`. However, as a defensive measure, add an early guard: if `id === "new"`, navigate to `/admin/templates/new`. This prevents deep-links or browser history from triggering the bug if routing order is ever accidentally changed.

## Data dependencies

### `api.admin.create`

Already exists at `convex/admin.ts:111`. Args: `{ category: v.string(), name: v.string(), globalGuidance: v.string(), coachInstructions: v.optional(v.string()), draftCoachInstructions: v.optional(v.string()) }`. Returns: the new template's `Id<"templates">`. The form must collect at minimum `category`, `name`, and `globalGuidance` (required). Coach instructions fields are optional.

### `api.admin.listAll`

Used by `AdminTemplatesPage` to populate the list. Not directly used by the create page, but after creation the admin navigates to the edit page which calls `api.admin.get`.

## Invariants

1. **Route ordering**: `/admin/templates/new` must appear before `/admin/templates/:id` in App.tsx. If this ordering is violated, the white-screen bug recurs.

2. **No invalid ID to Convex**: The create page never calls `api.admin.get` or `api.admin.listVersions`. It only calls `api.admin.create` on form submit.

3. **Admin-only access**: The route is wrapped in `<AdminRoute>`, which gates on the user's `role === "ADMIN"` server-side check. No additional auth logic needed in the component.

4. **Required fields**: `globalGuidance` is required by the mutation validator. The submit button must be disabled when `category`, `name`, or `globalGuidance` is empty.

## Edge cases

**Loading state**: The create form has no data to load (it's a blank form), so no loading skeleton is needed. The submit button shows "Creating..." while the mutation is in flight.

**Validation error**: If `api.admin.create` throws (e.g., duplicate name within category — not currently enforced but possible), display the error message below the form using the same `handleConvexError` pattern as `AdminTemplateEditPage`.

**Empty category list**: Category is a free-text string field (matching the existing mutation args). No server-side category list to load. Use a text input or a select with hardcoded categories matching the project's known categories (workplace, family, roommate, etc.). The existing schema uses `v.string()` so free-text is acceptable.

**Navigation after success**: On success, navigate to `/admin/templates/<returnedId>`. This lands the admin on the edit page for the template they just created, where they can see their initial version in the version history.

## Non-goals

- **No backend changes**: `api.admin.create` is already implemented and tested. This ticket only builds the missing frontend.
- **No role field**: The PRD mentions a "role" concept (Coach / Draft Coach / both) but the existing schema and mutation don't include it. Adding it is a separate enhancement.
- **No changes to edit/archive/version flows**: Those already work for existing templates.

## Test coverage

- **Unit tests** (`tests/unit/admin-template-create-page.test.tsx`): Verify form renders with required fields, submit button disabled when fields empty, successful submission calls `api.admin.create` with correct args, navigation occurs on success, error display on failure. Also verify that navigating to `/admin/templates/new` does NOT trigger the old white-screen path.

- **E2E test** (`e2e/admin-templates.spec.ts`): Add a test case within the existing admin-templates e2e spec that clicks "+ New Template", fills the form, submits, and verifies the admin lands on the edit page for the new template with no white-screen. This covers the full routing + Convex round-trip that was broken.
