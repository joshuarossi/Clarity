# Contract: WOR-149 — Private Coach chat does not render markdown (US-06a)

## Why this work exists

The PRD (US-06a) requires the private coaching chat UI to render markdown in Coach messages: "stream messages, markdown rendering, typing indicator." Currently, `MessageBubble` renders all message content as `<span>{content}</span>` — plain text. Markdown characters like `**bold**`, `### heading`, and `- list item` appear as literal source characters, making structured Coach responses hard to read. A hand-rolled markdown renderer already exists in `ReadyForJointView.tsx` (`SynthesisMarkdown`) but is not shared. This ticket extracts it into a reusable component and wires it into `MessageBubble` for coach variants.

## Files and exports

### `src/components/chat/MarkdownContent.tsx` (presentational, create)

A shared component that converts a markdown string into React elements. It handles the subset of markdown that the Coach AI actually produces: `###` headings, `**bold**`, `*italic*`, unordered lists (`- item`), ordered lists (`1. item`), and paragraph breaks (blank lines). This replaces the inline `SynthesisMarkdown` function in `ReadyForJointView.tsx` and is used by `MessageBubble` for coach variants.

**Why a new file rather than inline in MessageBubble:** The renderer is already needed in two places (synthesis view and chat bubbles) and may be needed in the Draft Coach panel. Extracting it keeps `MessageBubble.tsx` focused on layout/variant logic and the renderer independently testable.

**Why hand-rolled, not `react-markdown`:** The project has no markdown library dependency. The subset needed (headings, bold, italic, lists) is small. Adding `react-markdown` + `remark` + `rehype` would pull in ~15 transitive dependencies for something achievable in ~60 lines of React. The existing `SynthesisMarkdown` proves this approach works; this ticket generalizes it to also handle italic and lists.

### `src/components/chat/MessageBubble.tsx` (presentational, modify)

The only change is in the render body: replace `<span>{content}</span>` with a conditional. When `variant` is `"coach"`, `"coach-joint"`, or `"coach-intervention"`, render `<MarkdownContent content={content} />`. For all other variants (`"user"`, `"party-initiator"`, `"party-invitee"`, `"error"`), continue rendering `<span>{content}</span>`. No prop changes, no type changes, no export changes.

The decision of which variants get markdown is based on who produces markdown: only the AI Coach. User-typed messages and party messages are plain text. Error messages are system-generated plain text.

### `src/routes/ReadyForJointView.tsx` (connected, modify)

Remove the inline `SynthesisMarkdown` function (lines 12–47) and replace its usage with `<MarkdownContent content={synthesis.text} />`. This is a pure refactor — no behavioral change. The synthesis view already renders markdown; it just uses an inline implementation that should now come from the shared component.

## Data dependencies

None. This is a pure presentational change. No Convex queries, mutations, or actions are added or modified. The `content` string is already available as a prop on `MessageBubble` and as `synthesis.text` in `ReadyForJointView`.

## Invariants

**Variant-gated rendering.** Markdown rendering must ONLY apply to the three coach variants: `"coach"`, `"coach-joint"`, `"coach-intervention"`. The conditional in MessageBubble must be explicit — a set/array check, not a "not user" check — so that new variants added later default to plain text.

**No dangerouslySetInnerHTML.** The renderer builds a React element tree from parsed tokens. It never inserts raw HTML strings. This prevents XSS if message content ever contains `<script>` or similar.

**Safe element allowlist.** The renderer only produces: `<h3>`, `<p>`, `<strong>`, `<em>`, `<ul>`, `<ol>`, `<li>`, `<span>`. No other HTML elements.

**Copy button copies raw source.** The existing copy handler uses `navigator.clipboard.writeText(content)` — it copies the raw markdown string, not rendered HTML. This is correct and must not change.

**No API changes.** `MessageBubbleProps` does not gain a `renderMarkdown` prop or similar. The decision is internal, based on the `variant` prop that already exists.

## Edge cases

**Plain text content in a coach bubble.** If a coach message contains no markdown syntax, `MarkdownContent` renders it as a single `<p>` element. This is visually identical to the old `<span>` rendering.

**Streaming content.** During `STREAMING` status, the content string grows incrementally. `MarkdownContent` must handle partial markdown gracefully — e.g., `**bol` (incomplete bold) should render as literal `**bol` text, not crash. The simple regex approach handles this naturally: unmatched `**` is left as text.

**Empty content.** An empty string produces no elements. The bubble renders correctly (just the avatar/timestamp/copy button area).

**Nested markdown.** `**bold *and italic***` — the renderer does NOT need to handle nesting. Bold and italic are parsed independently at the line level. This matches the existing `SynthesisMarkdown` behavior.

**Very long lists.** Lists with many items render naturally with no truncation. They are constrained by the bubble's `max-width` CSS.

## Non-goals

**No external markdown library.** The project has no `react-markdown` dependency and this ticket does not add one. The hand-rolled approach covers the Coach's output subset.

**No full CommonMark support.** Code blocks (`` ` ``), tables, images, links, horizontal rules, and blockquotes are not rendered. If the Coach emits these, they appear as literal text — acceptable for v1.

**No markdown rendering for user messages.** Users type plain text. Rendering their input as markdown would cause unexpected formatting (e.g., a user typing `**not bold**` would see it rendered as bold).

**No changes to public API.** `MessageBubbleProps`, `BubbleVariant`, `MessageStatus`, `ChatMessage` — all unchanged.

**No CSS changes.** Markdown elements (`h3`, `p`, `strong`, `em`, `ul`, `ol`, `li`) inherit styling from the bubble's existing CSS context. No new CSS classes or style rules are added by this ticket. If the default browser styling for these elements is insufficient, that's a follow-up styling ticket.

3. *Sanitization* — Renders content containing `<script>alert('xss')</script>` and asserts no `<script>` element appears in the DOM. Renders content with HTML tags and asserts they appear as escaped text, not rendered HTML.

**`tests/unit/message-bubble.test.tsx`** — existing file, add tests for the new AC:

4. *Variant-gated markdown* — Render a `coach` variant with `**bold**` content, assert `<strong>` element exists.
