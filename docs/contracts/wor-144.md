# Contract: WOR-144 — Joint session has no Coach opening message (US-09b)

## Why this work exists

Per PRD US-09b, the Coach AI must post an opening message grounded in the case's main topic when a party first enters the joint session. Currently, `enterSession` only performs a status transition — it never schedules any Coach message insertion. The joint chat opens empty until a party sends the first message, violating the "neutral facilitator opens the conversation" UX.

## Files and exports

### `convex/jointChat.ts` (modify)

**`enterSession`** — The existing mutation already validates the `START_JOINT` transition and patches the case status. The fix adds a single `ctx.scheduler.runAfter(0, internal.jointChat.generateCoachOpeningMessage, { caseId })` call after the status patch. No other changes to this function.

**`generateCoachOpeningMessage`** — A new `internalAction` that:

1. Reads the case via `getCaseForCoach` and validates status is `JOINT_ACTIVE`.
2. Reads party states via `getPartyStatesForCoach` to get the INITIATOR's `mainTopic`.
3. Reads the template version via `getTemplateVersionForCoach`.
4. Assembles a COACH prompt using `assemblePrompt` with a context that includes the mainTopic and both parties' synthesis texts. The prompt's user message should instruct the Coach to produce an opening message for the joint session grounded in the case's main topic.
5. Inserts a STREAMING coach message via `insertCoachStreamingMessage`.
6. Calls the Claude API (Sonnet) with streaming (or uses mock mode), updating the message row as tokens arrive via `updateCoachStreamingMessage`.
7. Runs the privacy filter against both parties' raw private messages.
8. Finalizes the message via `finalizeCoachMessage`, or marks ERROR on failure via `markCoachMessageError`.

This action reuses the same streaming + retry + privacy filter pattern as `generateCoachResponse`, but without the Haiku classification step (the opening message is unconditional — there is no gate check).

## Data dependencies

- **`internal.jointChat.getCaseForCoach`** — Returns the case document. Fields consumed: `status`, `templateVersionId`.
- **`internal.jointChat.getPartyStatesForCoach`** — Returns all party states for the case. Fields consumed: `role`, `mainTopic`, `synthesisText`.
- **`internal.jointChat.getTemplateVersionForCoach`** — Returns the template version. Fields consumed: `globalGuidance`, `coachInstructions`.
- **`internal.jointChat.getPrivateMessagesForCoach`** — Returns private messages for privacy filter. Fields consumed: `role`, `content`.

All of these internal queries already exist and are unchanged.

## Invariants

**Single execution per case:** The state machine guarantees `START_JOINT` is only valid from `READY_FOR_JOINT`. Once the case is in `JOINT_ACTIVE`, calling `enterSession` again would fail validation. Therefore the `ctx.scheduler.runAfter` call in `enterSession` executes at most once per case lifecycle.

**Grounded in mainTopic:** The action reads the INITIATOR's `partyState.mainTopic` and passes it into the prompt context. The assembled prompt instructs the Coach to produce an opening message that references this topic, providing grounding per US-09b.

**Privacy compliance:** The generated opening message passes through the same `filterResponse` check used by `generateCoachResponse`. If the filter rejects the response (near-verbatim match with private content), the action retries up to 3 times, then emits a generic fallback.

**Error handling:** If the Claude API call fails after retries, the message row is set to `status: "ERROR"`. The UI already handles ERROR-status messages with a retry affordance.

## Edge cases

**Mock mode (`CLAUDE_MOCK=true`):** The action checks `isClaudeMockEnabled()` and uses `getMockClaudeResponse("COACH")` for the opening message content, following the same mock streaming pattern as `generateCoachResponse`.

**Missing mainTopic:** If the INITIATOR's partyState has no `mainTopic` (edge case — shouldn't happen in normal flow since form completion precedes private coaching), the action still proceeds with whatever context is available. The prompt will still produce a sensible opening without a topic string.

**Concurrent entry (solo mode):** In solo mode, both "parties" share the same user. The first call to `enterSession` transitions the case; a second call would fail the state machine check. The opening message is generated once regardless.

## Non-goals

- No UI changes. The joint chat component already renders all `jointMessages` rows including COACH messages. The opening message appears naturally as the first message in the chat.
- No schema changes. The opening message is a standard `jointMessages` document with `authorType: "COACH"`, `status: "STREAMING" → "COMPLETE"`, and `isIntervention: false`.
- No new Convex internal queries. All needed data is accessible through existing internal queries.
- No changes to the prompt assembly module (`convex/lib/prompts.ts`). The existing `assembleCoach` function handles the COACH role adequately — the opening message action will pass synthesis context and an instruction in `recentHistory` to direct the Coach to produce an opening.

## Test coverage

The acceptance criterion is covered by **unit tests** in `tests/unit/joint-chat.test.ts`:

1. **`enterSession` schedules `generateCoachOpeningMessage`** — After calling `enterSession` on a `READY_FOR_JOINT` case, verify that the scheduler was invoked with `internal.jointChat.generateCoachOpeningMessage` and the correct `caseId`.
2. **`generateCoachOpeningMessage` inserts a COACH message** — Using `convex-test` with `CLAUDE_MOCK=true`, call the internal action directly and verify a `jointMessages` row with `authorType: "COACH"` and `status: "COMPLETE"` is inserted for the case.
3. **Opening message references mainTopic** — In mock mode the content is canned, but the test can verify the prompt assembly receives the INITIATOR's `mainTopic` in context (by checking the assembled prompt or by verifying the action doesn't throw when `mainTopic` is set).
4. **Re-entry guard** — Calling `enterSession` on a case already in `JOINT_ACTIVE` throws a `CONFLICT` error (this test likely already exists but should be confirmed).

Unit tests are sufficient because: the state machine guard is already tested, the privacy filter is tested independently, and the Claude mock mode provides deterministic behavior for the generation path. The wiring (scheduler call from mutation → action execution) is testable via `convex-test`'s scheduler support.
