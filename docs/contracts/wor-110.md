# Contract: WOR-110 — Login/Register page and profile page

## Why this work exists

Users cannot currently sign in or manage their identity in Clarity. The app has a Convex Auth backend (WOR-109/T15) and shared UI primitives (WOR-105/T8), but the login page is still a stub `<h1>Login</h1>` and no profile page or `/profile` route exists. This ticket builds the two pages that sit on top of the auth infrastructure: a login/register page at `/login` with magic-link and Google OAuth, and a minimal profile page at `/profile` for viewing email, editing display name, and signing out.

## Files and exports

### `src/routes/LoginPage.tsx` (connected, create)

New file. This is a **connected** component — it calls `useConvexAuth()` for auth state and `signIn` from `@convex-dev/auth/react` to trigger magic-link send and Google OAuth. It manages three internal states: (1) default form with email input + buttons, (2) magic-link-sent confirmation ("Check your email..."), and (3) inline error below the email input. The component reads `?redirect=` from the URL search params (via `useSearchParams()`) to know where to navigate post-login; defaults to `/dashboard`.

The card is centered with `max-width: 400px`. Uses the project's `Button` component (`cc-btn-primary` for "Send magic link", `cc-btn-secondary` for "Continue with Google"). No password field.

This file replaces the inline stub `LoginPage` function currently in `src/App.tsx`.

### `src/routes/ProfilePage.tsx` (connected, create)

New file. **Connected** component — queries `api.users.me` for the current user's data and calls `api.users.updateDisplayName` mutation to save display name edits. Also calls `signOut` from `@convex-dev/auth/react` for the sign-out button.

Layout: simple centered card (matching login card width for visual consistency). Shows:

- Display name in an editable text input with a "Save" button (or inline save)
- Email as read-only text (not an input, or a disabled input)
- Sign-out button (`cc-btn-danger` or `cc-btn-secondary`)

No avatar upload per DesignDoc. This page is wrapped in `ProtectedRoute` in the route tree.

### `convex/users.ts` (query, modify)

Adds a new `updateDisplayName` mutation alongside the existing `me` query. The mutation calls `requireAuth(ctx)` from `convex/lib/auth.ts` to get the authenticated user, then patches the user's `displayName` field. Validates that `displayName` is a non-empty string.

### `src/App.tsx` (route, modify)

Three changes:

1. Import `LoginPage` from `src/routes/LoginPage.tsx` and `ProfilePage` from `src/routes/ProfilePage.tsx` (replacing the inline `LoginPage` stub).
2. Add a `/profile` route wrapped in `ProtectedRoute`.
3. The `/login` route element continues to redirect authenticated users to `/dashboard`.

The inline `LoginPage` stub function is removed from this file.

### `src/components/layout/ProtectedRoute.tsx` (connected, modify)

Modified to encode the current location as a `?redirect=` search param when redirecting unauthenticated users to `/login`. Changes from `<Navigate to="/login" replace />` to `<Navigate to={"/login?redirect=" + encodeURIComponent(location.pathname + location.search)} replace />`. Uses `useLocation()` from React Router.

## Data dependencies

### `api.users.me`

Returns the full user document or `null`. Fields consumed by this work:

- `email` — displayed read-only on the profile page
- `displayName` — displayed in an editable input on the profile page
- `_id` — not directly rendered but used internally

### `api.users.updateDisplayName`

New mutation. Takes `{ displayName: string }`. Authenticates via `requireAuth`, patches the user record. Returns `null`.

### `signIn` (from `@convex-dev/auth/react`)

Called on LoginPage with two provider configurations:

- Magic link: `signIn("magic-link", { email })` — sends a verification email
- Google OAuth: `signIn("google")` — redirects to Google consent screen

### `signOut` (from `@convex-dev/auth/react`)

Called on ProfilePage sign-out button click. Clears client and server session. After `signOut()` resolves, the component navigates to `/login`.

### `useConvexAuth` (from `@convex-dev/auth/react`)

Returns `{ isLoading: boolean, isAuthenticated: boolean }`. Used by LoginPage to confirm auth state, and already used by ProtectedRoute and App.tsx.

## Invariants

**Authenticated users cannot see LoginPage.** The `/login` route in `App.tsx` already has a guard: `isAuthenticated && !isLoading ? <Navigate to="/dashboard" replace /> : <LoginPage />`. This is unchanged.

**ProfilePage requires authentication.** It is wrapped in `ProtectedRoute` in the route tree. If an unauthenticated user navigates to `/profile`, they are redirected to `/login?redirect=%2Fprofile`.

**Post-login redirect uses URL search param.** The `?redirect=` param is set by `ProtectedRoute` and read by `LoginPage` after successful auth. This survives page reloads (unlike React Router state) and works with both magic-link (which opens a new tab/window from the email) and Google OAuth (which redirects back). If no `?redirect=` param is present, LoginPage redirects to `/dashboard`.

**Server-side auth enforcement on mutations.** `updateDisplayName` calls `requireAuth(ctx)` — there is no client-only trust. An unauthenticated call throws `UNAUTHENTICATED` (401).

**No password field.** The login form has exactly two sign-in methods: magic-link email and Google OAuth. No password input is rendered.

## Edge cases

**Loading state (LoginPage).** While `isLoading` is true from `useConvexAuth()`, the login page shows the card with a loading spinner or disabled form to prevent interaction before auth state is known. This prevents a flash of the login form for already-authenticated users.

**Loading state (ProfilePage).** While `api.users.me` is loading (returns `undefined` from `useQuery`), the profile page shows a loading spinner (using the existing `LoadingSpinner` component). Once loaded, the form populates.

**Magic-link send failure.** If `signIn("magic-link", { email })` throws, the error message is displayed inline below the email input. The form remains interactive so the user can retry. Common error: invalid email format.

**Google OAuth failure.** If Google OAuth fails (popup blocked, user cancels, network error), the login page returns to its default form state. An inline error may be shown if the auth library surfaces one.

**Empty display name on profile.** The `displayName` field is optional in the schema. If null/undefined, the input should show an empty string as the placeholder value, and the user can type and save. The mutation validates non-empty.

**Profile page when user record is null.** This shouldn't happen (ProtectedRoute ensures auth, and auth ensures a user record via `createOrUpdateUser` callback). But if `api.users.me` returns `null` for an authenticated user, show an error state rather than crashing.

## Non-goals

**Avatar upload** is explicitly out of scope. The DesignDoc notes the profile page is minimal with no avatar.

**Password-based registration** is not part of v1 per the PRD. Magic-link and Google OAuth only.

**Email change** is not supported. Email is read-only on the profile page.

**Account deletion** is manual via support email in v1 per the PRD; no UI for it.

**Session expiry (30-day)** is handled by the Convex Auth provider configuration, not by this ticket's UI code.
