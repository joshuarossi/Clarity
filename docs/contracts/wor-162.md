# Contract: WOR-162 — Tailwind (v4) + shadcn never scaffolded

## Why this work exists

WOR-103 established the project's design token foundation (CSS custom properties, `.cc-*` component classes, dark theme) but never installed or configured Tailwind CSS itself. As a result, ~20 source files use Tailwind utility classes (e.g. `mx-auto`, `max-w-3xl`, `px-4`, `space-y-4`, `text-red-600`, `bg-gray-200`) that are completely inert — they produce no styles, and those pages render unstyled. The TechSpec and STYLE_GUIDE both specify Tailwind v4 + shadcn/ui as the styling layer, and the STYLE_GUIDE explicitly maps design tokens to Tailwind utility names (e.g. `bg-canvas`, `text-primary`). This ticket completes the scaffolding that WOR-103 missed.

## Files and exports

### `package.json` (modify)

Add four new dependencies: `tailwindcss` (v4), `@tailwindcss/vite` (Tailwind's v4 Vite plugin, as a devDependency), `tailwind-merge` (for the `cn()` utility), and `clsx` (conditional class construction). These are the minimal packages needed for Tailwind v4 + shadcn-style class composition.

### `vite.config.ts` (modify)

Import and register `@tailwindcss/vite` as a Vite plugin. This is how Tailwind v4 integrates with Vite — it replaces the PostCSS-based approach used in v3. Also add a `resolve.alias` entry mapping `@` to `src/` so that `@/lib/utils`, `@/components/ui/button` etc. resolve correctly (required by shadcn CLI conventions and makes imports consistent).

### `tsconfig.json` (modify)

Add `baseUrl: "."` and `paths: { "@/*": ["./src/*"] }` to `compilerOptions`. This mirrors the Vite alias so TypeScript resolves `@/` imports at type-check time. Without this, `tsc --noEmit` would fail on any `@/` import.

### `src/styles/globals.css` (modify)

Add `@import "tailwindcss"` at the very top (before the existing `@import "./components.css"`). Then add a `@theme` block that maps all existing CSS custom properties to Tailwind's utility namespace. For example, `--color-canvas: var(--bg-canvas)` makes `bg-canvas` a valid Tailwind utility. The existing `:root` / `[data-theme="dark"]` custom property blocks and all other existing CSS remain unchanged — the `@theme` block sits alongside them and extends Tailwind's theme with project-specific utilities.

### `src/lib/utils.ts` (create)

Standard shadcn `cn()` utility — wraps `clsx()` with `twMerge()` so that conflicting Tailwind classes resolve correctly (last wins). This is the conventional export that all shadcn/ui components import for class composition.

### `components.json` (create)

shadcn/ui CLI configuration file at project root. Specifies the `new-york` style, enables `cssVariables`, and maps alias paths (`@/components`, `@/lib`, `@/components/ui`, `@/hooks`). This allows `npx shadcn@latest add <component>` to scaffold components into the right directories. The project already has a Button component; this config file makes it possible to add more via the CLI.

### `src/components/ui/button.tsx` (modify)

Replace the manual `[...].filter(Boolean).join(" ")` class composition with `cn(buttonVariants({ variant, size }), className)`. The public API (props, exports) is unchanged — this is purely an internal improvement to use the new `cn()` utility, which handles Tailwind class conflicts correctly.

## Data dependencies

None. This ticket has no Convex queries, mutations, or actions — it is purely frontend tooling infrastructure.

## Invariants

**Existing custom CSS must be preserved.** The `.cc-*` component classes in `components.css`, all CSS custom property declarations in `globals.css`, and the `[data-theme="dark"]` selector block must continue to work exactly as before. Tailwind v4's `@import "tailwindcss"` provides a base layer (preflight/reset + utilities) but must not clobber the project's existing styles. The `@theme` block extends Tailwind's theme — it does not replace the `:root` custom properties.

**Tailwind v4 uses CSS-first configuration.** There is no `tailwind.config.ts` or `tailwind.config.js` file. Tailwind v4 uses `@import "tailwindcss"` + `@theme { }` blocks directly in CSS. The STYLE_GUIDE mentions a `tailwind.config.ts` as a "v4 shim for tooling" but in practice Tailwind v4 does not require it, and creating one would introduce a conflicting configuration source.

**The cn() utility must use tailwind-merge.** Plain `clsx` would not handle Tailwind class conflicts (e.g. `cn('px-4', 'px-2')` must yield `px-2`, not `px-4 px-2`). The `tailwind-merge` library provides this behavior and is the standard approach used by shadcn/ui.

**Build must succeed.** `npm run build` (`tsc -b && vite build`) must complete without errors after all changes. This means the `@` path alias must be consistent between tsconfig and Vite, and all new imports must resolve.

## Edge cases

**Tailwind preflight vs. existing body styles.** Tailwind v4's preflight (CSS reset) sets `margin: 0` on body and resets heading sizes, list styles, etc. The project already sets `body { margin: 0; }` in globals.css, so no conflict there. However, any components relying on default browser heading sizes (without explicit sizing) may render differently after preflight. The `@theme` block and existing `.cc-*` classes should cover all styled elements, but if any unstyled headings or lists appear visually different, that is expected — they were relying on browser defaults, not intentional design.

**Tailwind v4 dark mode.** Tailwind v4 supports dark mode via `@variant dark` or the `dark:` prefix. The project uses a custom `[data-theme="dark"]` attribute, not the standard `class="dark"` or `prefers-color-scheme` that Tailwind uses by default. For the `dark:` variant to work with `data-theme`, a `@custom-variant dark` declaration would be needed. However, since this ticket is scaffolding-only and the existing dark theme uses CSS custom properties (which apply regardless of Tailwind), this is not required now. If future components want to use `dark:` utilities, a follow-up ticket should add the custom variant.

**Order of CSS imports.** The `@import "tailwindcss"` directive must appear before `@import "./components.css"` so that Tailwind's base layer is available when component classes reference Tailwind utilities. The existing `@import "./components.css"` line moves after the Tailwind import.

## Non-goals

**Migrating `.cc-*` classes to Tailwind utilities.** The 20+ files using custom `.cc-*` classes will continue to do so. Converting them to Tailwind utilities is a separate refactor effort, not part of this bug fix.

**Adding new shadcn/ui components.** This ticket scaffolds the infrastructure (components.json, cn(), path aliases) so that future tickets can add components via `npx shadcn@latest add`. It does not add any new components itself.

**Changing visual appearance.** This is infrastructure. After this ticket, pages that use Tailwind utility classes will render correctly (they were previously unstyled). Pages using only `.cc-*` classes will look identical.

**Creating a `tailwind.config.ts`.** Tailwind v4 uses CSS-based configuration. A JS config file is not needed and would introduce a conflicting config source.

## Test coverage

**`tests/unit/tailwind-setup.test.ts`** covers the scaffolding ACs: verifies that `package.json` lists the required dependencies (`tailwindcss`, `@tailwindcss/vite`, `tailwind-merge`, `clsx`), that `vite.config.ts` imports and registers the Tailwind plugin, that `globals.css` contains the `@import "tailwindcss"` directive and a `@theme` block, that `components.json` exists with correct configuration, that `src/lib/utils.ts` exports a `cn` function, and that `tsconfig.json` has the `@/*` path alias. These are structural/file-content checks verifiable without a running app.

**`tests/unit/button.test.tsx`** (existing file, extend) adds a test verifying the Button component uses `cn()` for class composition — e.g. passing a `className` prop that conflicts with a default class results in the override winning (tailwind-merge behavior).

**`e2e/dark-mode.spec.ts`** (existing file, extend) adds a check that after Tailwind is added, the existing dark mode toggle still works correctly — design tokens apply in both themes. This is the best existing e2e test to extend for visual regression, since it already exercises theme switching. Unit tests cannot verify that Tailwind's preflight doesn't clobber custom CSS; that requires a real browser.
