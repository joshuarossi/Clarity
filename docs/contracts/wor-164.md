# Contract: WOR-164 — requireAuth passes raw identity.subject to db.get ('Invalid ID length 65')

## Why this work exists

The `@convex-dev/auth` library stores `identity.subject` as a composite string `"userId|sessionId"` (two Convex IDs joined by `|`). The current code in `requireAuth`, `requireAdmin`, and `users.me` casts `identity.subject` directly to `Id<"users">` and passes it to `ctx.db.get()`. Since the composite string is ~65 characters (two 32-char IDs + separator), Convex throws "Invalid ID length 65". The fix is to use `getAuthUserId(ctx)` from `@convex-dev/auth/server`, which splits the composite subject on `|` and returns only the userId portion.

This is a follow-up to WOR-163, which fixed `identity.email` → `identity.subject` but didn't account for the subject being composite rather than a raw ID.

## Files and exports

### `convex/lib/auth.ts` (modify)

The central auth helper module. Two functions need changes:

- **`requireAuth`** (lines 15–40): Replace `identity.subject as Id<"users">` with a call to `getAuthUserId(ctx)`. If `getAuthUserId` returns `null`, throw the existing UNAUTHENTICATED error. Otherwise use the returned ID for `ctx.db.get()`. The function's public signature is unchanged.

- **`requireAdmin`** (lines 93–118): Same change as `requireAuth` — replace `identity.subject as Id<"users">` with `getAuthUserId(ctx)`. The function's public signature is unchanged.

The import block gains `import { getAuthUserId } from "@convex-dev/auth/server";`. The existing imports of `Auth` from `convex/server` and the custom type aliases remain. The `getUserIdentity()` calls inside `requireAuth` and `requireAdmin` are replaced entirely by `getAuthUserId`, which handles the null-identity check internally (it returns `null` when not authenticated).

`getUserByEmail` and `requirePartyToCase` are untouched — they don't interact with `identity.subject`.

### `convex/users.ts` (modify)

- **`me` query** (lines 6–20): Replace the manual `identity.subject as Id<"users">` pattern with `getAuthUserId(ctx)`. If it returns `null`, return `null`. Otherwise use the returned ID for `ctx.db.get()`.

- **`updateDisplayName` mutation**: Unchanged — it delegates to `requireAuth`.

The import block gains `import { getAuthUserId } from "@convex-dev/auth/server";`.

## Data dependencies

- **`ctx.db.get(userId)`**: Direct document lookup by `Id<"users">`. Returns the user document or `null`. This is unchanged — the fix is in how `userId` is derived (via `getAuthUserId` instead of raw `identity.subject`).

- **`getAuthUserId(ctx)`**: From `@convex-dev/auth/server`. Calls `ctx.auth.getUserIdentity()`, splits `identity.subject` on `"|"`, and returns the first segment as `GenericId<"users">`. Returns `null` if not authenticated.

## Invariants

### Composite subject format
`identity.subject` in `@convex-dev/auth` is always `"userId|sessionId"` — two Convex document IDs joined by the pipe character `|`. This is set in `@convex-dev/auth/src/server/implementation/tokens.ts` line 41: `sub: args.userId + TOKEN_SUB_CLAIM_DIVIDER + args.sessionId`. Passing this composite string directly to `db.get()` is always invalid.

### getAuthUserId is the canonical accessor
After this fix, no code in the project should use `identity.subject` directly for user resolution. All paths must go through `getAuthUserId(ctx)`, which correctly splits the composite subject. This is the pattern recommended by the `@convex-dev/auth` library documentation and examples.

### Error contract preserved
The error shapes (ConvexError with `code`, `message`, `httpStatus`) are unchanged. `requireAuth` still throws `UNAUTHENTICATED`/401, `requireAdmin` still throws `FORBIDDEN`/403, and `users.me` still returns `null` for unauthenticated users.

## Edge cases

### `getAuthUserId` returns `null`
This replaces the existing `identity === null` check. `getAuthUserId` returns `null` when `ctx.auth.getUserIdentity()` returns `null`. The existing error-throwing behavior for `requireAuth`/`requireAdmin` and null-returning behavior for `users.me` is preserved.

### User ID from `getAuthUserId` has no matching db row
This is the "orphaned identity" case — the auth layer knows about a user ID that doesn't exist in the `users` table. `requireAuth` and `requireAdmin` already handle this by checking `if (!user)` after `db.get()`. This behavior is preserved.

### Existing tests use `withIdentity({ subject: userId })`
The existing WOR-163 tests pass a raw userId as `subject` in `convexTest.withIdentity()`. In the real runtime, `identity.subject` is composite (`userId|sessionId`). The tests should be updated to pass composite subjects (e.g., `subject: \`${userId}|someSessionId\``) to exercise the split logic. Tests that pass a raw ID would still work with `getAuthUserId` (splitting on `|` yields the full string if no `|` exists), but composite subjects are more realistic.

## Non-goals

- **Changing `requirePartyToCase`**: It receives `userId` as a parameter and never calls `getUserIdentity`. Not affected by this bug.
- **Changing `convex/auth.ts` `createOrUpdateUser`**: The callback correctly returns `user._id`. The bug is solely in the consumers that read `identity.subject` without splitting.
- **Changing `getUserByEmail`**: Unrelated to identity subject resolution.
- **Adding new indexes or queries**: `db.get()` by ID is the correct O(1) lookup; no index changes needed.

## Test coverage

All ACs are covered by **unit tests** in `tests/unit/auth.test.ts`. This file already exists from WOR-163 with test cases for `requireAuth`, `requireAdmin`, `getUserByEmail`, and `requirePartyToCase`.

- **AC1–AC3** (getAuthUserId in requireAuth, requireAdmin, users.me): Tests should use `withIdentity({ subject: \`${userId}|${sessionId}\` })` to provide a composite subject, verifying that `getAuthUserId` correctly extracts the userId. The existing tests that pass raw `subject: userId` should be updated to use the composite format.

- **AC4** (composite subject handling): Covered by the same tests — if the composite subject is correctly split, `db.get()` succeeds and returns the user doc. If not, the test would get "Invalid ID length" instead of a user doc.

- **AC5** (no regression): The existing "no identity" and "orphaned identity" tests from WOR-163 cover this. They verify that `requireAuth` throws UNAUTHENTICATED/401 and `requireAdmin` throws FORBIDDEN/403.

Unit-only coverage is appropriate because the bug is entirely in server-side ID parsing logic — no browser interaction, no UI rendering, no cross-service calls. The `convex-test` harness with `withIdentity` provides sufficient fidelity.
