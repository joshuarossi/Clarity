# Contract: WOR-125 â€” Joint chat Coach AI action â€” Haiku gate + Sonnet generation with streaming

## Why this work exists

When a user sends a message in joint chat, the system must decide whether the Coach AI should respond (classification) and, if so, generate a contextually appropriate facilitative response streamed in real time. The two-step Haiku-then-Sonnet architecture keeps costs low for routine exchanges while ensuring the Coach speaks up for inflammatory remarks, meaningful progress, or direct questions. This is the core intelligence behind the Coach's participation in the shared conversation.

## Files and exports

### `convex/jointChat.ts` (modify)

This file already contains the queries, mutations, and a placeholder `generateCoachResponse` action (lines 312-321). The implementation replaces the placeholder with the full two-step pipeline and adds the supporting internal mutations and queries it needs.

**Why all new functions live in this file:** The existing `privateCoaching.ts` follows the same pattern â€” the AI action and its supporting internal mutations/queries (`insertStreamingMessage`, `updateStreamingMessage`, `finalizeStreamingMessage`, `markMessageError`) all live together in one module. Joint chat follows the same convention. The internal mutations and queries are prefixed with "Coach" to disambiguate from the private coaching equivalents in the other module.

**Key internal queries (data-fetching for the action):**

- `getCaseForCoach` â€” reads the case doc to validate JOINT_ACTIVE status and get the `templateVersionId`.
- `getPartyStatesForCoach` â€” reads both party states to extract `synthesisText` for each party.
- `getJointMessagesForCoach` â€” reads all joint chat messages (for prompt context + exchange counting for the timer override).
- `getPrivateMessagesForCoach` â€” reads BOTH parties' raw private messages. These are used ONLY by the privacy response filter; they are never passed to the LLM prompt.
- `getTemplateVersionForCoach` â€” reads the template version pinned to the case for template-specific Sonnet prompt instructions.

**Key internal mutations (streaming lifecycle):**

- `insertCoachStreamingMessage` â€” creates a jointMessages row with `{ authorType: "COACH", status: "STREAMING", content: "", isIntervention }`. Returns the message ID.
- `updateCoachStreamingMessage` â€” patches `content` on the row during streaming (~50ms flush interval).
- `finalizeCoachMessage` â€” sets `status: "COMPLETE"` with final content.
- `markCoachMessageError` â€” sets `status: "ERROR"`.

**The `generateCoachResponse` action pipeline:**

1. Read case via `getCaseForCoach` â†’ validate `status === "JOINT_ACTIVE"`, else throw CONFLICT.
2. Read party states via `getPartyStatesForCoach` â†’ extract both `synthesisText` values.
3. Read joint messages via `getJointMessagesForCoach` â†’ use for prompt context and exchange counting.
4. Read template version via `getTemplateVersionForCoach`.
5. **Step 1 â€” Haiku classification:** Call `claude-haiku-4-5-20251001` with the last user message + minimal context. Parse the response into one of: `INFLAMMATORY`, `PROGRESS`, `QUESTION_TO_COACH`, `NORMAL_EXCHANGE`.
6. **Gate check:** If classification is `NORMAL_EXCHANGE` AND `triggerType` is not `"mention"` AND the timer override condition is not met (fewer than 5 consecutive user exchanges since last Coach message), return early â€” no Coach response.
7. **Step 2 â€” Sonnet generation:** Call `assemblePrompt({ role: "COACH", ... })` with joint chat history + both synthesis texts. Use `claude-sonnet-4-5` for streaming generation.
8. Insert STREAMING row via `insertCoachStreamingMessage` (with `isIntervention: true` if classification was INFLAMMATORY).
9. Stream tokens, flushing via `updateCoachStreamingMessage` every ~50ms.
10. After streaming completes, read BOTH parties' raw private messages via `getPrivateMessagesForCoach`.
11. Apply `filterResponse(finalContent, allRawPrivateUserMessages)`.
12. If filter fails: discard content, retry from Step 7 (up to 2 retries, 3 total).
13. On 3rd filter failure: finalize with hardcoded fallback `"I'm having trouble responding to that right now. Could either of you rephrase?"`.
14. On success: call `finalizeCoachMessage`.
15. On API error: follow Â§6.5 retry pattern (429 â†’ retry with 2s exponential backoff; other errors â†’ retry once with 2s delay; after exhaustion â†’ `markCoachMessageError`).

## Data dependencies

### `internal.jointChat.getCaseForCoach`

Returns the `cases` doc. Fields consumed: `status` (must be `JOINT_ACTIVE`), `templateVersionId` (to load template), `isSolo` (for party state lookup).

### `internal.jointChat.getPartyStatesForCoach`

Returns both `partyStates` docs for the case. Fields consumed: `synthesisText` (per party, for prompt context), `userId` (for identifying parties), `role` (INITIATOR/INVITEE).

### `internal.jointChat.getJointMessagesForCoach`

Returns all `jointMessages` for the case, sorted by `createdAt`. Fields consumed: `authorType` (USER vs COACH â€” for exchange counting), `content` (for prompt context), `authorUserId` (for message attribution in prompt).

### `internal.jointChat.getPrivateMessagesForCoach`

Returns all `privateMessages` for the case (both parties). Fields consumed: `content` (USER-role messages only, for privacy filter input), `role` (to filter to USER messages only), `userId`/`partyRole` (to group by party).

### `internal.jointChat.getTemplateVersionForCoach`

Returns the `templateVersions` doc pinned to the case. Fields consumed: `globalGuidance`, `coachInstructions` â€” passed to `assemblePrompt` as `templateVersion`.

### `assemblePrompt` (from `convex/lib/prompts.ts`)

Called with `role: "COACH"`. Consumes `context.actingPartySynthesis`, `context.otherPartySynthesis`, `context.jointChatHistory`. Does NOT receive any private messages.

### `filterResponse` (from `convex/lib/privacyFilter.ts`)

Called with the final Coach response text and an array of all USER-role private message content strings from both parties. Returns `{ passed: boolean, matchedSubstring?: string }`.

### `getMockClaudeResponse` / `isClaudeMockEnabled` (from `convex/lib/claudeMock.ts`)

Used in mock mode (CLAUDE_MOCK=true). `getMockClaudeResponse("COACH")` returns the mock Coach response text.

## Invariants

### Raw private messages never enter the prompt

The action fetches raw private messages via `getPrivateMessagesForCoach`, but these are used ONLY as input to `filterResponse()`. The `assemblePrompt` call receives `context.actingPartySynthesis` and `context.otherPartySynthesis` (derived synthesis texts) and `context.jointChatHistory`, never raw private messages. This is the privacy boundary: the LLM sees synthesized themes, not anyone's raw words.

### Case must be JOINT_ACTIVE

The action validates `caseDoc.status === "JOINT_ACTIVE"` before any AI call. If the case has moved to a closed state, the action throws a CONFLICT error. This prevents stale scheduled actions from generating Coach messages after closure.

### Classification gate controls Coach verbosity

The Coach stays silent on `NORMAL_EXCHANGE` to avoid being intrusive. The two overrides are: (1) `triggerType === "mention"` â€” the user explicitly asked for Coach input, so always respond; (2) timer override â€” if 5+ consecutive user messages have been sent with no Coach message, the Coach speaks up to stay engaged.

### Privacy filter with retry

The privacy filter (`filterResponse`) checks for â‰Ą8 consecutive token matches between the Coach output and either party's raw private messages. On failure, the entire Sonnet generation is retried (not just a filter re-check). After 3 total failures, a hardcoded safe fallback is emitted. This prevents the Coach from accidentally quoting private content.

### INFLAMMATORY â†’ isIntervention

When Haiku classifies a message as INFLAMMATORY, the Coach message row is created with `isIntervention: true`. The UI uses this flag to render the message with distinct styling (4px left border per StyleGuide Â§6.4). This flag is set at insert time and never changes.

## Edge cases

### Haiku classification returns unexpected output

If Haiku returns a value that doesn't match any of the four expected classifications, treat it as `NORMAL_EXCHANGE` (safe default â€” Coach stays silent rather than speaking inappropriately).

### Mock mode behavior

When `CLAUDE_MOCK=true`, the Haiku classification step returns a deterministic classification (e.g., always `QUESTION_TO_COACH` to ensure the Sonnet step fires). The Sonnet step returns `getMockClaudeResponse("COACH")`. Mock failure simulation uses `CLAUDE_MOCK_FAIL_COUNT` and `CLAUDE_MOCK_FAIL_STATUS` env vars, matching the existing pattern from `privateCoaching.ts`.

### API error during Haiku step

If the Haiku classification call fails, the action should NOT insert a STREAMING row. It should retry the Haiku call once (same retry pattern as Sonnet), and if it still fails, mark the triggering user message's associated Coach response as ERROR or simply return without generating a response.

### API error during Sonnet step

If the Sonnet call fails after the STREAMING row has been inserted, apply the standard retry pattern (429 â†’ exponential backoff, other â†’ 2s delay, max 2 attempts). On exhaustion, call `markCoachMessageError` to set the row to ERROR status.

### Empty synthesis texts

If one or both parties have no `synthesisText` (should not happen in JOINT_ACTIVE, but defensively), proceed with whatever context is available. The `assemblePrompt` function already handles optional synthesis fields.

### Concurrent Coach responses

The `sendUserMessage` mutation schedules `generateCoachResponse` for each user message. If two messages arrive in quick succession, two action invocations may run concurrently. Each inserts its own STREAMING row. This is acceptable â€” the Coach may respond to each if classification warrants it.

## Non-goals

- **No UI changes.** This ticket is backend-only. The jointMessages table already has all the fields the UI needs (`authorType`, `status`, `isIntervention`, `content`). The UI (WOR-126 or similar) will consume these reactively.
- **No schema changes.** The `jointMessages` table schema already supports `authorType: "COACH"`, `status: "STREAMING" | "COMPLETE" | "ERROR"`, and `isIntervention: boolean`. No migration needed.
- **No transcript compression.** The TechSpec mentions transcript compression (Â§6.4) for contexts exceeding budget. That's a separate optimization; this ticket assembles context directly.
- **No Draft Coach.** Draft Coach (US-12) is a separate feature with its own action and message table.
- **No opening message.** The Coach's opening message when a joint session begins is a separate concern, likely triggered by `enterSession`.
