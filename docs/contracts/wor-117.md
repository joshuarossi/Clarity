# Contract: WOR-117 — Private coaching Convex module — queries + mutations

## Why this work exists

Private coaching is Clarity's core privacy-enforced interaction: each party has a confidential AI-guided conversation that the other party can never see. This module provides the reactive query and mutations that drive the private coaching phase — message retrieval with strict caller-only visibility, message insertion with AI scheduling, and the completion handshake that triggers synthesis when both parties finish. Without it, there is no private coaching flow.

## Files and exports

### `convex/privateCoaching.ts` (create, mutation-and-query-module)

A new Convex module following the established convention in `convex/cases.ts` and `convex/invites.ts` — one file per domain, top-level named exports of `query`, `mutation`, and `internalAction` instances. Imports `requireAuth` and `requirePartyToCase` from `convex/lib/auth.ts`, and error helpers from `convex/lib/errors.ts`.

**`myMessages`** — A `query` that returns all `privateMessages` rows for a given `caseId` where `userId` matches the authenticated caller. It uses the `by_case_and_user` index with `[caseId, caller._id]` to filter at the database level. Results are sorted by `createdAt` ascending (the natural order from the index query; if the index does not guarantee order, an explicit `.sort()` is applied). This query calls `requireAuth` but deliberately does NOT call `requirePartyToCase` — a non-party caller simply gets an empty array, per AC 8's privacy requirement to avoid leaking whether messages exist.

**`sendUserMessage`** — A `mutation` that:
1. Calls `requireAuth` to get the caller.
2. Calls `requirePartyToCase` to verify the caller is a party to the case.
3. Reads the case document and checks `status` is `"DRAFT_PRIVATE_COACHING"` or `"BOTH_PRIVATE_COACHING"`. If not, throws `ConvexError({ code: "CONFLICT", message: "Case is not in private coaching phase", httpStatus: 409 })`.
4. Inserts a `privateMessages` row: `{ caseId, userId: caller._id, role: "USER", content: args.content, status: "COMPLETE", createdAt: Date.now() }`.
5. Schedules the AI response: `ctx.scheduler.runAfter(0, internal.privateCoaching.generateAIResponse, { caseId: args.caseId, userId: caller._id })`.
6. Returns the inserted message's `Id<"privateMessages">`.

**`markComplete`** — A `mutation` that:
1. Calls `requireAuth` and `requirePartyToCase`.
2. Looks up the caller's `partyStates` row via the `by_case_and_user` index.
3. If `privateCoachingCompletedAt` is already set, returns `{ synthesisScheduled: false }` immediately (idempotency — no error, no side effects).
4. Patches the caller's `partyStates` row with `privateCoachingCompletedAt: Date.now()`.
5. Loads both `partyStates` rows for the case via the `by_case` index. If both have non-null `privateCoachingCompletedAt`, schedules `ctx.scheduler.runAfter(0, internal.synthesis.generate, { caseId })` and returns `{ synthesisScheduled: true }`.
6. Otherwise returns `{ synthesisScheduled: false }`.

**`generateAIResponse`** — An `internalAction` (not client-accessible) that receives `{ caseId, userId }`. In this ticket it is a minimal stub — it may insert a placeholder AI message or simply return. The real AI logic is a separate ticket. This export exists so that `sendUserMessage`'s `ctx.scheduler.runAfter` call has a valid target at the `internal.privateCoaching.generateAIResponse` path.

### `tests/unit/privateCoaching.test.ts` (create, test-infrastructure)

Integration tests using `convex-test` following the established pattern in `tests/unit/cases.test.ts`. All tests import the schema from `../../convex/schema` and use `convexTest(schema)` with `withIdentity` for auth simulation. The `expectConvexError` helper pattern from `tests/unit/cases.test.ts` should be reused or copied. Test fixtures seed two users, a case in `BOTH_PRIVATE_COACHING` status with two `partyStates` rows, and pre-seeded `privateMessages` for isolation tests.

## Data dependencies

This module reads and writes the following tables:

- **`privateMessages`** — the primary data this module manages. `myMessages` reads via the `by_case_and_user` index. `sendUserMessage` inserts rows. Schema: `{ caseId: Id<"cases">, userId: Id<"users">, role: "USER" | "AI", content: string, status: "STREAMING" | "COMPLETE" | "ERROR", tokens?: number, createdAt: number }`.

- **`cases`** — read by `sendUserMessage` to check `status`. Read by `requirePartyToCase` to verify party membership. Not modified by this module (status transitions are handled by the synthesis module).

- **`partyStates`** — read and written by `markComplete`. The `privateCoachingCompletedAt` field (optional number) is the key field. Read via `by_case_and_user` index for the caller's row, and via `by_case` index to check both parties' completion status.

- **`users`** — read indirectly via `requireAuth`.

Key fields that flow into rendered UI (relevant for test alignment):
- `privateMessages.content` — the message text displayed in the coaching chat.
- `privateMessages.role` — determines whether the message is shown as user-authored or AI-authored.
- `privateMessages.createdAt` — determines sort order in the chat.
- `partyStates.privateCoachingCompletedAt` — drives the "both completed" check and is surfaced to the other party as `hasCompletedPC` via `cases/partyStates` (from WOR-111).

## Invariants

### Caller-only visibility at the database level

The `myMessages` query uses the `by_case_and_user` index with the caller's `userId` as a fixed parameter. This means the database query itself can never return rows belonging to another user. This is the privacy invariant — enforcement happens in the index, not in application-level filtering. The `by_case` index on `privateMessages` must never appear in any client-facing query; it exists solely for server-side AI context assembly (e.g., the `generateAIResponse` action loading conversation history).

### Empty array for non-parties (not FORBIDDEN)

Unlike `sendUserMessage` and `markComplete` (which call `requirePartyToCase` and throw `FORBIDDEN`), the `myMessages` query deliberately omits the `requirePartyToCase` check. A non-party caller who somehow knows a `caseId` will receive an empty array — the `by_case_and_user` index query with their `userId` simply matches no rows. This avoids information leakage about whether messages exist for that case. `requireAuth` is still enforced — unauthenticated callers get `UNAUTHENTICATED`.

### Case status gating on sendUserMessage

`sendUserMessage` reads the case document and asserts `status` is one of `["DRAFT_PRIVATE_COACHING", "BOTH_PRIVATE_COACHING"]`. Any other status (including `READY_FOR_JOINT`, `JOINT_ACTIVE`, `CLOSED_*`) results in a `CONFLICT` error. This prevents messages from being added after the coaching phase has ended.

### Idempotent markComplete

`markComplete` must be safe to call multiple times by the same user. On the second (and subsequent) calls, it detects that `privateCoachingCompletedAt` is already set and returns early without patching the row or scheduling synthesis. The return value `{ synthesisScheduled: false }` distinguishes this from the first-call path. This prevents double-scheduling of the synthesis action.

### Synthesis scheduling guard

Synthesis is scheduled if and only if both `partyStates` rows for the case have non-null `privateCoachingCompletedAt` at the moment `markComplete` runs. Because `markComplete` is idempotent (early-returns if already complete), and because the "check both parties" logic runs atomically within the Convex mutation transaction, synthesis is scheduled exactly once — on whichever party completes second.

## Edge cases

### Loading state

`myMessages` returns an empty array when the caller has no messages yet (e.g., the coaching phase just started). The UI should show an empty state or welcome message — this is not an error.

### No messages for a valid party

A party who is part of the case but has sent no messages and received no AI responses gets `[]` from `myMessages`. This is distinct from the non-party empty-array case in semantics but identical in shape.

### Case in DRAFT_PRIVATE_COACHING (only initiator)

When the case is in `DRAFT_PRIVATE_COACHING`, only the initiator exists as a party (invitee hasn't joined yet). The initiator can call `sendUserMessage` and `myMessages`. `markComplete` can technically be called but `both parties completed` will never be true with only one `partyStates` row.

### markComplete before any messages

Nothing in the spec prevents a party from marking coaching complete without having sent any messages. The mutation does not validate message count. This is intentional — the user may have had a sufficient coaching session or may want to skip ahead.

### generateAIResponse target doesn't exist yet

The `generateAIResponse` internal action must exist in `convex/privateCoaching.ts` for `ctx.scheduler.runAfter` to have a valid reference at `internal.privateCoaching.generateAIResponse`. Even as a stub, it must be declared. If it were missing, the mutation would fail at scheduling time.

### Synthesis module not yet deployed

`markComplete` references `internal.synthesis.generate`, which may not exist when this module is first deployed. The implementation should handle this gracefully — either by importing the reference conditionally or by acknowledging that the synthesis module must be deployed before `markComplete`'s both-complete path works end-to-end. For testing purposes, `convex-test` can verify that `ctx.scheduler.runAfter` was called with the expected arguments without the target action existing.

## Non-goals

- **AI response generation** — The `generateAIResponse` action is a stub in this ticket. The actual Claude API integration, prompt assembly, streaming, and error retry logic are separate tickets that build on the prompt and privacy-filter helpers already in `convex/lib/`.
- **Synthesis generation** — `markComplete` schedules `internal.synthesis.generate` but does not implement it. The synthesis module is a separate ticket.
- **Case status transitions** — This module does not call `validateTransition` or change `cases.status`. The `COMPLETE_COACHING` transition (`BOTH_PRIVATE_COACHING` → `READY_FOR_JOINT`) happens inside the synthesis module after synthesis completes, not here.
- **UI components** — No React pages or components. The private coaching chat UI is a separate frontend ticket.
- **Message streaming** — `sendUserMessage` inserts with `status: "COMPLETE"`. Real-time streaming of AI responses is an AI integration concern.
- **Message editing or deletion** — Not in scope for v1.
