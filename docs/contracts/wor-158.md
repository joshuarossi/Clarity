# Contract: WOR-158 — Fix CI seed step to use `--preview-run` instead of standalone `convex run`

## Why this work exists

The CI e2e job deploys Convex functions to an ephemeral preview deployment
using a preview deploy key (`CONVEX_DEPLOY_KEY_CI`), then tries to seed
test data with a separate `npx convex run seed:seed` step. This fails
with "Unauthorized" because `convex run` against an `internalMutation`
requires the deployment context that only `convex deploy --preview-run`
provides. The production deploy workflow (`deploy.yml`) already uses the
correct `convex deploy --cmd` pattern. The CI workflow just needs to
fold the seed invocation into the deploy step via `--preview-run`.

## Files and exports

### `.github/workflows/ci.yml` (modify)

The e2e job currently has two separate steps (lines 84–92):

```yaml
- name: Deploy Convex functions
  run: npx convex deploy --cmd 'echo "deployed"'
  env:
    CONVEX_DEPLOY_KEY: ${{ secrets.CONVEX_DEPLOY_KEY_CI }}

- name: Seed test data
  run: npx convex run seed:seed
  env:
    CONVEX_DEPLOY_KEY: ${{ secrets.CONVEX_DEPLOY_KEY_CI }}
```

Replace with a single step that combines deploy and seed:

```yaml
- name: Deploy Convex functions and seed test data
  run: npx convex deploy --preview-run seed:seed --cmd 'echo "deployed"'
  env:
    CONVEX_DEPLOY_KEY: ${{ secrets.CONVEX_DEPLOY_KEY_CI }}
```

The `--preview-run seed:seed` flag tells `convex deploy` to run the
`seed:seed` internal mutation after the deployment succeeds, within the
authorized deployment context. This is the Convex-documented pattern for
running internal functions on preview deployments.

## Data dependencies

No Convex queries, mutations, or actions are added or modified. The
existing `seed:seed` internal mutation (`convex/seed.ts`) is unchanged;
only the CI invocation method changes.

## Invariants

1. **No standalone `convex run` in CI:** The e2e job must not use
   `npx convex run` as a separate step. Preview deploy keys authorize
   function execution only through `convex deploy --preview-run`, not
   through standalone `convex run`.

2. **Seed before Playwright:** The seed function must complete before
   `npx playwright test` runs. Since `--preview-run` executes
   synchronously as part of the `convex deploy` step, this ordering
   is preserved.

3. **Deploy key unchanged:** `CONVEX_DEPLOY_KEY_CI` remains the only
   secret used. No new secrets are introduced.

4. **No-op frontend build preserved:** `--cmd 'echo "deployed"'` is
   retained so `convex deploy` does not attempt to build a frontend
   (CI doesn't need the frontend built at this step; Playwright handles
   its own dev server).

## Edge cases

- **Seed idempotency:** The seed function is already idempotent (checks
  for existing records before inserting). No change needed.

- **Deploy failure:** If `convex deploy` fails, `--preview-run` does not
  execute, and the Playwright step is skipped (sequential step execution).
  This is the desired behavior.

- **Preview-run failure:** If `--preview-run seed:seed` fails after a
  successful deploy, the step exits non-zero and Playwright is skipped.
  The upload-artifact step (which runs `if: failure()`) will still
  capture any partial reports.

## Non-goals

- **No changes to `convex/seed.ts`:** The seed function itself is correct.
  The bug is purely in how CI invokes it.
- **No changes to `deploy.yml`:** The production deploy workflow already
  uses the correct `convex deploy --cmd` pattern and does not run seed.
- **No changes to other CI jobs:** lint, typecheck, and unit jobs are
  unaffected.
- **No new CI steps or environment variables.**
