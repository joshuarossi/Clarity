# Contract: WOR-113 — NewCasePage — case creation form with progressive disclosure

## Why this work exists

The case creation form is the first substantive user interaction after login. An initiator needs to describe their conflict — category, topic, private situation details, desired outcome, and the other party's name — in a way that feels conversational rather than bureaucratic. This page calls the existing `cases/create` mutation (WOR-111) and routes to the invite sharing page so the initiator can send the link. Without this page, there is no way for users to create cases through the UI.

## Files and exports

### `src/routes/NewCasePage.tsx` (create)

A connected page component following the project's page convention (see `src/routes/LoginPage.tsx`, `src/routes/ProfilePage.tsx`). The component is **connected** — it calls `useMutation(api.cases.create)` and `useNavigate()` internally. The test author should render it wrapped in a mocked Convex provider and a `MemoryRouter`.

The component is exported as a named export `NewCasePage` (matching the existing stub name in `App.tsx`). There is no separate presentational export — the form's layout, validation, and submission are tightly coupled, and splitting them would add complexity without test benefit. Tests verify behavior by interacting with the rendered DOM (clicking radio cards, typing in fields, submitting) and asserting on mutation calls and navigation.

The form renders as a single scrollable column with all fields visible (not a multi-step wizard). The "progressive disclosure" per DesignDoc §4.5 refers specifically to the solo mode toggle being hidden under an expandable "Advanced" section.

### `src/App.tsx` (modify)

Replace the inline `NewCasePage` stub function with an import: `import { NewCasePage } from "./routes/NewCasePage";`. Remove the stub. No other changes.

### `tests/unit/new-case-page.test.tsx` (create)

Unit and integration tests using Vitest + `@testing-library/react` + `jsdom` environment, following the pattern of `tests/unit/login-page.test.tsx` and `tests/unit/chat-window.test.tsx`. Uses `axe-core` for accessibility checks (pattern from `tests/unit/primitives-a11y.test.tsx`). Mocks Convex mutations and React Router navigation.

### `e2e/new-case.spec.ts` (create)

Playwright E2E tests using the project's custom fixtures from `e2e/fixtures.ts` (`authenticatedPage`). Tests the full form submission flow against the real Convex backend (or Claude mock mode in CI). Verifies keyboard navigation and tab order in a real browser.

## Data dependencies

### `api.cases.create` (mutation, called on submit)

**Args:**
```typescript
{
  category: string,        // one of "workplace" | "family" | "personal" | "contractual" | "other"
  mainTopic: string,       // free text, soft 140-char limit
  description: string,     // free text, multi-paragraph
  desiredOutcome: string,  // free text
  templateId?: Id<"templates">,  // NOT sent by this page — server resolves from category
  isSolo?: boolean,        // from the Advanced disclosure checkbox
}
```

**Returns:**
```typescript
{ caseId: Id<"cases">, inviteUrl: string | null }
```

`inviteUrl` is `null` when `isSolo` is true; otherwise a URL like `http://localhost:5173/invite/<token>`.

**Key field flows:**
- `category` → form radio card selection → mutation arg
- `mainTopic` → text input value → mutation arg. Rendered in helper text: "This will be visible to the other person when they accept the invitation. Keep it factual, not emotional."
- `description` → textarea value → mutation arg. Labelled "Private to you" with lock icon.
- `desiredOutcome` → textarea value → mutation arg. Labelled "Private to you" with lock icon.
- `otherPartyName` → text input value → NOT sent to mutation → passed via `navigate(path, { state: { otherPartyName } })` to the invite page for cosmetic display ("Send this link to [name]").
- `isSolo` → checkbox value → mutation arg (only when checked).

## Invariants

### Radio cards, not dropdown

The five categories are rendered as radio cards — large clickable rectangular areas, each containing a hidden `<input type="radio">` and a visible `<label>`. They must NOT be rendered as a `<select>` element. The selected card has distinct visual styling (e.g., a border color change or background highlight). Each card's `<input>` has `name="category"` and a unique `value` matching the category key.

### Soft character limit on main topic

The character counter displays `${mainTopic.length}/140` below the input. When `mainTopic.length <= 140`, the counter uses neutral/default text color. When `mainTopic.length > 140`, the counter text turns to a warning color (amber/red via a CSS class like `cc-text-warning` or `text-amber-600`). The input does NOT have a `maxLength` attribute — users can freely type past 140 characters.

### Privacy lock icon with tooltip

Both the description and desired outcome fields show a lock icon (from `lucide-react`, e.g., `<Lock />`) adjacent to their "Private to you" label. Each lock icon is wrapped in an interactive element (button or span with `tabIndex={0}`) that shows a tooltip on hover and focus with the text: "Only you and the AI coach will see this." The tooltip must be accessible (e.g., via `aria-describedby` pointing to a tooltip element, or via a `title` attribute as a fallback).

### otherPartyName is client-only

The `cases/create` mutation does not accept an `otherPartyName` argument. The form field collects it for the post-create invite page. On successful submit, the component calls `navigate(\`/cases/${caseId}/invite\`, { state: { otherPartyName } })`. The invite page (T20, out of scope) reads it from `useLocation().state`.

### Client-side validation

On submit attempt, the form validates:
- `category` — must not be empty string (required)
- `mainTopic` — must not be empty/whitespace-only (required)
- `description` — must not be empty/whitespace-only (required)
- `desiredOutcome` — not required
- `otherPartyName` — not required

Validation errors appear as inline text below each invalid field with `role="alert"`. The submit button is NOT disabled — validation runs on submit attempt. If validation fails, the form scrolls to / focuses the first invalid field.

### Advanced disclosure default state

The "Advanced" section is collapsed by default (solo checkbox not visible). It uses an HTML `<details>/<summary>` element or equivalent expandable pattern. When expanded, it reveals a single checkbox: "Create this as a solo test case (I'll play both parties)".

### Accessibility

All `<input>` and `<textarea>` elements have associated `<label>` elements via `htmlFor`/`id` pairing. All interactive elements (radio cards, text inputs, textareas, checkbox, submit button, lock icon tooltips) are keyboard-reachable via Tab and operable via Enter/Space. The form meets WCAG AA contrast ratios (4.5:1 for body text, 3:1 for large text).

## Edge cases

### Loading state during submission

While the `cases/create` mutation is in flight, the submit button shows a loading state (disabled + spinner or "Creating..." text) to prevent double submission. The form fields remain visible but should not be re-submittable.

### Mutation error

If `cases/create` throws (e.g., no template found for category), the error is displayed as a banner or inline message near the submit button. The form remains filled so the user can retry. The error message should be user-friendly (not a raw ConvexError dump).

### Empty state

The page loads with all fields empty and no validation errors shown. The radio cards have no selection. The character counter shows "0/140".

### Auto-growing textarea for description

The description textarea starts at 5 rows (`rows={5}`). As the user types content that exceeds the visible area, the textarea grows in height to fit the content (via a `resize: none` style + JS-driven height adjustment on input, or CSS `field-sizing: content` where supported). It does not shrink below 5 rows.

### Desired outcome textarea

The desired outcome textarea is 3 rows (`rows={3}`). It does NOT auto-grow (simpler, shorter field).

## Non-goals

- **No template selection UI** — the server picks the template from the category automatically. There is no template picker, dropdown, or advanced template selection flow in this form.
- **No autosave or draft persistence** — form state lives in React `useState`. Navigating away loses all input. Draft saving is not in scope for v1.
- **No invite sharing page** — this ticket creates the case and routes to `/cases/:caseId/invite`. Building that page is T20's responsibility.
- **No other-party-name storage** — `otherPartyName` is a cosmetic client-side field passed via router state. It is not persisted in the database.
- **No step-by-step wizard** — despite the "progressive disclosure" language in the ticket title, DesignDoc §4.5 specifies a single-column layout where all steps are visible. The only hidden element is the solo toggle under "Advanced". Do not build a multi-step wizard.
