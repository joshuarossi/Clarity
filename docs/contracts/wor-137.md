# Contract: WOR-137 — Landing page

## Why this work exists

Clarity needs a public-facing landing page that communicates the product's value proposition in under 15 seconds and funnels visitors toward registration. Currently the `/` route renders a bare stub. The landing page is the first impression for prospective users and must convey calm, safety, and clarity of purpose per the DesignDoc §4.1 spec.

## Files and exports

### `src/routes/LandingPage.tsx` (create, connected)

The landing page is a **connected component** because it calls `useConvexAuth()` to detect authentication state and redirects authenticated users to `/dashboard` via `<Navigate>`. This is the same pattern used by `LoginPage` at line 71 of `App.tsx`. The component is self-contained — no props required.

Internally it renders four semantic sections:

1. **Hero** (`<section>`) — tagline, subhead, primary CTA button linking to `/login`
2. **Three-step explainer** (`<section>`) — three items with Lucide icons: MessageCircle (Private Coaching), Users (Shared Conversation), ShieldCheck (Resolution)
3. **Privacy section** (`<section>`) — heading text with a link to `/privacy` (or `#` placeholder if no privacy page exists yet)
4. **Footer** (`<footer>`) — links for terms, privacy, contact

Typography follows DesignDoc §2.2 and the implementation notes: hero headline uses display type scale (32px mobile / 40px desktop, weight 500, -0.02em tracking). Spacing uses the 8-point grid. Color tokens from the style guide apply (`--bg-canvas`, `--accent` for the CTA, `--text-primary`/`--text-secondary`).

### `src/App.tsx` (modify)

Remove the inline `HomePage` stub function. Import `LandingPage` from `./routes/LandingPage`. Replace the `/` route element:

```diff
- <Route path="/" element={<HomePage />} />
+ <Route path="/" element={<LandingPage />} />
```

The auth-redirect logic lives inside `LandingPage` itself (not in App.tsx), matching the pattern already established for the `/login` route.

## Data dependencies

None. The landing page is fully static. The only hook it calls is `useConvexAuth()` from `@convex-dev/auth/react` to check `isAuthenticated` and `isLoading` for the redirect guard. No Convex queries, mutations, or actions are invoked.

## Invariants

### Authenticated users never see this page

If `useConvexAuth()` returns `{ isAuthenticated: true, isLoading: false }`, the component renders `<Navigate to="/dashboard" replace />` and nothing else. This mirrors the existing pattern on the `/login` route.

### No forbidden content

The page must never contain: testimonials, pricing information, "Get started free" text, or any CTA other than the single "Start a case" button. This is a DesignDoc §4.1 hard constraint.

### Static — no data fetching

The page calls no Convex queries. It renders immediately with no loading state for data (though there is a brief moment while `isLoading` is true from the auth check — during this time, render nothing or a minimal shell to avoid flash).

### CTA routes to /login

The "Start a case" button is a `<Link to="/login">` (React Router), not an `<a href>`. It does not trigger any auth action directly.

## Edge cases

### Loading state (auth check in progress)

While `useConvexAuth()` returns `{ isLoading: true }`, render nothing (return `null`). This avoids a flash of the landing page for already-authenticated users. The auth check resolves quickly (sub-100ms for cached sessions).

### No privacy policy page yet

The privacy section links to a privacy policy. If no `/privacy` route exists in v1, use `/privacy` as the href anyway — it will 404 to the NotFoundPage, which is acceptable for now. Do not use `#` or omit the link.

### Very narrow viewports

At viewports below 375px (minimum target per NFR-BROWSER), the three-step explainer stacks vertically. No horizontal overflow is acceptable at any supported viewport width.

## Non-goals

- **No about page** — deferred to v1.1 per site map.
- **No dark mode specific work** — dark mode is handled app-wide via CSS custom properties; the landing page just uses the tokens.
- **No SEO/meta** — SPA with client-side rendering; no SSR or meta tag management in v1.
- **No analytics or tracking** — out of scope for v1.
- **No i18n** — English only for v1.
