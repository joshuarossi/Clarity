# Contract: WOR-132 — Admin Convex module — template CRUD, versioning, audit log writes

## Why this work exists

Clarity's admin users need to manage coaching templates without deploying code. Templates drive all AI coaching behavior, so the CRUD layer must enforce immutable versioning: existing cases pinned to an older template version must never be disrupted when a new version is published or a template is archived. This module provides the five server-side functions (two queries, three mutations) that the admin UI pages will consume, plus the audit trail that makes template lifecycle changes traceable.

## Files and exports

### `convex/admin.ts` (create)

A new Convex module file following the project convention: domain logic as a top-level file under `convex/`, with named `query`/`mutation` exports. The Convex runtime maps these to API paths like `api.admin.listAll`, `api.admin.create`, etc. This matches the TechSpec §7.1-7.2 naming convention `admin/templates/*` — since Convex maps file-level exports to `api.<filename>.<export>`, a single `convex/admin.ts` file with exports named `listAll`, `listVersions`, `create`, `publishNewVersion`, `archive` produces paths like `api.admin.listAll` (equivalent to `admin/listAll`). The TechSpec's conceptual grouping `admin/templates/listAll` and `admin/templateVersions/list` maps to flat exports in one file because all five functions operate on the same domain and share the `requireAdmin` guard.

**`listAll`** — Query returning all rows from the `templates` table, including archived ones. Calls `requireAdmin` first. Returns the full `Doc<"templates">[]` array. No filtering, no sorting requirement (caller can sort client-side), though returning in insertion order is natural.

**`listVersions`** — Query returning all `templateVersions` rows for a given `templateId`, sorted by `version` descending. Calls `requireAdmin` first. Uses the `by_template` index to fetch, then sorts in-memory by `version` desc. Returns `Doc<"templateVersions">[]`.

**`create`** — Mutation that:

1. Calls `requireAdmin` to get the admin user doc.
2. Inserts a `templates` row with `{ category, name, createdAt: Date.now(), createdByUserId: admin._id }`. `currentVersionId` is initially unset.
3. Inserts a `templateVersions` row with `{ templateId, version: 1, globalGuidance, coachInstructions, draftCoachInstructions, publishedAt: Date.now(), publishedByUserId: admin._id }`.
4. Patches the template to set `currentVersionId` to the new version's ID.
5. Inserts an `auditLog` row: `{ actorUserId: admin._id, action: "TEMPLATE_CREATED", targetType: "template", targetId: templateId (as string), metadata: { category, name, versionId }, createdAt: Date.now() }`.
6. Returns the `templateId`.

**`publishNewVersion`** — Mutation that:

1. Calls `requireAdmin`.
2. Reads the template by ID; throws `NOT_FOUND` if missing.
3. Queries all `templateVersions` for this template via the `by_template` index. Computes `newVersion = max(existing versions) + 1`. If no versions exist (edge case), starts at 1.
4. Inserts a new `templateVersions` row with the computed version number and provided content fields.
5. Patches the template to set `currentVersionId` to the new version.
6. Inserts an `auditLog` row: `{ actorUserId: admin._id, action: "TEMPLATE_PUBLISHED", targetType: "templateVersion", targetId: newVersionId (as string), metadata: { templateId, version: newVersion }, createdAt: Date.now() }`.
7. Returns the new `templateVersionId`.

**`archive`** — Mutation that:

1. Calls `requireAdmin`.
2. Reads the template by ID; throws `NOT_FOUND` if missing.
3. Patches the template with `archivedAt: Date.now()`.
4. Inserts an `auditLog` row: `{ actorUserId: admin._id, action: "TEMPLATE_ARCHIVED", targetType: "template", targetId: templateId (as string), metadata: {}, createdAt: Date.now() }`.
5. Returns `null`.

## Data dependencies

This module reads and writes the following tables:

- **`templates`** — Created by `create`, read by all five functions, patched by `create` (to set `currentVersionId`), `publishNewVersion` (to update `currentVersionId`), and `archive` (to set `archivedAt`). Uses the `by_category` index only implicitly (not queried by category in this module; the public `templates/list` does that).
- **`templateVersions`** — Created by `create` and `publishNewVersion`. Read by `listVersions` and by `publishNewVersion` (to compute next version number). Uses the `by_template` index. **Never patched or deleted** — this is the immutability contract.
- **`auditLog`** — Created by all three mutations. Never read by this module (audit log viewing is a separate concern). Uses the `by_actor` index only at the table level.
- **`users`** — Read indirectly via `requireAdmin` from `convex/lib/auth.ts`.

Key fields:

- `templates.currentVersionId` — updated by `create` and `publishNewVersion`. This is the pointer that the public `templates/list` query and `cases/create` mutation resolve to get the active version.
- `templates.archivedAt` — set by `archive`. Presence of this field is the signal for the public query to exclude the template.
- `templateVersions.version` — monotonic integer within a template. Used for sorting and for computing the next version number.
- `auditLog.action` — one of `TEMPLATE_CREATED`, `TEMPLATE_PUBLISHED`, `TEMPLATE_ARCHIVED`.
- `auditLog.targetId` — stored as `string` (not `Id<>`) because the schema defines it as `v.string()`. The implementation converts the Convex ID to string.

## Invariants

### Admin-only access

All five functions call `requireAdmin` from `convex/lib/auth.ts` as their first operation. This helper calls `requireAuth` (which throws `UNAUTHENTICATED` if no session) and then checks `user.role === "ADMIN"` (which throws `FORBIDDEN` if not admin). Non-admin users must never receive template data or be able to modify templates.

### Immutable versions

`templateVersions` rows are insert-only. Once a row is created by `create` (v1) or `publishNewVersion` (v2+), no function in this module or any other module may patch or delete it. This guarantees that cases pinned to a specific `templateVersionId` always resolve to the exact same content. The implementation enforces this by never calling `ctx.db.patch` or `ctx.db.delete` on the `templateVersions` table.

### Monotonic version numbering

`publishNewVersion` computes the next version number by querying all existing versions for the template, finding the maximum `version` field, and adding 1. This guarantees strict monotonic ordering within each template. Concurrent publishes are safe because Convex mutations are transactional — two concurrent `publishNewVersion` calls for the same template will be serialized, and each will see the other's insert.

### Case pinning unaffected

This module never reads or writes the `cases` table. Cases pin `templateVersionId` at creation time (in `convex/cases.ts`), and that field is never changed. Publishing a new version or archiving a template only affects the `templates` and `templateVersions` tables — existing cases are structurally unaffected.

### Audit log completeness

Every mutation inserts exactly one `auditLog` row before returning. The audit row includes `actorUserId` (the admin who performed the action), `action` (the action enum), `targetType` (`"template"` or `"templateVersion"`), `targetId` (the affected document ID as string), `metadata` (action-specific context), and `createdAt` (timestamp). This ensures a complete, append-only record of every admin template operation.

### Archive is soft-delete

Archiving sets `archivedAt` on the template row. It does not delete the template, its versions, or any references to them. `listAll` continues to return archived templates. The public `templates/list` query (not in this module) filters them out. Cases pinned to archived template versions continue to resolve.

## Edge cases

### Loading state

`listAll` returns `[]` if no templates exist. `listVersions` returns `[]` if no versions exist for the template (unlikely but structurally possible if a template was created without the initial version step completing). Neither is an error.

### Template not found

`publishNewVersion` and `archive` throw `ConvexError({ code: "NOT_FOUND" })` if the `templateId` does not resolve to a document. This covers deleted-by-other-means or invalid IDs.

### Archive idempotency

Calling `archive` on an already-archived template overwrites `archivedAt` with the current timestamp and writes a new audit log entry. This is intentional — it's simpler than checking and throwing, and the audit log captures every invocation.

### Version numbering with no prior versions

If `publishNewVersion` is called on a template that somehow has zero existing versions, it creates version 1. This is a defensive edge case — normal flow always has v1 from `create`.

### Non-admin calling any function

All five functions throw `ConvexError({ code: "FORBIDDEN", message: "Admin access required", httpStatus: 403 })` if the caller's `user.role` is not `"ADMIN"`. Unauthenticated callers throw `UNAUTHENTICATED` before the admin check is reached.

## Non-goals

- **Public `templates/list` query** — This module does not implement or modify the public-facing template list. That query (which filters out archived templates) is a separate concern, potentially already seeded or in another ticket.
- **Admin UI components** — No React pages or components. The admin list page (`/admin/templates`) and edit page (`/admin/templates/:id`) are separate tickets.
- **Unarchive/restore** — There is no mutation to clear `archivedAt`. Out of scope for v1.
- **Hard delete** — Templates are never deleted, only archived.
- **Audit log viewing** — Reading/querying the `auditLog` table for display is a separate ticket.
- **Template duplication** — No clone/copy functionality.
