# Contract: WOR-132 — Admin Convex module — template CRUD, versioning, audit log writes

## Why this work exists

Admin users need server-side functions to manage coaching templates — create, publish new immutable versions, and archive — without disrupting existing cases pinned to older versions. This builds the `admin/templates` and `admin/templateVersions` query and mutation layer from TechSpec §7.1–7.2, with audit logging for traceability of all admin actions.

## Files and exports

### `convex/admin/templates.ts` (create)

This file contains the admin-only template management queries and mutations. It uses the existing `requireAdmin` helper from `convex/lib/auth.ts` for authorization. The file is organized as a flat set of Convex function exports following the project convention (similar to `convex/cases.ts`, `convex/invites.ts`).

Exports:
- **`listAll`** — Query that returns all template documents (including archived). No filtering by `archivedAt`; the admin sees everything. Enforces `requireAdmin`.
- **`create`** — Mutation that creates a `templates` row and an initial `templateVersions` row (version 1), sets `currentVersionId`, and writes an audit log entry with `action: "TEMPLATE_CREATED"`.
- **`publishNewVersion`** — Mutation that reads existing versions for the template, computes `max(version) + 1`, inserts a new immutable `templateVersions` row, patches `templates.currentVersionId`, and writes an audit log entry with `action: "TEMPLATE_PUBLISHED"`.
- **`archive`** — Mutation that sets `archivedAt = Date.now()` on the template and writes an audit log entry with `action: "TEMPLATE_ARCHIVED"`. Does NOT delete the template or its versions.

### `convex/admin/templateVersions.ts` (create)

Exports:
- **`list`** — Query that returns all `templateVersions` rows for a given `templateId`, sorted by `version` descending (newest first). Enforces `requireAdmin`.

## Data dependencies

### Queries/mutations this work defines (not consumes)

All five functions are new exports. They read/write:
- **`templates` table** — full CRUD (create row, read all, patch `currentVersionId` and `archivedAt`).
- **`templateVersions` table** — insert new rows, read by `templateId` index. Never update existing rows.
- **`auditLog` table** — insert-only. Each admin write produces exactly one row.
- **`users` table** — read via `requireAdmin` (checks `role === "ADMIN"`).

### Audit log entry shape

```ts
{
  actorUserId: Id<"users">,       // the admin performing the action
  action: "TEMPLATE_CREATED" | "TEMPLATE_PUBLISHED" | "TEMPLATE_ARCHIVED",
  targetType: "template" | "templateVersion",
  targetId: string,               // the template or templateVersion _id as string
  metadata: { ... },              // action-specific; e.g. { version: 2, templateId: "..." }
  createdAt: number,              // Date.now()
}
```

For `TEMPLATE_CREATED`: `targetType = "template"`, `targetId = templateId`, `metadata = { name, category, versionId }`.
For `TEMPLATE_PUBLISHED`: `targetType = "templateVersion"`, `targetId = versionId`, `metadata = { templateId, version }`.
For `TEMPLATE_ARCHIVED`: `targetType = "template"`, `targetId = templateId`, `metadata = { name }`.

## Invariants

**Admin-only access:** Every function in these files calls `requireAdmin(ctx)` as its first operation. If the user is not authenticated or not an admin, a `ConvexError` with `code: "FORBIDDEN"` is thrown before any data is read or written. This matches the existing pattern in `convex/lib/auth.ts:94-108`.

**Immutable versions:** Once a `templateVersions` row is inserted, it is never patched or deleted. The `publishNewVersion` mutation only inserts; it never modifies existing version rows. This guarantees that cases pinned to a `templateVersionId` always resolve to the exact content that existed when the case was created.

**Monotonic version numbering:** `publishNewVersion` queries all existing versions for the template (using the `by_template` index), finds the maximum `version` number, and uses `max + 1`. For a fresh template (which already has v1 from `create`), the first `publishNewVersion` call produces v2.

**Soft-delete archival:** `archive` sets `archivedAt` but does not remove data. The public `templates/list` query (which already exists or will be built separately) filters out templates with `archivedAt !== undefined`. The admin `listAll` returns everything.

**Audit completeness:** Every mutation writes exactly one `auditLog` row before returning. The insert is part of the same Convex transaction as the template/version write, so either both succeed or neither does.

## Edge cases

**Archiving an already-archived template:** The `archive` mutation should be idempotent — if `archivedAt` is already set, it can either no-op or overwrite with the current timestamp. Decision: overwrite with new timestamp and still write an audit entry (the admin explicitly re-archived; logging the action is correct).

**Publishing to an archived template:** This is allowed. Archiving hides from the public picker but doesn't lock the template from admin edits. An admin may archive, then un-archive later (out of scope for this ticket), so allowing version publication on archived templates avoids unnecessary restrictions.

**Empty template name or guidance:** Validation is minimal in v1 — the Convex validator requires non-empty strings for `category`, `name`, and `globalGuidance` (they are `v.string()` which permits empty string). The implementation should enforce non-empty with a runtime check: throw a `ConvexError` with code `"INVALID_ARGS"` if `name.trim() === ""` or `globalGuidance.trim() === ""`.

**Template not found:** `publishNewVersion` and `archive` should throw `ConvexError` with code `"NOT_FOUND"` if the `templateId` does not exist in the database.

## Non-goals

- **No admin UI** — This ticket is backend-only. The admin pages (`/admin/templates`, `/admin/templates/:id`) are separate work.
- **No public `templates/list` query** — That query (non-archived templates for the category picker) is either already implemented or belongs to a different ticket. This ticket only builds the admin-scoped functions.
- **No un-archive mutation** — Not specified in the AC. If needed later, it's a separate ticket.
- **No self-serve admin promotion** — Admin role is assigned manually in the DB per the implementation notes.
- **No template deletion** — Only soft-delete (archive) is supported.
