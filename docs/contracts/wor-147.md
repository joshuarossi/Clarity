# Contract: WOR-147 — Invited party never fills their own intake form; synthesis is one-sided (US-04)

## Why this work exists

The PRD (US-04) requires: "Invited party then fills their own case form (same shape as initiator's) — their entries are private to them." Currently, after accepting an invite, `InviteAcceptPage` navigates the invitee directly to `/cases/:caseId/private` (the private coaching chat). This bypasses the `InviteeFormView` that already exists inside `CaseDetailPage`, so the invitee's `partyState` row is created with empty `mainTopic`, `description`, and `desiredOutcome` fields. Consequently, `synthesis.generate` only has the initiator's intake — producing one-sided, asymmetric guidance that undermines the "both parties heard" promise.

## Files and exports

### `src/routes/InviteAcceptPage.tsx` (modify)

**Single change:** In `handleAccept` (line 94), the post-redeem navigation target changes from `` `/cases/${result.caseId}/private` `` to `` `/cases/${result.caseId}` ``. This routes the invitee to the `CaseDetailPage` orchestrator, which checks `showInviteeForm` (line 226) and renders `InviteeFormView` when the invitee's `formCompletedAt` is unset. After the invitee submits the form, `formCompletedAt` is set, the orchestrator re-evaluates, and renders `<Navigate to="/cases/:caseId/private" replace />` — landing the invitee in private coaching with their perspective captured.

No other changes to InviteAcceptPage — the component's exported signature, its query/mutation usage, and all other UI states (invalid token, consumed token, logged-out view, decline flow) remain identical.

### `src/routes/CasePrivatePage.tsx` (modify)

**Defense-in-depth guard:** Inside `CasePrivatePageInner`, after `partyStates` resolves, add a check: if the user is an invitee (`partyStates.self.role === "INVITEE"`) and their `formCompletedAt` is falsy, render `<Navigate to={`/cases/${caseId}`} replace />`. This prevents an invitee from reaching the chat via direct URL entry, browser history, or any other navigation path that bypasses `CaseDetailPage`.

The guard uses data already fetched — `partyStates` is queried on line 170 via `api.cases.partyStates`. No new queries are needed. The guard is placed after the loading check (line 201–208) so it doesn't fire while data is still undefined.

The component's exported signature is unchanged. All existing functionality (chat messages, mark-complete, read-only banner, solo mode) is unaffected for users who have completed the form.

## Data dependencies

- **`api.invites.redeem`** — called by InviteAcceptPage on accept. Returns `{ caseId }`. No change to what this mutation does; only the navigation target after its completion changes.

- **`api.cases.partyStates`** — already called by both CaseDetailPage (line 192) and CasePrivatePage (line 170). Returns `{ self: { role, formCompletedAt, ... }, other: { ... } | null }`. The `self.role` and `self.formCompletedAt` fields drive both the orchestrator's form-display logic and the new guard in CasePrivatePage.

- **`api.cases.updateMyForm`** — called by InviteeFormView (CaseDetailPage line 58) on form submit. Sets `mainTopic`, `description`, `desiredOutcome`, and `formCompletedAt` on the invitee's `partyState` row. No change to this mutation.

## Invariants

**Invitee cannot skip the intake form.** Two layers enforce this: (1) InviteAcceptPage navigates to the orchestrator, which shows the form when `formCompletedAt` is unset; (2) CasePrivatePage redirects invitees without `formCompletedAt` back to the orchestrator. Both checks use the same `partyStates.self.formCompletedAt` field.

**Form shape matches initiator's.** The `InviteeFormView` has three required fields: `mainTopic` (text input), `description` (textarea), and `desiredOutcome` (textarea). These match the initiator's `NewCasePage` intake fields. Category is displayed read-only from the case document — the invitee does not re-select it, which is correct because category is a case-level property set at creation.

**Privacy of invitee entries.** The `updateMyForm` mutation (convex/cases.ts line 303) is server-side authenticated and scoped: it looks up the caller's partyState row by `userId` and only writes to that row. The `partyStates` query returns `self` (the caller's data) and `other` (a privacy-filtered view of the other party — no `mainTopic`, `description`, or `desiredOutcome`). This is unchanged and already enforced.

**Existing InviteeFormView is reused.** The component at CaseDetailPage.tsx lines 51–142 is not modified. It renders three fields, validates all are non-empty, calls `updateMyForm`, and handles errors. After submission, `formCompletedAt` is set on the server, Convex reactivity updates the `partyStates` query result, `showInviteeForm` flips to false, and the orchestrator renders the `<Navigate>` to private coaching.

## Edge cases

**Direct URL to /cases/:caseId/private.** An invitee who types the URL directly or navigates via browser history is caught by the CasePrivatePage guard and redirected to `/cases/:caseId` to complete the form. Once the form is submitted, the redirect no longer fires and the invitee can access private coaching normally (including returning via URL).

**Solo mode invitee.** In solo mode, the invitee partyState is created with empty form fields (convex/cases.ts create mutation). When the user toggles to the invitee role and visits CaseDetailPage, `showInviteeForm` is true (isInvitee && !formCompletedAt && isPrivateCoachingStatus), and the form is shown. The CasePrivatePage guard also applies: if the solo user navigates to `/cases/:caseId/private` while acting as INVITEE without completing the form, they're redirected. This is correct — solo mode should exercise the same flow.

**Initiator accessing /cases/:caseId/private.** The guard only fires for `role === "INVITEE"`. Initiators always have `formCompletedAt` set at case creation time (the create mutation populates their form fields). The guard does not affect them.

**Loading state.** Both the CaseDetailPage orchestrator and the CasePrivatePage guard render `<LoadingSpinner />` while `partyStates` is undefined. The redirect/form-check only evaluates after data loads. No flicker.

## Non-goals

- **No changes to InviteeFormView.** The existing component has the correct fields, validation, mutation call, and error handling. It does not need styling, accessibility, or functional changes for this ticket.
- **No changes to updateMyForm mutation.** It already sets `formCompletedAt` and scopes writes to the caller's partyState.
- **No changes to the redeem mutation.** Creating the partyState with empty form fields is intentional — the form is filled in a separate step after redeem.
- **No new routes.** The orchestrator pattern (CaseDetailPage shows form or redirects to subroute) handles this without a dedicated `/cases/:caseId/intake` route.
- **No changes to synthesis.** Once both partyStates have populated fields, `synthesis.generate` already consumes both sides. The bug was that the invitee's fields were never populated — this fix ensures they are.
