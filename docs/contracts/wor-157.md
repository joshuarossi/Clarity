# Contract: WOR-157 — Remove duplicate users index to unblock Convex deploy

## Why this work exists

The `users` table in `convex/schema.ts` defines two indexes on the same
field (`["email"]`): one named `"by_email"` (line 24) and one named
`"email"` (line 25). Convex rejects schemas with duplicate-field indexes
during `npx convex deploy`, blocking CI end-to-end tests.

## Files and exports

### `convex/schema.ts` (modify)

Delete line 25: `.index("email", ["email"])`. This is the only change.
The surviving index `"by_email"` is the one referenced by all five
`withIndex("by_email", ...)` call sites across `convex/users.ts`,
`convex/lib/auth.ts`, and `convex/seed.ts`. No code anywhere references
an index named `"email"`.

## Data dependencies

No new queries, mutations, or actions are introduced. Existing call
sites are unchanged.

## Invariants

1. **Single email index:** After the fix, the users table has exactly one
   index on `["email"]`, named `"by_email"`. Convex deploy must succeed.
2. **No broken references:** No `.withIndex("email", ...)` calls exist
   in the codebase (verified by grep). Removing the `"email"` index
   breaks nothing.

## Edge cases

- **Existing deployed index:** If the `"email"` index already exists in a
  deployed Convex environment, removing it from the schema will cause
  Convex to drop the index on next deploy. This is intentional — the
  index is redundant.
- **Test assertion:** The existing `tests/unit/schema.test.ts` (line 110)
  asserts `totalIndexes` across all tables equals 22. After removing one
  index this becomes 21. The test must be updated to expect 21.

## Non-goals

- Renaming `"by_email"` to `"email"` — all call sites already use
  `"by_email"` and changing them would be unnecessary churn.
- Changing field definitions or adding/removing fields.
- Any other schema modifications.
