# Contract: WOR-156 — Convex deploy fails: convex/auth.config.ts has wrong shape

## Why this work exists

The CI e2e job runs `npx convex deploy` which reads `convex/auth.config.ts` to configure the Convex platform's auth layer. The Convex platform expects each entry in the `providers` array to be a simple `{ domain, applicationID }` object that tells the platform how to validate JWTs. The current file instead exports full provider constructor objects (Email from `@convex-dev/auth/providers/Email` and Google from `@auth/core/providers/google`), which are runtime provider configurations meant for `convexAuth()` in `convex/auth.ts`. This shape mismatch causes `npx convex deploy` to fail, blocking all CI e2e tests.

## Files and exports

### `convex/auth.config.ts` (modify, config)

This file must be rewritten to export the platform-facing auth configuration. The Convex deploy process reads this file to know how to validate auth tokens. The correct shape is a default export with a `providers` array of `{ domain, applicationID }` objects. For Convex Auth (self-hosted auth), this is a single entry: `{ domain: process.env.CONVEX_SITE_URL, applicationID: "convex" }`.

All provider constructor logic (Email, Google) and the `@auth/core` import must be removed from this file. The Email `sendVerificationRequest` handler and Google OAuth credentials belong in `convex/auth.ts` where `convexAuth()` is configured.

### `convex/auth.ts` (modify, helper)

The provider configuration currently imported from `auth.config.ts` must be moved inline. The file already calls `convexAuth({ providers: authConfig.providers, ... })`. After the fix, it will define the Email and Google providers directly and pass them to `convexAuth()` without importing from `auth.config.ts`.

The `createOrUpdateUser` callback is unchanged.

The public exports (`auth`, `signIn`, `signOut`, `store`) are unchanged.

### `tests/unit/convex-auth.test.ts` (modify, test-infrastructure)

Existing tests that import `authConfig` from `convex/auth.config.ts` and assert it has providers with `id: "magic-link"` and `id: "google"` must be updated. The auth config no longer contains provider objects with `id` fields. Tests should:

1. Verify `auth.config.ts` exports `{ providers: [{ domain, applicationID }] }` — the deploy-compatible shape.
2. Verify provider configuration (magic-link and Google) by testing `convex/auth.ts` exports instead — confirm `signIn` and `signOut` are exported functions.
3. The Google OAuth `clientId`/`clientSecret` env-var test should move to testing that `convex/auth.ts` configures Google with env vars (or be adapted to test the provider setup in auth.ts).

## Data dependencies

### `convex/lib/auth.getUserByEmail` (consumed, unchanged)

Called inside the `createOrUpdateUser` callback in `convex/auth.ts`. No changes to this dependency.

## Invariants

**Deploy-compatible config shape.** `convex/auth.config.ts` must export `{ providers: [{ domain: process.env.CONVEX_SITE_URL, applicationID: "convex" }] }`. This is the shape the Convex platform parses during `npx convex deploy`. Any other shape causes a deploy failure.

**Provider config lives in auth.ts.** The Email (magic-link) and Google OAuth provider constructors, including the `sendVerificationRequest` handler and OAuth credentials, must be defined in `convex/auth.ts` where `convexAuth()` is called. These are runtime provider configurations, not platform configuration.

**No @auth/core import.** `@auth/core` is not in `package.json` as a direct dependency. The current import `from "@auth/core/providers/google"` resolves only because it's a transitive dependency of `@convex-dev/auth`. The Google provider should be imported from `@convex-dev/auth`'s public API (e.g. via the `@convex-dev/auth/providers/` namespace) or configured through the mechanism documented by `@convex-dev/auth`. If `@convex-dev/auth` re-exports `@auth/core` providers, that re-export path should be used. If Google OAuth requires `@auth/core` as a direct dependency, it must be added to `package.json` explicitly.

**Behavioral parity.** After the fix, the auth system must still configure exactly two providers (magic-link and Google OAuth), the user upsert callback must still call `getUserByEmail`, and all existing auth flows must continue to work.

## Edge cases

**`CONVEX_SITE_URL` not set.** The `domain` field in auth.config.ts reads from `process.env.CONVEX_SITE_URL`. During local dev, this is set by `npx convex dev`. During CI deploy, it is set by the Convex deploy process. If unset, the value will be `undefined`, which is acceptable — the Convex platform handles this case.

**Provider import paths.** The implementation author must verify which import path to use for the Google OAuth provider. If `@convex-dev/auth` does not re-export Google, the implementation should add `@auth/core` to `package.json` explicitly (as the setup guide recommends `@auth/core@0.37.0`) and import from `@auth/core/providers/google`. The key constraint is: this import must NOT be in `auth.config.ts`.

**Test compatibility.** The existing `convex-auth.test.ts` tests assert specific shapes on the auth config export. These assertions will fail after the config shape changes. The test author must update these assertions to match the new shape while preserving the intent (verifying that auth is correctly configured).

## Non-goals

**No auth behavior changes.** This is purely a config-shape fix. The auth flow, providers, user upsert, session management — all remain identical.

**No new files.** No `convex/http.ts` or other new modules are created by this ticket.

**No CI workflow changes.** The deploy command (`npx convex deploy --cmd 'echo "deployed"'`) is correct; it's the config file shape that's wrong.

**No package.json changes** unless the Google provider import requires `@auth/core` as an explicit dependency (see Edge Cases).
