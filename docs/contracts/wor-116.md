# Contract: WOR-116 — Dashboard page — case list with status indicators

## Why this work exists

The dashboard is the user's home base after login — it answers "what needs my attention?" at a glance. Currently `/dashboard` is a stub. This ticket builds the full case-list experience with active/closed grouping, status pills with shape+color+text semantics, the "+ New Case" entry point, and skeleton/empty states per the DesignDoc §4.4.

## Files and exports

### `src/routes/DashboardPage.tsx` (create, connected)

The public export is the **connected** component `DashboardPage`. It takes no props and internally calls `useQuery(api.cases.listForDashboard)` for data. CaseRow is an internal (non-exported) presentational component that renders a single case card. The connected component handles three states: loading (skeleton), empty (zero cases), and populated (active + closed sections).

The decision to make `DashboardPage` connected (not presentational) follows the project convention established by `LoginPage`, `NewCasePage`, `InviteSharingPage`, and `InviteAcceptPage` — all page-level components in `src/routes/` are connected and call Convex hooks directly. CaseRow is kept internal because it's a layout detail of the dashboard, not reusable elsewhere.

### `convex/cases.ts` (modify, query)

A new `listForDashboard` query is added alongside the existing `list` query. The new query enriches each case with:

- `otherPartyName`: the displayName of the other party (resolved by looking up the user by ID), or `null` if `inviteeUserId` is not set.
- `otherPartyRole`: `"initiator"` or `"invitee"` — the role of the _other_ party relative to the caller (so the caller knows which avatar color to use).
- `statusVariant`: one of `"pill-turn"`, `"pill-waiting"`, `"pill-ready"`, `"pill-closed"` — computed server-side.
- `statusLabel`: human-readable text for the pill: `"Your turn"`, `"Waiting"`, `"Ready for joint"`, `"Closed"`.

This keeps status-pill logic server-side, avoiding duplication between the UI and any future consumer of case-list data. The existing `list` query is left unchanged for backward compatibility.

### `src/App.tsx` (modify, route)

Replace the inline `DashboardPage` stub function with an import of the new component from `src/routes/DashboardPage.tsx`. No route path changes — `/dashboard` stays wrapped in `<ProtectedRoute>`.

## Data dependencies

### `api.cases.listForDashboard` (NEW)

Returns an array of enriched case objects. For each case the authenticated user is party to (as initiator or invitee), the query:

1. Collects cases from `by_initiator` and `by_invitee` indexes (same as existing `list`).
2. For each case, looks up the _other_ party's user record to get `displayName`. If `inviteeUserId` is null, `otherPartyName` is null.
3. Fetches the caller's `partyState` to determine whether private coaching is complete (needed for "your turn" vs "waiting" logic).
4. Computes `statusVariant` and `statusLabel` based on case status and caller's party state:

| Case status                                    | Caller's PC complete? | Variant        | Label               |
| ---------------------------------------------- | --------------------- | -------------- | ------------------- |
| `DRAFT_PRIVATE_COACHING` (caller is initiator) | n/a                   | `pill-waiting` | `"Waiting"`         |
| `BOTH_PRIVATE_COACHING`                        | No                    | `pill-turn`    | `"Your turn"`       |
| `BOTH_PRIVATE_COACHING`                        | Yes                   | `pill-waiting` | `"Waiting"`         |
| `READY_FOR_JOINT`                              | n/a                   | `pill-ready`   | `"Ready for joint"` |
| `JOINT_ACTIVE`                                 | n/a                   | `pill-ready`   | `"In session"`      |
| `CLOSED_RESOLVED`                              | n/a                   | `pill-closed`  | `"Closed"`          |
| `CLOSED_UNRESOLVED`                            | n/a                   | `pill-closed`  | `"Closed"`          |
| `CLOSED_ABANDONED`                             | n/a                   | `pill-closed`  | `"Closed"`          |

For `DRAFT_PRIVATE_COACHING` where the caller is the _invitee_ (just accepted invite, hasn't filled form): variant is `pill-turn`, label is `"Your turn"`.

Fields consumed by the UI per row: `_id`, `status`, `isSolo`, `category`, `createdAt`, `updatedAt`, `otherPartyName`, `otherPartyRole`, `statusVariant`, `statusLabel`.

## Invariants

**Privacy.** The `listForDashboard` query returns only case metadata and the other party's `displayName`. It never returns `description`, `desiredOutcome`, coaching messages, synthesis text, or any private-to-one-party data.

**Status variant is server-authoritative.** The client renders whatever `statusVariant` the query returns — it does not re-derive it. This prevents client/server divergence and makes the query the single source of truth for pill state.

**"Your turn" logic.** In `BOTH_PRIVATE_COACHING`, the caller's `privateCoachingCompletedAt` determines the variant. If the caller has completed PC but the other hasn't, it's `pill-waiting`. If the caller hasn't completed PC, it's `pill-turn`. This ensures the pill always reflects _what the viewer needs to do_, not the case's global state.

**Closed section collapsed.** The Closed Cases section renders its contents inside a disclosure element (e.g. `<details>` or an accordion) that is closed by default. Users can expand it manually. This prevents visual clutter from resolved/abandoned cases.

**Skeleton count.** Exactly 3 skeleton rows during loading, matching DesignDoc §6.3. Skeletons are rendered inline (not a spinner overlay).

## Edge cases

**Loading state.** While `listForDashboard` is undefined (query in flight), render 3 skeleton case-row placeholders. No "Active Cases" / "Closed Cases" section headers during loading — just the skeletons under the "+ New Case" button.

**Empty state.** When the query returns an empty array, render the verbatim empty-state message: "No cases yet. When you're ready to work through something, start a new case." The empty state replaces both the Active and Closed sections.

**Invitee hasn't joined.** When `otherPartyName` is null, the case row displays "Waiting for invite" in place of the party name. The `PartyAvatar` is still rendered but with a placeholder initial (e.g. "?" or the first letter of "Waiting"). The `otherPartyRole` will be `"invitee"` since the caller is always the initiator in this case.

**Solo mode.** Cases with `isSolo: true` display a "Solo" badge (text label) adjacent to the category. This satisfies the AC without inventing a complex visual treatment — a simple text badge is sufficient for developer/test mode.

**All cases closed.** If all cases are in CLOSED\_\* states, the Active Cases section shows a message like "No active cases" and the Closed Cases section auto-expands (or the empty-state logic applies if appropriate). Implementation author should render Active section header with no rows, and Closed section with the cases.

## Non-goals

**No case search or filtering.** The DesignDoc mentions Cmd/Ctrl-K case search as v1.1. This ticket renders a flat list.

**No "Nudge" button** for cases where the invitee hasn't joined yet. DesignDoc §9 mentions this as a separate interaction for cases >7 days old.

**No real-time typing indicators or presence.** Deferred to v1.1 per TechSpec §8.3.

**No pagination.** The PRD targets 50 concurrent active cases max in v1 — a flat list is fine.
