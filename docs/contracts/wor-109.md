# Contract: WOR-109 — Auth Convex module — Convex Auth setup, magic link + Google OAuth, user upsert

## Why this work exists

Clarity's authorization model (WOR-97's auth helpers, ProtectedRoute, AdminRoute) depends on a working authentication layer, but no authentication provider is configured yet. The app has the `@convex-dev/auth` package installed, `ConvexAuthProvider` wired into `src/main.tsx`, and `useConvexAuth()` used in route guards — but there is no `convex/auth.ts` or `convex/auth.config.ts` to actually enable sign-in. This ticket bridges that gap: it configures Convex Auth with magic link and Google OAuth providers, wires up user upsert on first login (reusing WOR-97's `getUserByEmail`), and exposes the `signIn`/`signOut` actions that the client-side auth flow will call.

## Files and exports

### `convex/auth.config.ts` (create, config)

The Convex Auth provider configuration file. This file exports a default config object that lists the authentication providers available to the app. It configures exactly two providers:

1. **MagicLink** — email-based one-time login link from `@convex-dev/auth/providers/MagicLink`. This is the primary, low-friction auth method per PRD US-01.
2. **Google OAuth** — from `@auth/core/providers/google`. Configured with `clientId` and `clientSecret` read from Convex environment variables `GOOGLE_OAUTH_CLIENT_ID` and `GOOGLE_OAUTH_CLIENT_SECRET`.

No password provider is included. This is deliberate: PRD US-01 explicitly excludes password-based registration to reduce friction and eliminate password-reset flows.

### `convex/auth.ts` (create, helper)

The Convex Auth setup module. Calls `convexAuth()` from `@convex-dev/auth/server` with the configured providers and exports four items:

- **`auth`** — HTTP action handler that Convex Auth uses for OAuth redirect callbacks and token endpoints. Automatically mounted by Convex.
- **`signIn`** — Action that initiates a sign-in flow. Client calls `useAuthActions().signIn("magic-link", { email })` or `useAuthActions().signIn("google")`.
- **`signOut`** — Action that invalidates the server-side session and signals the client to clear credentials. This is the logout mutation required by AC 6 / US-02.
- **`store`** — Internal mutation used by Convex Auth for session and user management.

The critical implementation detail is the **user upsert callback**. The `convexAuth()` helper accepts a `callbacks` option with an `createOrUpdateUser` hook. This callback receives the Convex mutation context and the user profile from the auth provider. It must:

1. Extract the `email` from the profile.
2. Call `getUserByEmail(ctx, email)` from `convex/lib/auth.ts` to upsert a row in the app's `users` table.

This reuses the already-tested upsert logic from WOR-97: first call creates a row with `role: "USER"`, `displayName` derived from the email prefix, and `createdAt: Date.now()`; subsequent calls return the existing row unchanged.

## Data dependencies

### `convex/lib/auth.getUserByEmail` (consumed, read-only dependency)

Called inside the `createOrUpdateUser` callback. Takes a writable `ctx.db` and an email string. Returns the upserted `Doc<"users">`. This function already exists and is fully tested by WOR-97.

### Convex Auth internal tables

`@convex-dev/auth` manages its own internal tables (`authSessions`, `authAccounts`, `authRefreshTokens`, `authVerificationCodes`, etc.) via the `store` mutation. These are internal to the auth library and not referenced by application code. The `convex/schema.ts` may need `authTables` from `@convex-dev/auth/server` added — however, since WOR-95 (schema) is already merged and does not include auth tables, this ticket's implementation author should check whether `@convex-dev/auth@^0.0.92` auto-creates its tables or requires explicit schema registration. If explicit registration is needed, `convex/schema.ts` must be modified to include `...authTables` in the schema definition. This is the one potential schema modification; the contract flags it here so the test and implementation authors are aligned.

## Invariants

**Exactly two providers, no password.** The auth config must include MagicLink and Google OAuth and nothing else. If a password provider were added, it would violate PRD US-01's explicit "no password-based registration" requirement and introduce password-reset complexity the product deliberately avoids.

**User upsert is keyed by email via the by_email index.** The `getUserByEmail` function (WOR-97) uses the `by_email` index on the `users` table. The upsert callback must pass the email from the auth provider's profile, not any other identifier. This ensures that a user who signs in with magic link and then later with Google OAuth (same email) resolves to the same `users` row.

**Idempotent upsert — no side effects on repeat login.** The second and subsequent logins with the same email must not modify `createdAt`, `role`, `displayName`, or any other field on the existing `users` row. This is already guaranteed by `getUserByEmail`'s implementation but is called out as a contract invariant because the test author must verify it.

**Session managed by Convex Auth internals.** The session token, refresh logic, and cookie/localStorage persistence are all handled by `@convex-dev/auth` and `ConvexAuthProvider`. This ticket does not implement custom session management. Session validity is until explicit logout or Convex Auth's default expiry (the 30-day expiry mentioned in the AC is a Convex Auth configuration if supported, otherwise the default is accepted for v1).

**signOut invalidates server-side.** Calling `signOut` must invalidate the session on the Convex backend (so that `ctx.auth.getUserIdentity()` returns null for that session going forward) AND signal the client to clear credentials. This is handled by Convex Auth's built-in `signOut` action.

**getUserIdentity returns identity after sign-in.** After a successful sign-in via either provider, any Convex function calling `ctx.auth.getUserIdentity()` must receive a non-null identity object with at minimum an `email` field matching the sign-in email. This is foundational — WOR-97's `requireAuth` depends on it.

## Edge cases

**Magic link email delivery.** The magic link provider requires a configured email transport (e.g., Resend). If the transport environment variable is not set, the sign-in action will fail. The implementation should let this fail naturally (Convex Auth surfaces the error); there is no custom error handling for misconfigured email transport in v1. The test environment will mock or skip actual email delivery.

**Google OAuth without credentials.** If `GOOGLE_OAUTH_CLIENT_ID` or `GOOGLE_OAUTH_CLIENT_SECRET` are not set, the Google OAuth flow will fail at runtime. The auth config file should read these from `process.env` and the provider will fail gracefully if they're missing. No compile-time validation is needed — missing env vars are a deployment issue.

**Same email, different providers.** A user who first signs in with magic link (email: alice@example.com) and later with Google OAuth (same email) should resolve to the same `users` row. This is guaranteed by `getUserByEmail`'s email-keyed upsert. Convex Auth internally links multiple auth accounts to the same user via email matching.

**Race condition on first login.** Two concurrent sign-in completions for the same email could both trigger the upsert callback simultaneously. Convex mutations are serializable, so one will succeed and the other will be retried. `getUserByEmail` handles this correctly (the retry finds the existing row). No application-level deduplication is needed.

**Loading state during auth.** `useConvexAuth()` returns `{ isLoading: true, isAuthenticated: false }` while the auth state is being resolved. `ProtectedRoute` already handles this by showing `LoadingSpinner`. No changes needed.

## Non-goals

**No login/registration UI.** This ticket configures the Convex Auth backend module. The `LoginPage` component (currently a stub in `App.tsx`) that renders sign-in forms with magic link input and Google OAuth button is a separate concern. The client will use `useAuthActions()` from `@convex-dev/auth/react` to call `signIn`/`signOut`, but the UI that triggers those calls is not part of this ticket.

**No custom token store.** Convex Auth manages session tokens internally. The contract explicitly avoids custom cookie handling, localStorage management, or token refresh logic.

**No admin self-promotion.** Per TechSpec §4.2, admin role promotion is a manual DB action in v1.

**No modifications to existing source files.** `src/main.tsx` already has `ConvexAuthProvider`. `convex/lib/auth.ts` already has `getUserByEmail`. `convex/schema.ts` already has the `users` table. The only potential modification is adding `authTables` to the schema if `@convex-dev/auth` requires it (see Data Dependencies section).

**No Clerk fallback implementation.** TechSpec §4.1 notes Clerk as the swap-out plan if Convex Auth beta bugs block progress. This ticket implements Convex Auth only; a Clerk migration would be a separate ticket if needed.

## Test coverage

### `tests/unit/convex-auth.test.ts` — ACs 1, 2, 3, 4, 7, 8 (unit)

Tests the auth configuration shape and user upsert behavior using `convex-test` and direct module imports.

**AC 1 (magic link provider configured):** Import the auth config from `convex/auth.config.ts` and assert the providers array includes an entry with id `"magic-link"` (or the canonical provider name used by `@convex-dev/auth`). Verify it is a function/object that represents the MagicLink provider.

**AC 2 (Google OAuth provider configured):** Assert the providers array includes an entry for Google OAuth. Verify it references `GOOGLE_OAUTH_CLIENT_ID` and `GOOGLE_OAUTH_CLIENT_SECRET` (the config reads from env vars, not hardcoded strings).

**AC 3 (user row created on first login):** Using `convex-test`, simulate the auth callback flow: call the user upsert logic with a test email. Query the `users` table by email and assert: row exists, `role === "USER"`, `createdAt` is a valid timestamp, `displayName` is derived from the email prefix.

**AC 4 (idempotent upsert):** Call the upsert logic twice with the same email. Assert only one `users` row exists. Assert `createdAt` is unchanged on the second call. Assert `_id` is identical.

**AC 7 (no password registration):** Assert the auth config providers array does not include a password provider. This is a negative test on the config shape.

**AC 8 (getUserIdentity returns identity):** Using `convex-test` with a configured identity (via `withIdentity`), call a function that invokes `ctx.auth.getUserIdentity()` and assert it returns an object with the expected `email` field. This validates the integration between Convex Auth's identity layer and downstream function calls.

### `e2e/auth-session.spec.ts` — ACs 5, 6 (e2e)

Playwright tests requiring a running Convex dev server and a real browser context.

**AC 5 (session persists across browser reloads):** Sign in (using a test helper or programmatic auth), navigate to `/dashboard`, assert the page renders (not redirected to `/login`). Reload the page. Assert `/dashboard` still renders and the user is still authenticated.

**AC 6 (logout clears session):** Sign in, trigger logout (click logout button or programmatically call signOut), assert redirect to the login page. Attempt to navigate to `/dashboard`, assert redirect to `/login` (session is gone). Optionally verify that a direct Convex function call returns null for `getUserIdentity`.
