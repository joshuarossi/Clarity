# Contract: WOR-111 — Cases Convex module — create, get, list, partyStates queries + mutations

## Why this work exists

The Cases module is the core CRUD layer that every downstream feature depends on. Without it, there is no way to create a mediation case, retrieve its details, or manage per-party form state. This ticket provides the foundational queries and mutations that the dashboard, private coaching, invite flow, and joint chat all consume.

## Files and exports

### `convex/cases.ts` (create)

This is a new Convex module file following the project convention established by `convex/users.ts` — domain logic lives as top-level files under `convex/`, with functions exported as named `query` or `mutation` instances. The Convex runtime maps these exports to API paths automatically (e.g., `api.cases.list`, `api.cases.create`).

**`list`** — A query that returns all cases where the caller is either `initiatorUserId` or `inviteeUserId`. It queries both the `by_initiator` and `by_invitee` indexes, deduplicates (a solo case will appear in both result sets), and sorts by `updatedAt` descending. Returns the full `Doc<"cases">` array.

**`get`** — A query that returns a single case document. Uses `requirePartyToCase` from `convex/lib/auth.ts` to enforce that the caller is a party. Throws `ConvexError` with `code: "FORBIDDEN"` if not.

**`partyStates`** — A query returning the caller's full `partyStates` document and a stripped view of the other party's state. The other party's view contains only `{ role, hasCompletedPC: boolean }` — computed as `Boolean(privateCoachingCompletedAt)`. No form content (`mainTopic`, `description`, `desiredOutcome`), no `synthesisText`, no timestamps beyond what `hasCompletedPC` implies. If the other party has not yet joined (no partyStates row exists for them), `other` is `null`.

**`create`** — A mutation that:

1. Calls `requireAuth` for the caller identity.
2. Resolves the template: if `templateId` is provided, uses it; otherwise looks up the template for the given `category` via the `by_category` index. Reads `template.currentVersionId` and sets it as `cases.templateVersionId`. If no template or no `currentVersionId` exists, throws `INVALID_INPUT`.
3. Inserts the `cases` row with `schemaVersion: 1`, `status: "DRAFT_PRIVATE_COACHING"` (or `"BOTH_PRIVATE_COACHING"` for solo), `isSolo`, `category`, `initiatorUserId`, `createdAt: Date.now()`, `updatedAt: Date.now()`.
4. For solo mode: sets `inviteeUserId` to the caller, creates two `partyStates` rows (one `INITIATOR`, one `INVITEE`), does NOT generate an invite token. Returns `{ caseId, inviteUrl: null }`.
5. For standard mode: leaves `inviteeUserId` undefined, creates one `partyStates` row (`INITIATOR`), generates a 32-character crypto-random URL-safe token, inserts an `inviteTokens` row with `status: "ACTIVE"`. Returns `{ caseId, inviteUrl: "${SITE_URL}/invite/${token}" }`. The `SITE_URL` comes from `process.env.SITE_URL` or a sensible default.
6. The initiator's `partyStates` row is created with `mainTopic`, `description`, `desiredOutcome` from the args and `formCompletedAt: Date.now()`.

**`updateMyForm`** — A mutation that:

1. Calls `requireAuth` and `requirePartyToCase`.
2. Looks up the caller's `partyStates` row via the `by_case_and_user` index.
3. Patches `mainTopic`, `description`, `desiredOutcome` on the `partyStates` row.
4. Patches `updatedAt: Date.now()` on the `cases` row.
5. Returns `null`.

### `tests/unit/cases.test.ts` (create)

Integration tests using `convex-test` following the project's existing pattern in `tests/unit/convex-auth.test.ts` and `tests/unit/seed.test.ts`. All tests import the schema from `../../convex/schema` and use `convexTest` to set up an in-memory Convex environment. Test structure maps 1:1 to the test-gen brief in the task description.

## Data dependencies

All data lives in the Convex database. This module defines its own queries/mutations — it does not call other Convex functions via the API. It directly reads/writes the following tables:

- **`cases`** — primary table. Created by `create`, read by `list`/`get`, updated by `updateMyForm`.
- **`partyStates`** — per-party state. Created by `create`, read by `partyStates` query, updated by `updateMyForm`.
- **`inviteTokens`** — created by `create` (standard mode only). Read only to verify no duplicate tokens (edge case).
- **`templates`** — read by `create` to resolve category to template.
- **`templateVersions`** — NOT directly read. The `currentVersionId` from `templates` is used as-is for `cases.templateVersionId`.
- **`users`** — read indirectly via `requireAuth` helper.

Key fields consumed by downstream features:

- `cases.status` — drives the state machine and UI routing.
- `cases.templateVersionId` — pinned at creation, used by AI prompt assembly.
- `partyStates.mainTopic`, `.description`, `.desiredOutcome` — rendered in case detail and fed to AI coaching.
- `partyStates.privateCoachingCompletedAt` — used to derive `hasCompletedPC` for the other party's phase-level view.

## Invariants

### Party isolation on queries

`cases/get` and `cases/partyStates` must verify the caller is `initiatorUserId` or `inviteeUserId` on the case document. This is enforced by calling `requirePartyToCase` from `convex/lib/auth.ts`, which throws `ConvexError({ code: "FORBIDDEN" })`. A non-party user must never receive case details or party state.

### Privacy in partyStates

The `partyStates` query returns a two-part response: `self` (the caller's full `Doc<"partyStates">`) and `other` (a stripped object). The stripped object contains ONLY `{ role: "INITIATOR" | "INVITEE", hasCompletedPC: boolean }`. The boolean is computed as `Boolean(otherPartyState.privateCoachingCompletedAt)`. All other fields — `mainTopic`, `description`, `desiredOutcome`, `synthesisText`, `synthesisGeneratedAt`, `formCompletedAt`, `closureProposed`, `closureConfirmed` — are excluded from the `other` object. This enforces the PRD's visibility matrix: one party never sees the other's form content.

### Template pinning immutability

`cases.templateVersionId` is set during `create` by reading `templates.currentVersionId` for the resolved template. Once set, no mutation in this module or any future module may change it (TechSpec invariant 3.2.1). The `updateMyForm` mutation does not touch this field.

### Schema version

Every case created by `cases/create` sets `schemaVersion: 1` (literal). This is a forward-compatibility field per TechSpec invariant 3.2.6.

### State machine usage

`cases/create` for solo mode conceptually applies the `ACCEPT_INVITE` transition (skipping directly to `BOTH_PRIVATE_COACHING`). However, since there is no prior state to transition from (the case is being created, not transitioned), the implementation sets `status` directly to the correct initial value. The state machine helper (`validateTransition`) is relevant for _subsequent_ transitions (e.g., when both parties complete coaching), not for initial case creation. The contract acknowledges AC "All functions use the state machine helper for status transitions" as satisfied because: (a) creation is not a transition — it's an initialization; (b) `updateMyForm` does not change case status; (c) any future status-changing mutations in this module would use `validateTransition`.

### Auth enforcement

Every exported function calls `requireAuth` as its first operation. Unauthenticated calls receive `ConvexError({ code: "UNAUTHENTICATED", httpStatus: 401 })`.

## Edge cases

### Loading state / no cases

`cases/list` returns an empty array `[]` if the caller has no cases. This is not an error condition.

### Other party not yet joined

`cases/partyStates` when the invitee has not yet accepted returns `{ self: <initiator's partyState>, other: null }`. The `other` field is `null`, not an error. The caller (initiator) knows the case exists but the invitee hasn't joined yet.

### Solo case deduplication in list

A solo case has `initiatorUserId === inviteeUserId`. When `cases/list` queries both indexes, the same case document appears in both result sets. The implementation must deduplicate (e.g., by `_id` using a Set or Map) before sorting and returning.

### Template resolution failure

If `create` is called with a `category` that has no template (or the template has no `currentVersionId`), the mutation throws `ConvexError({ code: "INVALID_INPUT" })`. If a specific `templateId` is provided but the template doesn't exist or has no `currentVersionId`, same error.

### Invite token uniqueness

The 32-character random token is generated using crypto-random bytes encoded as URL-safe characters. Collision probability is negligible but the `by_token` index enforces uniqueness at the DB level.

## Non-goals

- **Invite redemption** — The `invites/redeem` mutation is a separate ticket. This module only creates the token; it does not consume it.
- **Private coaching messages** — Reading/writing to `privateMessages` is handled by downstream tickets.
- **Joint chat, synthesis, closure** — All handled by later tickets in the epic.
- **Template CRUD** — Admin template management is a separate P1 ticket.
- **UI components** — This is a backend-only module. No React components are created or modified.
- **Cron jobs** — The abandoned-case cleanup cron is a separate concern.
