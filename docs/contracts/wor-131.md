# Contract: WOR-131 — ClosedCaseView page — read-only archive

## Why this work exists

Once a case reaches a terminal status (CLOSED_RESOLVED, CLOSED_UNRESOLVED, or ABANDONED), users need a permanent, read-only archive of everything that happened. Currently the closed-case route is a stub placeholder. This ticket builds the real ClosedCaseView page at `/cases/:caseId/closed` with header, outcome details, tabbed transcript navigation, and privacy-preserving read-only display.

## Files and exports

### `src/routes/ClosedCaseView.tsx` (CREATE — connected component)

The main deliverable. This is a **connected** route-level component — it extracts `caseId` from `useParams`, calls `useQuery` for all data, and manages tab state via `useSearchParams`. It takes no props. This matches the established pattern of `JointChatView`, `CasePrivatePage`, and other route-level components in this project.

The component has an outer `ClosedCaseView` (exported) that extracts and validates `caseId`, and an inner `ClosedCaseViewInner` that receives the typed `caseId` and does all the work. This matches the `JointChatView` / `JointChatViewInner` pattern exactly.

**Internal structure:**
- Header section: case name (`caseDoc.category`), closure date (`caseDoc.closedAt` formatted), outcome text mapped from status
- Closure summary section: only rendered when `caseDoc.status === "CLOSED_RESOLVED"` and `caseDoc.closureSummary` is present
- Banner: static text "This case is closed. No new messages can be added." with `data-testid="closed-banner"`
- Tab bar: three tabs implemented as `<div role="tablist">` with `<button role="tab">` elements, using `aria-selected`, `aria-controls`, and `id` attributes for accessibility. Active tab stored in `?tab=joint|private|guidance` URL search param (default: `joint`)
- Tab panels: `<div role="tabpanel">` keyed to active tab
  - **Joint Chat tab:** reuses `ChatWindow` with joint messages mapped to `ChatMessage[]` format. No `MessageInput`, no retry buttons, no `DraftCoachPanel`
  - **My Private Coaching tab:** reuses `ChatWindow` with the caller's private coaching messages from `api.privateCoaching.myMessages`. Only the caller's messages are ever returned by this query (server-enforced)
  - **My Guidance tab:** renders the caller's synthesis text from `api.jointChat.mySynthesis`. Plain text display (matching the synthesis panel in `JointChatView`)

**Solo mode:** Supports `useSoloActingParty` hook for party toggling, passing `viewAsRole` to all queries that accept it. Shows `PartyToggle` in `PhaseHeader` when `solo.isSolo` is true. This matches `JointChatView` exactly.

**Data-testid attributes:**
- `data-testid="page-case-closed"` on the outer `<main>`
- `data-testid="closed-banner"` on the banner
- `data-testid="closed-header-outcome"` on the outcome text element
- `data-testid="closed-header-closure-date"` on the closure date element
- `data-testid="closed-closure-summary"` on the closure summary block (only present when resolved)
- `data-testid="tab-joint"`, `data-testid="tab-private"`, `data-testid="tab-guidance"` on the tab buttons
- `data-testid="tabpanel-joint"`, `data-testid="tabpanel-private"`, `data-testid="tabpanel-guidance"` on the tab panels

## Data dependencies

### `api.cases.get({ caseId })` → Case document
Fields consumed: `status`, `category` (used as case name in header), `closedAt` (formatted as date for header), `closureSummary` (displayed when resolved), `initiatorUserId` (for mapping joint message author variants), `isSolo`.

### `api.cases.partyStates({ caseId, viewAsRole? })` → Party state info
Fields consumed: `self.role` (for solo mode context), `self.synthesisText` (fallback if mySynthesis query differs). Primarily needed for solo mode support and the `PhaseHeader` party toggle.

### `api.jointChat.messages({ caseId, viewAsRole? })` → Joint chat messages array
Fields consumed per message: `_id`, `authorType` ("USER" | "COACH"), `authorUserId`, `content`, `status`, `isIntervention`, `createdAt`. Mapped to `ChatMessage[]` using the same variant logic as `JointChatView`: COACH → "coach-joint" / "coach-intervention", USER → "party-initiator" / "party-invitee" based on `authorUserId === caseDoc.initiatorUserId`.

### `api.privateCoaching.myMessages({ caseId, partyRole? })` → Caller's private coaching messages
Fields consumed per message: `_id`, `role` ("USER" | "AI"), `content`, `status`, `createdAt`. Mapped to `ChatMessage[]` with variant "user" for USER role and "coach" for AI role. This query is server-scoped to only return the authenticated caller's messages — the other party's messages are never returned.

### `api.jointChat.mySynthesis({ caseId, viewAsRole? })` → Caller's synthesis
Fields consumed: `text` (the synthesis prose). Rendered as plain text in the "My Guidance" tab panel. Returns `null` if no synthesis exists (edge case: case abandoned before synthesis generated).

## Invariants

### Privacy: other party's data never shown
The ClosedCaseView never calls any query that could return the other party's private coaching or synthesis. `api.privateCoaching.myMessages` is scoped server-side to the authenticated user. `api.jointChat.mySynthesis` returns only the caller's synthesis. There is no tab, button, or code path that fetches the other party's private data. The UI does not even present the option — tabs are labeled "My Private Coaching" and "My Guidance," not "Private Coaching" generically.

### Read-only: no mutations, no input controls
The component calls zero mutations. It renders no `MessageInput`, no `<textarea>`, no send button, no "Draft with Coach" button, no retry buttons on error messages, no closure proposal UI. The `ChatWindow` component is passed `onRetry={undefined}` (or the prop is omitted). If a message has `status: "ERROR"`, the bubble renders without a retry button.

### Route guard: only closed statuses
If `caseDoc.status` is not one of `CLOSED_RESOLVED`, `CLOSED_UNRESOLVED`, or `CLOSED_ABANDONED`, the component renders `<Navigate to={/cases/${caseId}} replace />` to bounce back to the orchestrator, which will route to the correct phase view.

### Tab state in URL
The active tab is stored as `?tab=joint|private|guidance` via `useSearchParams`. Default is `joint`. Switching tabs calls `setSearchParams` with `replace: true` to avoid polluting browser history. The tab value survives page refresh.

### Closure summary conditional
`closureSummary` block (with `data-testid="closed-closure-summary"`) is rendered if and only if `caseDoc.status === "CLOSED_RESOLVED"` AND `caseDoc.closureSummary` is truthy. For CLOSED_UNRESOLVED and CLOSED_ABANDONED, this block does not appear in the DOM.

## Edge cases

### Loading state
While any of the queries (`caseDoc`, `jointMessages`, `synthesis`, `privateMessages` for the active tab) return `undefined`, the component renders `<LoadingSpinner />`. Private coaching messages are only fetched when the "My Private Coaching" tab is active (conditional query with `useQuery` skip pattern — pass `"skip"` as args when tab !== "private").

### Empty joint chat
If no joint messages exist (e.g., case was abandoned before any joint chat happened), the Joint Chat tab panel renders `ChatWindow` with an empty messages array. No special empty-state message is needed — the banner already explains the case is closed.

### Missing synthesis
If `mySynthesis` returns `null` (e.g., case abandoned before synthesis was generated), the "My Guidance" tab panel renders "Synthesis not available." — matching the fallback text in the existing `SynthesisPanel` component in `JointChatView.tsx`.

### Missing closure summary
If the case is CLOSED_RESOLVED but `closureSummary` is empty/null (shouldn't happen in normal flow but defensive), the closure summary block is not rendered.

### Solo mode
Solo mode works identically to `JointChatView`: `useSoloActingParty(caseId)` provides `actingRole` and `isSolo`, and `viewAsRole` is passed to all queries that accept it. `PartyToggle` is shown in the header when solo.

## Non-goals

- **No re-open functionality.** There is no button to re-open a closed case. This is out of scope for v1.
- **No PDF export.** Case history export (US-18) is deferred to v1.1.
- **No other party's data.** The other party's private coaching and synthesis are never shown — not even a grayed-out or redacted version. The tabs are explicitly scoped to "My" data.
- **No Radix Tabs package.** The project does not currently have `@radix-ui/react-tabs` installed. Rather than adding a dependency for three tabs, implement with native HTML + ARIA attributes (`role="tablist"`, `role="tab"`, `role="tabpanel"`, `aria-selected`, `aria-controls`, `id`). Keyboard navigation (arrow keys between tabs, Enter/Space to activate) must be implemented per WAI-ARIA Tabs pattern.
