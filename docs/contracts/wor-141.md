# Contract: WOR-141 — Fix MessageInput test/impl contradiction (Send button disabled-state)

## Why this work exists

The test file `tests/unit/message-input.test.tsx` contains a contradictory assertion introduced by a pipeline bug (now fixed in Archon PR #35). Line 85-92 asserts the Send button is enabled (`disabled === false`) when rendering `<MessageInput isAiResponding={false} />` with an empty textarea. But the correct implementation — `disabled={isAiResponding || !hasText}` — disables the button when the textarea is empty. This causes 1 of 13 tests to fail. The fix is test-only: type text into the textarea before asserting the button is enabled.

## Files and exports

### `tests/unit/message-input.test.tsx` (modify)

This is the only file touched. It is an existing Vitest unit test file for the `MessageInput` component. The modification is confined to the test case at lines 85-92 within the `"AC: Send button disabled while AI is responding"` describe block.

**What changes:**

The test `"Send button is enabled when isAiResponding is false"` (lines 85-92) must be updated to:

1. Rename the test description to `"Send button is enabled when isAiResponding is false and textarea has content"` to accurately reflect what it tests.
2. After rendering, query for the `textarea` element.
3. Fire a `change` event on the textarea with `{ target: { value: "hello" } }` (or any non-empty string) to put the component into the "has text" state.
4. Then assert `sendBtn!.disabled` is `false`.

This makes the test consistent with the component's contract: the button is enabled only when `isAiResponding === false` AND the textarea has content.

No exports change. No new imports are needed (`fireEvent` is already imported).

## Data dependencies

None. This is a pure test fix with no backend interactions.

## Invariants

**No production code changes.** `src/components/chat/MessageInput.tsx` implements `disabled={isAiResponding || !hasText}` which is the correct behavior per the WOR-104 contract. This ticket fixes only the test that contradicts this behavior.

**Send button disabled-state truth table (unchanged):**

| `isAiResponding` | `hasText` | `disabled` |
|---|---|---|
| `false` | `false` | `true` |
| `false` | `true` | `false` |
| `true` | `false` | `true` |
| `true` | `true` | `true` |

The line-91 test was testing the `(false, false)` cell but expecting `disabled === false`. After the fix, it tests the `(false, true)` cell which correctly expects `disabled === false`.

**All 13 tests pass.** No test is deleted or added. The total test count remains 13.

## Edge cases

**Empty textarea + isAiResponding=false:** Button is disabled. This is already covered by the `"AC: Send button disabled when input is empty"` describe block (line 127-133). The line-91 test does NOT need to re-test this case — it should test the positive case (text present, AI not responding, button enabled).
